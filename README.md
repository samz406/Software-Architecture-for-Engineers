# 《面向工程师的软件架构：从代码实现到系统设计》

本仓库用于本书的章节化写作与迭代管理，目标读者为 2-8 年经验的软件工程师、高级工程师、技术负责人，以及正在向架构师成长的开发者。

## 这本书想解决什么问题
很多工程师并不缺“把功能写出来”的能力，真正难的是：
- 需求频繁变化后，系统还能不能稳住边界；
- 业务复杂度上来后，代码和数据模型是否还能被团队理解；
- 多团队协作、线上故障、历史包袱叠加时，如何做出可持续的架构选择。

本书主线是：**代码能力 → 模块能力 → 系统能力 → 业务能力 → 组织协同能力 → 架构演进能力**。

## 仓库结构
- `init.md`：全书写作总纲与章节规划。
- `chapters/`：按“每章一个文档”初始化的章节文件（`chapter-01.md` ... `chapter-38.md`）。

## 章节导读

### 第一部分：架构不是画图，而是解决复杂性（01-03）
从“为什么工程师需要架构能力”切入，建立从代码实现走向系统思考的认知起点。

### 第二部分：架构的底层能力（04-08）
聚焦抽象、边界、分层、建模与权衡，形成架构设计的基础动作。

### 第三部分：工程架构的核心主题（09-16）
覆盖接口、数据、状态、并发一致性、性能、稳定性、安全等高频工程议题。

### 第四部分：架构风格与技术方案（17-21）
讨论单体、微服务、事件驱动、DDD 与平台化，强调“阶段匹配”与“约束下取舍”。

### 第五部分：架构设计方法论（22-25）
提供从架构设计到文档、评审、ADR 的完整方法闭环。

### 第六部分：架构落地（26-29）
聚焦实施推进、技术债治理、系统重构与持续演进。

### 第七部分：案例篇（30-34）
以订单、优惠券、支付、统一账号与集团平台为主，讲清真实业务复杂性。

### 第八部分：架构师成长（35-38）
讨论能力模型、判断力训练、跨角色沟通与成长路径。

## 使用建议
1. 先读 `init.md` 理解全书地图。
2. 按章节文件逐章撰写，每章遵循统一结构：场景 → 复杂性 → 概念 → 误区 → 方法 → 案例 → 判断清单 → 小结。
3. 每章完成后进行质量审稿，重点检查是否“有现场感、有取舍、可落地、非模板化”。
