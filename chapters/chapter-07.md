# 第 7 章：建模能力：从业务语言到系统模型

## 1. 从一个真实问题开始

一个做了两年的电商系统，订单表有一个 status 字段，类型是 int。最早上线时只有四种状态：待支付、已支付、已发货、已完成。后来加了取消、退款中、退款完成、部分退款、超时关闭。再后来支持了预售，又加了待成团、成团失败。再后来支持了分期，又加了分期审核中、分期拒绝。

到现在这个字段有 14 种取值，每种取值的含义散落在代码的各个角落。有些状态之间可以互相转换，有些不能。但哪些可以、哪些不能，没有文档记录，只能看代码里的 if-else 判断。

新来的工程师问："订单状态为 7 的时候，用户能不能申请退款？"没有人能立刻回答，需要去翻代码。

更糟的是，订单状态和支付状态之间有隐含依赖。订单状态为"已支付"的前提是支付状态为"支付成功"，但这个前提条件没有被显式建模，只是在业务代码里通过 if 判断来保证。偶尔出现数据不一致——订单状态是"已支付"但支付记录是"支付失败"——排查起来很痛苦。

这些问题的根源不是代码写得差，而是**在系统设计之初没有做建模**。需求来了就建表、写接口、写逻辑。字段是根据需求一个一个加上去的，而不是从一个完整的业务模型推导出来的。

## 2. 问题为什么会变复杂

### 2.1 "需求驱动建表"的惯性

大多数工程师的工作习惯是：接到需求 → 分析功能点 → 设计数据库表 → 写接口 → 测试上线。在这个过程中，"建模"这一步被跳过了。

需求说"订单要有一个状态"，工程师就加一个 status 字段。需求说"要支持退款"，工程师就给 status 加一个新的取值。这种做法在需求简单的时候没有问题，但随着业务复杂度上升，问题会逐渐暴露：

- 一个字段承载了太多含义；
- 字段之间的约束关系没有被显式表达；
- 不同的业务概念被混在同一张表里；
- 相同的业务概念在不同的表里有不同的表示方式。

### 2.2 "模型"缺失带来的连锁反应

当系统没有清晰的业务模型时，三种问题会持续积累：

**第一，沟通成本增加。** 产品说"已完成的订单"，工程师理解为 status = 4。但产品其实是指"已发货且签收超过 7 天且没有售后纠纷"的订单。一个业务概念对应一个复杂的数据库查询条件，每个人的理解可能不一样。

**第二，变更难度增大。** 当需要加一种新的订单类型（比如虚拟商品订单），发现现有的状态流转对不上。虚拟商品没有发货环节，但代码里到处都在判断"是否已发货"。要兼容新类型，就得在每个判断点加特殊逻辑。

**第三，数据质量下降。** 因为字段含义不清晰、状态流转没有约束，线上会出现"不应该存在"的数据组合。比如一个已取消的订单关联了一笔成功的支付记录。这类数据不一致问题一旦积累，修数据的成本远大于写代码的成本。

### 2.3 业务越复杂，模型的价值越大

简单的 CRUD 系统不太需要精细建模。一个内部工具、一个管理后台，数据结构和业务逻辑基本一一对应，表结构设计得合理就够了。

但一旦业务涉及到**流程**（订单流转、审批流转）、**规则**（优惠计算、风控策略）、**状态**（支付状态、退款状态）、**多方协作**（用户、商家、平台、物流、支付渠道），建模就变成了架构设计中最重要的一环。

因为在这些场景下，代码实现的正确性高度依赖于模型的正确性。如果模型是歪的，代码只能通过大量的 if-else 来补偿。

## 3. 核心概念解释

### 3.1 什么是建模

建模就是**把业务世界中的关键概念、关系和规则，用结构化的方式表达出来**。

它不是画 ER 图——ER 图只是建模的一种输出形式。建模的核心动作是：

1. **识别业务中的核心概念**：订单、商品、用户、支付、优惠券，这些是名词。
2. **定义概念之间的关系**：一个订单属于一个用户，一个订单包含多个商品项，一个订单关联一笔支付。
3. **定义概念的生命周期**：一个订单从创建到最终状态，经历了哪些阶段？每个阶段之间的转换条件是什么？
4. **识别业务规则和约束**：一个订单在支付成功之前不能发货。一张优惠券核销后不能再次使用。退款金额不能超过实付金额。

### 3.2 五种常用模型

在实际工程中，建模通常涉及以下几种模型，它们之间不是替代关系，而是互补关系。

**业务模型（Business Model）**

描述业务是怎么运作的。比如电商的业务模型：用户浏览商品 → 加入购物车 → 下单 → 支付 → 发货 → 签收。业务模型帮助工程师理解"我们在做什么"，避免技术方案偏离业务目标。

**领域模型（Domain Model）**

把业务中的核心概念抽象为对象，定义它们的属性、行为和关系。比如"订单"是一个领域对象，它有金额、状态、创建时间等属性，有"取消""申请退款"等行为，和"用户""商品""支付"有关联关系。

领域模型不等于数据库表。一个领域对象可能对应多张表，一张表也可能包含多个领域概念的数据。

**状态模型（State Model）**

描述一个对象在其生命周期中会经历哪些状态，以及状态之间的转换条件。最常见的表达形式是状态机。

比如订单状态机：

```
待支付 --[支付成功]--> 已支付
待支付 --[超时]--> 已关闭
待支付 --[用户取消]--> 已取消
已支付 --[发货]--> 已发货
已支付 --[申请退款]--> 退款中
已发货 --[签收]--> 已完成
退款中 --[退款成功]--> 已退款
退款中 --[退款失败]--> 已支付
```

状态模型的价值在于**把隐含在代码 if-else 中的状态流转规则显式化**。

**流程模型（Process Model）**

描述一个业务流程的步骤、分支和异常处理。和状态模型的区别是：状态模型关注一个对象的状态变化，流程模型关注多个参与方之间的协作过程。

比如退款流程：用户发起退款 → 系统校验退款条件 → 调用支付渠道退款接口 → 等待渠道回调 → 更新退款状态 → 退还优惠券 → 释放库存 → 通知用户。

**数据模型（Data Model）**

描述数据如何存储。包括表结构设计、字段定义、索引设计、表之间的关系。数据模型是领域模型在存储层的映射，但不一定一一对应。

### 3.3 模型之间的关系

这五种模型之间的理想关系是：

- 业务模型决定了系统需要处理哪些业务场景；
- 领域模型从业务模型中提取核心概念和关系；
- 状态模型和流程模型描述领域对象的行为和协作方式；
- 数据模型是领域模型在存储层的落地。

在实际项目中，很少有团队会完整地做完所有五种模型。但至少需要对核心业务做领域模型和状态模型。否则，数据库表设计就只能靠直觉和经验，而直觉在业务复杂度高的时候往往不够用。

## 4. 常见误区

### 4.1 把建模等同于画 ER 图

ER 图是数据模型的可视化工具，但建模远不止画 ER 图。ER 图能告诉你"订单表和支付表是一对一关系"，但它不能告诉你"订单在什么条件下可以发起支付"。

如果只画 ER 图而不做领域模型和状态模型，设计出来的系统会有数据结构但缺少行为约束。数据能存，但数据之间的业务规则只能靠代码来保证，容易遗漏。

### 4.2 模型和代码脱节

有些团队在项目启动时做了详细的建模，画了类图、状态图、流程图。但开发过程中，代码逐渐偏离模型。一年后，模型文档还是旧的，代码已经面目全非。

模型和代码脱节的根本原因是**模型没有在代码中得到体现**。如果状态流转规则只画在文档里，代码里还是用一堆 if-else 判断，那么模型就只是摆设。

好的做法是让模型在代码中有对应的结构。比如用枚举定义状态、用状态机框架管理状态转换、用领域对象封装业务规则。这样模型和代码是同步演进的，而不是两张皮。

### 4.3 过度建模

在一个日活几百人的内部管理系统上搞完整的 DDD 建模、画六七层架构图、设计复杂的领域事件，这就是过度建模。

建模的深度应该匹配业务的复杂度。简单的 CRUD 系统，表结构设计合理、字段含义清晰就够了。中等复杂度的系统，需要领域模型和状态模型。只有核心业务系统（比如订单、支付、风控）才值得做完整的建模。

判断标准是：**如果不做模型，代码中的 if-else 会不会失控？如果答案是不会，那就不需要精细建模。**

### 4.4 把数据模型当成领域模型

这是一个很常见的问题。工程师先设计数据库表，然后根据表结构生成实体类，把这些实体类当成领域对象来用。

结果就是：领域对象里只有 getter/setter，没有业务行为。所有的业务逻辑都写在 Service 里。Service 操作的是"数据"而不是"领域概念"。

比如一个订单实体类，只有 getStatus()/setStatus() 方法，没有 cancel()、applyRefund()、ship() 这样的业务方法。取消订单的逻辑写在 OrderService.cancelOrder() 里，里面直接调用 order.setStatus(CANCELLED)。

这种贫血模型的问题是：业务规则散落在各个 Service 里，没有一个集中的地方可以看到"订单可以做什么、不能做什么"。

### 4.5 忽视状态模型

很多工程师对"状态机"的印象停留在理论课上的有限状态自动机。在实际项目中，他们不会主动去画状态图，而是直接在代码里写 if (status == XXX) 的判断。

当状态只有三四种时，这样做问题不大。但当状态增长到十几种，状态之间有复杂的转换条件和约束时，散落在代码各处的 if 判断就变成了地雷——你不知道改了一个条件会不会导致某个边缘情况出错。

状态模型的价值就在于把这些隐含的规则变成显式的、可查看的、可验证的结构。

## 5. 设计方法

### 5.1 建模的基本步骤

**第一步：识别核心业务概念**

从需求文档、产品原型或业务流程中，提取关键的"名词"。这些名词通常就是领域对象的候选。

比如做一个电商系统，核心名词可能是：用户、商品、订单、订单项、支付单、优惠券、库存、物流单、退款单。

注意区分**核心概念**和**辅助概念**。核心概念是系统的主角，它们有自己的生命周期和状态。辅助概念是配角，通常依附于核心概念存在。比如"订单"是核心概念，"订单备注"是辅助概念。

**第二步：定义概念之间的关系**

关系通常分为几类：
- 一对一：一个订单对应一个支付单；
- 一对多：一个订单包含多个订单项；
- 多对多：一个用户可以有多张优惠券，一张优惠券可以被多个用户领取（通过券实例关联）；
- 依赖关系：退款单依赖于订单存在；
- 引用关系：订单引用商品信息，但商品变化不应该影响历史订单。

**第三步：定义核心对象的生命周期**

对于有状态的对象（订单、支付单、优惠券、退款单），画出它的状态转换图。明确：
- 有哪些状态？
- 每个状态的含义是什么？
- 从状态 A 到状态 B 的条件是什么？
- 哪些状态转换是不允许的？

**第四步：识别业务规则和约束**

列出系统中重要的业务规则。比如：
- 优惠券只能在有效期内使用；
- 退款金额不能超过实付金额；
- 已发货的订单不能直接取消，需要先退货；
- 一个订单只能使用一张满减券。

这些规则应该在模型中有对应的位置，而不是散落在代码各处。

**第五步：从领域模型推导数据模型**

基于领域模型设计数据库表。注意几个原则：
- 一个领域对象通常对应一张主表，但复杂的对象可能需要拆分成多张表；
- 关系可以通过外键、关联表或冗余字段来实现，具体取决于查询模式和性能要求；
- 状态字段的设计要和状态模型一致，状态的取值应该用枚举而不是魔法数字。

### 5.2 状态建模的实践方法

状态模型是业务系统建模中最容易被忽视但也最有价值的部分。

**用枚举定义状态**

不要用 int 或 String 存储状态，用枚举。枚举可以强制约束取值范围，还可以附加行为。

```java
public enum OrderStatus {
    PENDING_PAYMENT,
    PAID,
    SHIPPED,
    COMPLETED,
    CANCELLED,
    CLOSED,
    REFUNDING,
    REFUNDED;
}
```

**用状态转换表定义合法转换**

不要让状态转换散落在各个 Service 方法里。可以用一个集中的状态转换表来定义哪些转换是合法的：

```java
// 定义合法的状态转换
Map<OrderStatus, Set<OrderStatus>> transitions = Map.of(
    PENDING_PAYMENT, Set.of(PAID, CANCELLED, CLOSED),
    PAID, Set.of(SHIPPED, REFUNDING),
    SHIPPED, Set.of(COMPLETED),
    REFUNDING, Set.of(REFUNDED, PAID)
);
```

每次修改状态前，先检查转换是否合法。这样即便业务代码有 bug，也不会产生不合法的状态数据。

**记录状态变更历史**

核心业务对象的状态变更应该有日志。不是打 log 那种日志，而是结构化的状态变更记录表：

```sql
CREATE TABLE order_status_log (
    id BIGINT PRIMARY KEY,
    order_id BIGINT,
    from_status VARCHAR(32),
    to_status VARCHAR(32),
    operator VARCHAR(64),
    reason VARCHAR(256),
    created_at TIMESTAMP
);
```

这张表在排查问题和审计时非常有用。当线上出现数据异常时，可以通过状态变更日志还原整个过程。

### 5.3 领域模型到代码结构的映射

领域模型不应该只停留在文档里，它需要在代码中有对应的结构。

**领域对象应该包含业务行为，而不只是数据容器。** 比如：

```java
public class Order {
    private OrderStatus status;
    private Money totalAmount;
    private Money paidAmount;
    
    public void cancel(String reason) {
        if (this.status != OrderStatus.PENDING_PAYMENT) {
            throw new BusinessException("只有待支付的订单可以取消");
        }
        this.status = OrderStatus.CANCELLED;
        // 记录取消原因
    }
    
    public void applyRefund(Money refundAmount) {
        if (this.status != OrderStatus.PAID && this.status != OrderStatus.SHIPPED) {
            throw new BusinessException("当前状态不允许退款");
        }
        if (refundAmount.greaterThan(this.paidAmount)) {
            throw new BusinessException("退款金额不能超过实付金额");
        }
        this.status = OrderStatus.REFUNDING;
    }
}
```

这样做的好处是：看到 Order 类就知道订单可以做什么、有什么约束。不需要在四五个 Service 里翻找业务规则。

## 6. 业务案例

### 6.1 案例：优惠券系统的模型设计

优惠券是一个看似简单但建模容易出错的业务场景。

**常见的错误做法：一张表搞定一切**

```sql
CREATE TABLE coupon (
    id BIGINT PRIMARY KEY,
    name VARCHAR(128),
    type INT,           -- 1:满减 2:折扣 3:立减
    value DECIMAL,      -- 面值或折扣率
    threshold DECIMAL,  -- 使用门槛
    user_id BIGINT,     -- 持有用户
    status INT,         -- 1:未使用 2:已使用 3:已过期 4:已冻结
    order_id BIGINT,    -- 使用时关联的订单
    start_time DATETIME,
    end_time DATETIME
);
```

这张表把"优惠券模板"和"优惠券实例"混在了一起。name、type、value、threshold 是模板属性，user_id、status、order_id 是实例属性。

当运营创建一个"满 200 减 30"的活动，发放 10000 张券时，这张表里会有 10000 条记录，每条记录的 name、type、value、threshold 都一样，冗余严重。更大的问题是：如果活动规则需要修改（比如改成满 200 减 25），是改所有记录还是只改新发放的？语义不清。

**合理的建模：区分模板和实例**

通过领域建模，可以识别出两个核心概念：

- **券模板（CouponTemplate）**：定义了一种优惠券的规则——类型、面值、门槛、有效期、库存、使用条件。
- **券实例（CouponInstance）**：一张具体发到用户手里的券。它关联一个模板，有自己的状态和使用记录。

```sql
CREATE TABLE coupon_template (
    id BIGINT PRIMARY KEY,
    name VARCHAR(128),
    type VARCHAR(32),        -- FULL_REDUCTION, DISCOUNT, FIXED_AMOUNT
    value DECIMAL,
    threshold DECIMAL,
    total_stock INT,
    used_stock INT,
    start_time DATETIME,
    end_time DATETIME,
    rules JSON               -- 使用规则：品类限制、互斥规则等
);

CREATE TABLE coupon_instance (
    id BIGINT PRIMARY KEY,
    template_id BIGINT,
    user_id BIGINT,
    status VARCHAR(32),      -- UNUSED, USED, EXPIRED, FROZEN, RETURNED
    used_order_id BIGINT,
    used_at DATETIME,
    received_at DATETIME
);
```

这样模板和实例各自有清晰的生命周期：

**模板的状态：** 未开始 → 进行中 → 已结束 → 已下架

**实例的状态：** 未使用 → 已使用 / 已过期 / 已冻结 → 已退回

退款时需要退回优惠券，只需要把对应实例的状态从"已使用"改回"未使用"（或一个专门的"已退回"状态），逻辑清晰。

### 6.2 案例：退款流程的模型设计

退款是一个经常被低估的业务场景。很多系统一开始只做了"全额退款"，后来追加了"部分退款""多次退款""仅退款不退货"，模型设计不到位的话会非常痛苦。

**核心建模决策：退款单是独立实体还是订单的一个状态？**

如果把退款当作订单的一个状态（order.status = REFUNDING），那么一个订单只能有一次退款，无法支持多次部分退款。而且订单的状态机会变得过于复杂——要处理"部分退款中""部分退款完成""退款失败回退到已支付"等各种情况。

合理的做法是**把退款单建模为独立的实体**。退款单有自己的生命周期，和订单是关联关系而不是从属关系。

```sql
CREATE TABLE refund_order (
    id BIGINT PRIMARY KEY,
    order_id BIGINT,
    refund_amount DECIMAL,
    reason VARCHAR(256),
    status VARCHAR(32),     -- PENDING, PROCESSING, SUCCESS, FAILED
    channel_refund_id VARCHAR(128),
    created_at DATETIME,
    completed_at DATETIME
);
```

退款单的状态机：

```
待处理 --[提交退款请求]--> 处理中
处理中 --[渠道退款成功]--> 退款成功
处理中 --[渠道退款失败]--> 退款失败
退款失败 --[重新发起]--> 处理中
```

这样一个订单可以有多个退款单，每个退款单独立跟踪自己的状态。订单本身的状态只需要关注"是否有未完结的退款"。

### 6.3 从案例中总结建模的关键判断

1. **区分"模板"和"实例"**：当一个业务概念有"定义"和"使用"两个阶段时，通常需要拆分为两个实体。优惠券如此，商品和 SKU 的关系也是如此。

2. **独立的生命周期意味着独立的实体**：如果一个对象有自己的状态流转、不依赖于另一个对象的状态来决定自身行为，它应该是一个独立的实体，而不是另一个实体的属性。

3. **状态字段过多是建模不足的信号**：如果一个实体的 status 字段有十几种取值，通常意味着这个实体承担了太多职责，或者多个不同维度的状态被混在了一个字段里。

## 7. 架构判断清单

- 如果一个状态字段有超过 8 种取值，考虑是否需要拆分为多个维度的状态，或者将部分职责拆分到独立实体。
- 如果产品和工程师对同一个业务概念有不同的理解，说明领域模型不够清晰，需要先对齐语义。
- 如果新增一种业务类型（比如新增虚拟商品订单）需要修改大量已有代码的 if-else，说明模型缺少扩展能力，需要重新审视建模。
- 如果排查数据问题时无法快速还原一个对象的状态变化历史，说明缺少状态变更日志。
- 如果两个表之间的关联关系只能通过业务代码来理解（而不能通过表结构和字段名看出来），说明数据模型的设计不够自描述。
- 如果同一个业务规则在三个以上的地方被重复实现，说明这个规则应该内聚到领域对象中，而不是散落在 Service 层。
- 如果需求变更时，工程师第一反应是"加个字段"，而不是"这个概念在模型中应该怎么表达"，说明团队缺少建模意识。

## 8. 本章小结

建模是一种把业务知识转化为系统结构的能力。它不同于写代码——写代码解决的是"怎么实现"，建模解决的是"系统中有什么、它们之间是什么关系、它们怎么变化"。

很多系统的技术债务，根源不在代码质量，而在模型质量。一个模型清晰的系统，即便代码写得一般，也能通过重构逐步改善。但一个模型混乱的系统，重构的第一步往往不是改代码，而是重新理解业务、重新建模——这个成本比写代码高得多。

建模不需要追求完美。没有哪个模型能在第一版就覆盖所有未来的变化。好的建模方式是：在项目初期花一两天时间把核心概念和状态流转理清楚，在后续迭代中根据实际需要逐步补充和调整。关键是让模型和代码保持同步，而不是画一版文档然后束之高阁。

对于工程师来说，建模能力的核心标志是：面对一个新业务需求时，能不能先用几句话描述出核心实体、它们的关系和状态流转，再去想数据库怎么设计、接口怎么写。这个顺序看起来只是一个小调整，但它决定了系统最终是"从模型长出来的"还是"从字段堆出来的"。
