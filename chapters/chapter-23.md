# 第 23 章：架构文档怎么写

## 1. 从一个真实问题开始

一个工程师花了三天时间写了一份架构设计文档，十二页，配了四张图。评审会上大家看了半小时，问了一堆问题。

"这个方案的目标是什么？解决什么问题？"——第三页有提，但埋在一大段技术描述里。
"选 Kafka 不选 RabbitMQ 的原因是什么？"——文档里没写，工程师口头解释了五分钟。
"如果这个方案上线后出问题，回滚方案是什么？"——没考虑。
"第一期和第二期怎么分？"——没写。

评审会开了两个小时，结论是"回去补充后再评审"。

第二版文档补充了选型理由和分期计划，但又收到新的反馈："图画了四张但看不懂它们之间的关系""技术细节写了很多但业务场景没交代清楚""用了很多术语但没有解释在这个场景下具体是什么意思"。

第三版才通过评审。从开始写到通过评审，花了两周。

**写架构文档的难点不在于"会不会写"，而在于"知道该写什么、写给谁看、写到什么程度"。**

## 2. 问题为什么会变复杂

### 2.1 文档的读者不是自己

很多工程师写架构文档时，默认读者和自己有相同的上下文。"我知道为什么选 Kafka，这不是显而易见的吗？"但评审的人可能不了解你面对的具体约束，不知道你考虑过的其他方案。

文档要写的是**你脑子里有但读者脑子里没有的信息**。你考虑了三个方案选了一个——读者需要知道三个方案分别是什么、为什么选了这个。你做了一个假设——读者需要知道这个假设是什么以及如果假设不成立会怎样。

### 2.2 技术细节和全局视图的平衡

写太粗，评审的人觉得"不落地"。写太细，评审的人在细节里迷失方向看不到全局。

有人的文档全是代码片段和表结构，缺少整体的设计思路。有人的文档全是概念和原则，没有具体的数据模型和接口定义。

### 2.3 文档不只是给评审用的

架构文档在评审之后的用途往往比评审本身更重要：
- 新人加入团队，需要通过文档理解系统的设计思路。
- 半年后修改这个模块，需要翻回文档看当初为什么这么设计。
- 出了线上问题，需要查文档看设计中是否考虑过这种场景。

一份好的架构文档是系统的"设计图纸"，在系统的整个生命周期内都有价值。

## 3. 核心概念解释

### 3.1 架构文档的结构

一份实用的架构文档，推荐以下结构：

**1. 背景和目标**

为什么要做这个事？解决什么问题？业务上的目标是什么？

不要写"为了优化系统架构"这种空话。要写具体的："当前订单查询的 P95 响应时间已经到 3 秒，大促期间会超时。目标是将 P95 降到 500ms 以内，支撑 10 倍于当前的查询 QPS。"

**2. 非目标（Non-goals）**

明确这次设计**不**做什么。

"本次不涉及订单写入链路的优化。""本次不做分库分表，数据量在一年内不需要。""本次不改变现有的订单状态机。"

非目标和目标同样重要。它帮助读者理解设计的边界，避免在评审时讨论设计范围之外的话题。

**3. 现状和问题分析**

当前系统是什么样的？问题出在哪？用数据和事实说话。

"当前订单表有 8000 万条数据，查询走了全表扫描因为缺少联合索引。高峰期数据库 CPU 达到 90%，连接池经常排队。"

**4. 方案设计**

这是文档的主体。包括：
- 整体架构（模块划分、依赖关系）
- 数据模型（核心表、关键字段、索引）
- 核心流程（关键场景的时序图或流程图）
- 接口定义（模块间或服务间的核心接口）

**5. 方案对比和取舍**

如果有多个可选方案，列出每个方案的优劣和选择理由。

```
方案 A：加联合索引 + Redis 缓存
  优点：改动小，一周内上线
  缺点：数据量继续增长后还是会遇到瓶颈

方案 B：订单读写分离，读请求走从库
  优点：读性能显著提升，扩展性好
  缺点：引入主从延迟问题，需要改造查询链路

方案 C：订单查询走 Elasticsearch
  优点：灵活查询，高性能
  缺点：引入新组件，运维成本高，数据同步有延迟

选择方案 A。原因：当前数据量 8000 万在 MySQL 能力范围内，
加索引+缓存足够解决当前问题。数据量超过 3 亿时再考虑方案 B。
```

**6. 风险和应对**

设计中有哪些风险？每个风险的应对方案是什么？

**7. 演进计划**

分几期做？每期的范围和时间？当前设计对后续演进有什么支撑？

### 3.2 架构图怎么画

架构文档中的图要回答明确的问题，不是为了画图而画图。

**系统上下文图**：回答"这个系统和哪些外部系统交互"。适合在文档开头用，帮读者建立全局认知。

**模块关系图**：回答"系统内部分哪些模块、模块之间怎么交互"。适合在方案设计部分用。

**时序图**：回答"核心场景中各模块的交互顺序是什么"。适合描述具体的业务流程。

**数据模型图**：回答"核心实体之间的关系是什么"。适合有复杂数据模型时用。

每张图旁边需要有文字说明。图解决的是"直观理解结构"，文字解决的是"理解细节和决策"。只有图没有文字，读者看不懂；只有文字没有图，读者建不起全局的画面。

### 3.3 写作风格

**用具体的描述代替抽象的概念。**

不好的写法："系统具有良好的扩展性，支持水平扩展。"
好的写法："订单查询服务是无状态的，部署 3 个实例，通过负载均衡分发请求。需要扩容时加实例即可，不需要改代码。"

**用数据代替感觉。**

不好的写法："当前系统性能较差，需要优化。"
好的写法："订单列表接口 P95 响应时间 2800ms，高峰期 QPS 约 500。目标是 P95 降至 500ms，支撑 5000 QPS。"

**写决策的理由，不只是决策本身。**

不好的写法："消息队列使用 Kafka。"
好的写法："消息队列使用 Kafka。原因：支付成功事件需要被多个消费者独立消费（积分、通知、报表），Kafka 的消费者组机制天然支持这种场景。RabbitMQ 也能实现但需要额外配置 Exchange，团队对 Kafka 更熟悉。"

## 4. 常见误区

### 4.1 "写了没人看"

很多团队的架构文档写完评审后就再也没人打开了。原因通常是文档内容和实际实现脱节——文档写的是方案 A，实际做的时候改成了方案 B，但文档没有更新。

解决方法：不追求文档面面俱到，重点记录**关键决策和取舍理由**。即使具体实现变了，当初的决策思路仍然有价值。

### 4.2 贴大量代码

有的文档里贴了几百行代码。架构文档不是代码 Walkthrough。代码是实现层面的细节，应该在代码审查中讨论，不是在架构文档中展示。

文档中可以有代码片段，但应该是关键的、说明设计意图的代码，比如接口定义、核心的数据模型定义。不应该有完整的业务逻辑代码。

### 4.3 只写"是什么"，不写"为什么"

很多文档只描述了设计方案本身——"用 Redis 做缓存，过期时间 5 分钟"。但没有解释为什么用 Redis 不用本地缓存、为什么过期时间是 5 分钟不是 1 分钟。

半年后看到这份文档，"5 分钟"这个数字就变成了一个黑盒。没人知道能不能改、改了会怎样。

### 4.4 文档过长

有的架构文档写了三十页。评审的人需要花两个小时才能读完。实际上大多数人读到第五页就开始走马观花了。

控制文档长度。核心内容控制在五到八页以内。细节部分（详细的表结构、完整的接口列表）可以放在附录中，有需要的人再看。

### 4.5 没有模板就不会写

有些工程师觉得"没有模板我不知道怎么开始写"。但模板只是一个起点，不同类型的项目侧重点不同：
- 新系统设计侧重模块划分和数据模型。
- 性能优化侧重现状分析和方案对比。
- 系统重构侧重风险分析和迁移计划。

用本章给出的结构作为起点，根据实际需要增删。

## 5. 设计方法

### 5.1 写之前先想清楚

动笔之前回答三个问题：

1. **这份文档的主要读者是谁？** 如果是团队内部对齐，可以假设读者有基本的技术背景。如果是跨团队评审，需要交代更多上下文。

2. **读者看完后应该知道什么？** 理解为什么要做、大概怎么做、有什么风险。

3. **读者可能问什么问题？** 提前在文档中回答。"为什么不用 X？""这个方案的性能上限是多少？""上线后出问题怎么回滚？"

### 5.2 文档的维护

架构文档在系统的生命周期中需要维护：

- 方案设计阶段写初稿。
- 评审后更新调整的部分。
- 开发中如果方案有重大变更，更新文档中的关键决策。
- 上线后补充实际效果（性能数据、上线过程中遇到的问题）。

不需要维护得面面俱到——关键决策和取舍理由保持准确就够了。

### 5.3 文档模板

提供一个精简版的架构文档模板：

```markdown
# [项目名称] 架构设计

## 1. 背景和目标
- 业务背景：为什么要做
- 技术现状：当前系统的问题（用数据说话）
- 目标：要达到什么效果（可衡量）
- 非目标：这次不做什么

## 2. 方案设计
- 整体架构（图 + 说明）
- 核心流程（时序图 + 说明）
- 数据模型（表结构 + 关键索引）
- 接口定义（核心接口的入参出参）

## 3. 方案对比
- 方案 A / B / C 的优劣对比
- 选择理由

## 4. 风险和应对
- 风险一：描述 + 应对
- 风险二：描述 + 应对

## 5. 分期计划
- 第一期：范围、时间
- 第二期：范围、依赖

## 附录
- 详细表结构
- 完整接口列表
```

## 6. 业务案例

### 6.1 案例：订单查询优化的架构文档

以下是一份精简版的架构文档示例，展示每个部分应该怎么写。

**背景和目标**：

> 订单列表查询是用户访问频率最高的接口。当前 P95 响应时间为 2800ms，高峰期频繁超时。订单表数据量 8000 万行，按当前增速半年后将超过 1.5 亿。
>
> 目标：P95 响应时间降至 500ms，支撑 5000 QPS（当前约 500）。
>
> 非目标：本次不改订单写入流程，不做分库分表。

**方案设计**：

> 整体方案：索引优化 + 查询缓存 + 接口拆分。
>
> 1. 给 orders 表增加联合索引 `(user_id, order_status, created_at)`，覆盖用户订单列表的主查询场景。
> 2. 用户最近 20 条订单缓存到 Redis，缓存 key 为 `order:list:{userId}`，过期时间 5 分钟。订单状态变更时删除该用户缓存。
> 3. 订单详情接口拆分：基本信息从订单表读取，物流信息异步加载，商品快照信息从冗余字段读取，不再实时查商品服务。

**方案对比**：

> 对比了三个方案（索引+缓存 vs 读写分离 vs ES）。选择方案 A（索引+缓存），理由：改动最小，一周内上线，能解决当前问题。数据量超过 3 亿时启动方案 B 的规划。

**风险和应对**：

> 风险一：缓存和数据库不一致。订单状态变更后，缓存可能有 1-2 秒的旧数据。应对：订单状态变更时主动删除缓存；用户对订单列表 1-2 秒的延迟可接受。
>
> 风险二：联合索引增加写入开销。orders 表每天新增约 10 万条数据。应对：测试显示新增一个联合索引后写入性能下降约 5%，在可接受范围内。

这份文档不长，但关键信息都有：问题是什么、怎么解决、为什么这么选、有什么风险。

## 7. 架构判断清单

- 如果架构文档只有图没有文字说明，说明"为什么这么设计"的信息丢失了。
- 如果文档中没有"非目标"部分，评审时容易在范围之外的话题上纠缠。
- 如果文档中只有一个方案、没有备选方案对比，读者无法判断这个方案是不是最好的选择。
- 如果文档超过二十页且没有明确的分层（概览 + 详细 + 附录），大多数人不会认真读完。
- 如果三个月后再看文档找不到"为什么选了方案 A"的解释，说明决策理由没有记录。
- 如果文档中的架构图和实际代码对不上，说明文档在开发过程中没有维护。
- 如果团队新人加入后要花一周时间才能理解系统设计，而文档看完后仍然不清楚，说明文档的"全局视图"不够。

## 8. 本章小结

架构文档的核心不是"写得全面"，而是"写清楚关键信息"。关键信息包括：为什么要做、怎么做、为什么这么做而不是那么做、有什么风险。

好的架构文档不长。五到八页，结构清晰，每个部分回答一个明确的问题。读者看完后能理解设计的全貌和关键决策。

写文档时永远记住：你写的不是给自己看的笔记，而是给其他人传递信息的工具。你脑子里那些"显而易见"的假设和决策理由，对读者来说一点都不显而易见。把它们写出来，就是好文档和坏文档的区别。
