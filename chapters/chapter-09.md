# 第 9 章：高内聚低耦合：最朴素也最重要的架构原则

## 1. 从一个真实问题开始

一个电商系统做了三年，团队从五个人扩到了二十人。最近一次迭代遇到了一个典型场景：

商品团队要做一个需求——"商品下架时自动关闭相关的优惠活动"。看起来不复杂，但开发排期排了两周。原因是：

- 商品下架的逻辑在 ProductService 里。ProductService 直接调用了 PromotionService 来查当前商品关联了哪些活动。但 PromotionService 的查询接口入参不满足这个场景——它只支持按活动 ID 查，不支持按商品 ID 查。需要 PromotionService 先加个接口。
- PromotionService 加接口后，需要查 promotion_product 关联表。但这张表的写入方是 ProductService（在"商品参加活动"操作时写的），查询方是 PromotionService。两个 Service 共用同一张表，谁改表结构对方都可能受影响。
- 关闭活动的逻辑涉及修改活动状态。活动状态的变更需要通知订单模块（因为有些订单引用了活动价格）。所以 PromotionService 又依赖了 OrderService 的一个通知接口。
- 商品团队排期时发现：改 ProductService 需要自己改，PromotionService 需要营销团队排期，OrderService 的通知接口需要订单团队确认。三个团队协调排期，两周已经算快了。

一个"商品下架关闭活动"的需求，牵动了三个团队、三个模块。

CTO 在周会上说了一句话："我们的系统，改一个模块要动三个团队。这说明模块之间耦合太深了。"

## 2. 问题为什么会变复杂

### 2.1 耦合是怎么产生的

没有人会故意设计一个高耦合的系统。耦合是在一次一次的"就近调用"中逐渐积累的。

最初 ProductService 和 PromotionService 是独立的。后来有个需求"在商品详情页展示该商品的促销信息"，ProductService 需要调 PromotionService 获取促销数据。直接注入一个 PromotionService，调用一下，需求完成。

再后来"创建促销活动时校验商品是否存在"，PromotionService 需要调 ProductService 校验商品。又直接注入调用。

到这一步，两个 Service 已经互相依赖了。再加上它们共享了 promotion_product 表，数据层面也耦合了。

每次"直接调用"都是最快的实现方式。但这些快速实现累加起来，就变成了难以拆解的依赖网。

### 2.2 耦合的代价

**改动成本高**：改一个模块，需要理解并适配其他模块的实现细节。一个简单需求涉及多个团队的协调。

**故障传播**：A 模块依赖 B 模块的接口，B 模块的接口超时或报错，A 模块跟着出问题。依赖链越长，系统越脆弱。

**测试困难**：测试一个模块需要把它的所有依赖都准备好。依赖越多，测试环境越难搭建。

**发布耦合**：A 模块和 B 模块互相依赖，发布时需要协调顺序——先发 A 还是先发 B？如果接口有变化，两个模块需要同时发布，风险叠加。

### 2.3 内聚不足的代价

耦合问题的另一面是内聚不足。

如果优惠券的使用校验逻辑散落在 OrderService、CartService、SettlementService 里，而不是集中在 CouponService 中，那每次改优惠券规则都要找遍所有 Service。

内聚不足意味着一个业务概念的逻辑被分散到了多个地方。改一个业务规则要动多个文件，漏改一个就是 bug。

## 3. 核心概念解释

### 3.1 什么是内聚

内聚描述的是**一个模块内部各部分之间的关联程度**。

高内聚意味着：模块内部的代码都在为同一个目的服务。一个 CouponService 里的方法都和优惠券相关——创建券、发券、校验券、核销券、退券。这些功能逻辑相关、数据相关、变化原因一致。

低内聚意味着：模块内部的代码为多个不相关的目的服务。如果 CouponService 里既有优惠券逻辑，又有积分计算，还有会员等级判断，那这个 Service 的内聚度就很低——这三块逻辑各自有不同的变化原因，不应该放在一起。

判断内聚度的一个简单方法：**用一句话描述这个模块的职责。如果需要用"和"来连接多个不相关的职责，说明内聚度不够。**

### 3.2 什么是耦合

耦合描述的是**两个模块之间的依赖程度**。

耦合有不同的强度，从高到低：

**内容耦合**（最紧）：一个模块直接修改另一个模块的内部数据。比如 OrderService 直接操作 coupon 表的字段，绕过了 CouponService。

**公共耦合**：多个模块共享全局变量或公共数据结构。比如两个模块共享同一张数据库表，各自读写。

**控制耦合**：一个模块通过传入标志位来控制另一个模块的行为。比如调用 `notify(userId, type)` 时，type 参数决定了通知模块走哪个分支——调用方需要了解通知模块内部的分支逻辑。

**数据耦合**（最松）：模块之间只通过明确的接口传递必要的数据。调用方不需要了解对方的内部实现。

目标不是消除所有耦合——模块之间不交互的系统是不存在的。目标是把耦合控制在数据耦合的级别：**模块之间通过清晰的接口传递数据，不共享内部状态，不了解对方的实现细节。**

### 3.3 耦合不是不能有，而是要可控

有些耦合是必要的。订单创建时需要锁库存——订单模块和库存模块之间必须有交互。问题不在于"有没有耦合"，而在于"耦合是否可控"。

可控的耦合有三个特征：
- **方向单一**：A 依赖 B，但 B 不依赖 A。没有循环依赖。
- **接口稳定**：A 依赖的是 B 的接口定义，而不是 B 的内部实现。B 重构内部代码不影响 A。
- **可替代**：A 和 B 之间的交互可以从同步改为异步、可以加一层缓存、可以用消息队列解耦——有调整的空间。

## 4. 常见误区

### 4.1 "把所有依赖去掉"

有些团队在看到模块间依赖后，试图把所有依赖都消除。结果是：原本一次直接调用就能完成的事情，变成了发消息、等回调、补偿重试的异步流程，复杂度反而增加了。

降低耦合不等于消除交互。两个模块之间需要交互，关键是**用什么方式交互**。直接调对方的内部方法是紧耦合，通过接口调用是中等耦合，通过事件通知是松耦合。选择哪种方式取决于业务场景的需要。

### 4.2 "同步调用就是紧耦合"

同步调用本身不等于紧耦合。如果 A 通过 B 的公开接口同步调用 B，接口定义明确、版本稳定，这是正常的数据耦合。

紧耦合的标志不是调用方式，而是依赖的深度：A 是否需要了解 B 的内部实现才能正确调用？A 调用 B 时传的参数是否包含了 B 的内部概念？A 是否直接操作了 B 的数据？

### 4.3 "微服务天然就是低耦合"

把代码拆成微服务不会自动降低耦合。如果两个微服务之间每个请求都要同步调用对方好几次，或者共享数据库，那它们只是"部署在不同进程里的紧耦合模块"。

物理拆分（部署层面）和逻辑解耦（设计层面）是两回事。先在设计层面把依赖关系理清楚，再考虑是否需要物理拆分。

### 4.4 过度追求内聚导致职责过窄

有些团队把内聚理解为"每个类只做一件事"，然后把一个业务流程拆成了几十个小类，每个类只有一个方法。读代码时需要在几十个文件之间跳转，比看一个大 Service 还累。

内聚的粒度不是越细越好。**内聚的单位应该是一个完整的业务概念**，而不是一个技术操作。OrderService 里包含订单创建、查询、取消、状态管理这几个方法，这是合理的内聚——它们都围绕"订单"这个核心概念。没必要把每个操作拆成一个独立的类。

### 4.5 "高内聚低耦合是设计阶段的事"

高内聚低耦合不是在设计阶段做一次就完了。随着业务迭代，模块的职责会漂移、依赖会增加。一个原本内聚的模块，在经过两年的需求堆叠之后，可能已经变成了一个职责混杂的"大泥球"。

需要定期审视模块的职责和依赖关系。每次做架构评审时，看一下核心模块的依赖图：有没有新增的循环依赖？有没有依赖数量异常增长的模块？

## 5. 设计方法

### 5.1 识别耦合问题

**方法一：画依赖图。** 列出核心模块，画出它们之间的调用关系。重点看：
- 有没有循环依赖？（A → B → A）
- 有没有"明星模块"？（一个模块被五个以上的模块依赖，或依赖五个以上的模块）
- 调用方向是否合理？（底层模块依赖上层模块是不正常的）

**方法二：统计改动扩散。** 查最近三个月的提交记录，统计每次提交涉及的模块数量。如果大部分提交都只涉及一两个模块，说明模块间边界清晰。如果经常出现一个提交涉及四五个模块，说明耦合问题严重。

**方法三：模拟替换。** 假设要把某个模块整体替换（比如换一套优惠券系统），估算需要修改多少其他模块的代码。修改的地方越多，耦合越深。

### 5.2 降低耦合的具体手段

**手段一：接口隔离。** 模块之间通过明确的接口（Facade）交互，不直接调用内部方法。接口的定义使用业务语言，不暴露实现细节。

**手段二：事件解耦。** 对于"A 做完之后 B 也要做一些事"的场景，用事件代替直接调用。A 发出事件，B 监听事件。A 不需要知道 B 的存在。

适合用事件的场景：
- 后续操作是异步的、可以延迟的：订单创建后发通知。
- 后续操作可以失败重试：支付成功后送积分。
- 有多个下游需要响应：订单取消后退券、退库存、退积分。

不适合用事件的场景：
- 操作需要同步返回结果：下单时查库存够不够。
- 强一致性要求：扣款和发货必须在同一个事务中。

**手段三：数据隔离。** 每个模块管理自己的数据，不共享数据库表。需要对方的数据时通过接口获取。

**手段四：适配层。** 如果两个模块之间的调用关系复杂，可以加一个适配层（Anti-Corruption Layer），把对方的接口转换为自己的内部语言。这样对方的接口变化只需要改适配层，不影响核心逻辑。

### 5.3 提高内聚的方法

**方法一：按业务概念分模块，而不是按技术功能。** "订单""支付""优惠券"是按业务概念分的。"数据库操作""消息发送""缓存管理"是按技术功能分的。前者通常内聚度更高。

**方法二：把散落的业务规则收拢。** 如果"优惠券使用校验"的逻辑分散在三个 Service 里，应该收拢到 CouponService 或 Coupon 领域对象中。

**方法三：检查模块的公开接口数量。** 如果一个模块的对外接口超过 15-20 个，它可能承担了太多职责，需要拆分。

## 6. 业务案例

### 6.1 案例：订单取消的耦合治理

订单取消的原始实现：

```java
public void cancelOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    order.cancel();
    orderRepository.save(order);
    
    // 以下每一行都是一个直接依赖
    couponService.returnCoupon(order.getCouponId());
    inventoryService.releaseStock(order.getItems());
    pointService.returnPoints(order.getUserId(), order.getPointsUsed());
    memberService.rollbackGrowthValue(order.getUserId(), order.getGrowthValue());
    notificationService.sendCancelNotice(order.getUserId(), orderId);
    reportService.recordCancellation(orderId);
}
```

六个直接依赖。任何一个下游服务出问题，取消操作就失败。

改造后：

```java
public void cancelOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    order.cancel();
    orderRepository.save(order);
    
    // 只发一个事件，不直接依赖任何下游
    eventPublisher.publish(new OrderCancelledEvent(
        order.getId(), order.getUserId(), order.getCouponId(),
        order.getItems(), order.getPointsUsed()
    ));
}
```

优惠券、库存、积分、会员、通知、报表各自独立监听 OrderCancelledEvent，自行处理。

改造前，订单模块依赖六个模块。改造后，订单模块不依赖任何下游模块。下游模块的增减不影响订单取消的核心逻辑。

### 6.2 案例：共享表的拆分

商品模块和营销模块共享 promotion_product 表（记录"哪些商品参加了哪个活动"）。商品模块在"商品参加活动"时写这张表，营销模块在"查看活动商品列表"时读这张表。

问题：商品模块想给表加一个 sort_order 字段（商品在活动中的排序），加完后营销模块的查询 SQL 要跟着改。

拆分方案：
- promotion_product 表归属营销模块。
- 商品模块不再直接写这张表。"商品参加活动"的操作通过营销模块的接口完成：`promotionFacade.bindProduct(promotionId, productId)`。
- 表结构的变更只影响营销模块内部代码。

## 7. 架构判断清单

- 如果一个模块的改动经常需要其他团队配合，说明模块间的耦合度偏高。
- 如果画出模块依赖图后发现循环依赖，说明模块边界划分需要重新审视。
- 如果一个模块被超过五个其他模块直接依赖，需要检查它是否承担了太多职责，或者是否应该通过事件来解耦部分场景。
- 如果两个模块共享了数据库表，需要明确表的归属，并把非属主模块的直接读写改为接口调用。
- 如果一个业务规则的实现散布在三个以上的模块中，说明内聚度不够。
- 如果修改一个接口的入参需要同时发布三个以上的服务，说明接口设计不够稳定或者服务拆分粒度有问题。
- 如果一个简单需求（改个字段、加个校验）的开发周期超过预期三倍以上，大概率是耦合问题导致的协调成本。

## 8. 本章小结

高内聚低耦合是架构设计中最基本的原则，也是最容易说但最难持续做到的原则。因为每次"直接调用"都是当下最快的选择，但这些快速选择累加起来就形成了难以维护的依赖网。

降低耦合的核心方法是：模块之间通过稳定的接口交互、不共享内部数据、用事件解耦不需要同步返回的场景。提高内聚的核心方法是：按业务概念组织代码、把散落的业务规则收拢到对应的模块中。

高内聚低耦合不是一次性工作。随着业务迭代，耦合会自然增加、内聚会自然下降。工程师需要在日常开发中持续关注这两个指标——不需要追求完美，但要在耦合问题变得不可控之前主动治理。
