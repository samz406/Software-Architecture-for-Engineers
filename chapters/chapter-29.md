# 第 29 章：架构演进：系统不是设计出来的，是长出来的

## 1. 从一个真实问题开始

有一个技术负责人接手了一个运行了五年的业务系统。他看了一遍代码和架构，跟 CTO 说："这个系统架构不行，需要推倒重来。"

CTO 问了三个问题：
- 现在系统能不能用？——能用，每天几百万请求，没出大问题。
- 推倒重来要多久？——保守估计半年。
- 重来期间业务需求还做不做？——做不了，全部精力投入重构。

CTO 拒绝了方案。

后来这位技术负责人换了一种方式。他不再追求"理想架构"，而是从系统当前的最大痛点入手：订单查询在高峰期经常超时。他用两周时间做了查询优化和缓存改造，效果立竿见影。然后用两个月的时间，把最耦合的订单和支付逻辑做了模块化拆分。再后来，在一次大的业务调整中顺带完成了数据库拆分。

一年后回头看，系统的架构已经和当初大不一样了。但这个变化不是一次完成的，而是在多次小的改进中逐步演进的。

这就是架构演进的现实：**你几乎不可能一步到位设计出完美的架构，但你可以让架构在每个阶段都做出合适的选择。**

## 2. 问题为什么会变复杂

### 2.1 架构和业务一起成长

系统的第一版架构通常是为当时的业务规模设计的。一个日活几百人的工具，单体 + 单库完全够用。一个日活几万人的电商，可能需要读写分离和缓存。一个日活百万的平台，可能需要微服务和分库分表。

业务在增长，架构也需要跟着变。但架构的变化不是自动发生的——需要有人识别出"当前的架构已经不适合现在的规模了"，然后推动改造。

问题在于这个判断点不好把握。太早改造是浪费（业务还没到那个量级），太晚改造是救火（系统已经开始出问题了）。

### 2.2 技术债在每个阶段都会积累

每个阶段的架构选择都有其合理性。早期选单体是对的——快速迭代、简单部署。但单体里积累的耦合逻辑、大表设计、同步调用，在下一个阶段就变成了技术债。

技术债不是谁犯了错，而是当初的合理选择在新的上下文下变成了负担。一个"够用"的订单表设计，三年后在数据量膨胀和业务规则复杂化后就不够用了。这不是三年前的设计有问题，而是上下文变了。

### 2.3 "一步到位"是最大的幻觉

有些团队在系统早期就想做"最终架构"：上来就微服务、分库分表、全套中间件。结果是：

- 系统复杂度远超当前需要，运维成本很高；
- 团队大部分精力花在维护基础设施上，而不是做业务；
- 很多提前设计的抽象和分层，在实际业务中用不上，或者用的方式和预想的不同。

架构设计需要匹配当前阶段。三个人的团队、每天几百个订单的系统，不需要分库分表。等到团队二十人、每天几十万订单时，再考虑分库分表，到时候对业务的理解也更深，设计也更准确。

## 3. 核心概念解释

### 3.1 架构演进的阶段

一个典型的业务系统，架构演进大致经历以下阶段：

**阶段一：单体 + 单库**

适合：团队 3-5 人，用户量小，业务还在验证阶段。

特点：所有代码在一个项目里，所有数据在一个数据库里。开发部署简单快速。

关注点：代码结构清晰（模块化的单体）、数据库表设计合理。

**阶段二：模块化单体 + 数据优化**

适合：团队 5-15 人，业务开始稳定增长。

特点：代码按业务模块划分包结构，模块之间通过内部接口交互。数据库做了索引优化、读写分离或缓存。

关注点：模块边界清晰、接口契约明确、数据库性能可控。

**阶段三：服务化**

适合：团队 15-50 人，多个业务线并行发展。

特点：核心模块拆分为独立服务，有自己的数据库和部署。服务之间通过 API 或消息通信。

关注点：服务拆分粒度合理、服务治理（注册发现、配置中心、链路追踪）到位、分布式问题（一致性、幂等、补偿）有方案。

**阶段四：平台化**

适合：团队 50+ 人，多条业务线，有共同的底层能力需求。

特点：通用能力抽取为平台（用户平台、支付平台、消息平台），业务线在平台之上构建差异化功能。

关注点：平台和业务的边界清晰、平台接口稳定、平台不成为瓶颈。

每个阶段不是非此即彼的。一个系统可能在某些模块处于阶段二，另一些模块处于阶段三。演进也不一定按顺序来——有些系统可能直接从阶段一跳到阶段三（如果业务增长很快且团队有经验）。

### 3.2 演进的触发信号

怎么知道当前阶段的架构已经不够用了？以下是一些常见的信号：

**需要从阶段一升级的信号：**
- 代码库超过 10 万行，新人两周以上才能上手；
- 不同模块的修改频繁冲突，合并代码成为痛点；
- 数据库开始出现慢查询，加索引已经不够解决问题。

**需要从阶段二升级的信号：**
- 多个团队在同一个代码库里开发，互相阻塞；
- 某个模块需要独立扩容（比如订单查询需要加机器，但其他模块不需要）；
- 发布一个小功能需要部署整个系统，风险过高。

**需要从阶段三升级的信号：**
- 多条业务线在重复建设相同的底层能力；
- 想做跨业务线的数据分析或统一运营，但数据不通。

### 3.3 演进的原则

**原则一：每次只解决当前最痛的问题。** 不要试图一次解决所有问题。找到当前最大的瓶颈，集中精力解决它。

**原则二：改造和交付并行。** 不要停下业务需求去做架构改造。理想的方式是在业务需求中顺带完成架构改进，或者用 20% 的时间做技术改进、80% 的时间做业务交付。

**原则三：为下一步留余地，但不提前建设。** 比如当前在做模块化单体时，模块之间通过接口调用而不是直接依赖——这为将来拆服务留了余地。但不需要现在就搭建服务注册中心。

**原则四：记录每次演进的决策和理由。** 下一个阶段的改造者需要知道当前架构为什么是这样的，才能做出更好的决策。

## 4. 常见误区

### 4.1 追求"一步到位"

前面已经讲过。在系统早期投入大量精力设计"最终架构"，通常是浪费。因为业务还在变化，你对业务的理解还不够深，设计出来的架构很可能和实际需要对不上。

### 4.2 "架构是一次性的"

有些团队认为系统的架构在设计阶段就确定了，之后只需要在这个架构上写功能。这种认知导致架构长期不更新，直到积累了大量的技术债才被迫大改。

架构不是一次性的，而是持续演进的。每次业务规模的增长、团队的扩张、技术的更新，都可能触发架构的调整。

### 4.3 把所有问题都归结为"架构不行"

当系统出了问题——性能差、bug 多、开发效率低——有些人第一反应是"架构要推倒重来"。但很多时候问题的根源不是架构，而是代码质量、测试缺失、流程不规范或者需求管理混乱。

在决定做架构改造之前，先确认问题的真正原因。如果是代码质量问题，加代码规范和 Review 比换架构更有效。

### 4.4 演进路径不连续

有些系统从阶段一直接跳到阶段三（比如从单体直接拆成 20 个微服务），跳过了模块化的过程。结果是：模块边界在单体里就没划清楚，拆成微服务后边界问题不但没解决，反而因为分布式通信变得更复杂。

每个阶段都是下一个阶段的基础。跳过阶段通常会导致基础不牢。

### 4.5 只关注技术演进，忽视组织演进

康威定律说：系统的架构倾向于反映组织的沟通结构。如果团队结构没变，只是把代码拆成了微服务，实际上的协作模式并没有变。

架构演进需要和组织演进配合。拆服务的同时，要让团队按服务来划分，每个团队负责自己的服务的开发、测试和运维。否则拆了服务但还是一个大团队负责所有服务，拆分的价值就打了折扣。

## 5. 设计方法

### 5.1 架构现状评估

在决定演进方向之前，先评估当前架构的现状：

**技术维度：**
- 系统的整体结构是什么？单体/模块化/微服务？
- 数据库的状态如何？单库/读写分离/分库分表？
- 有哪些性能瓶颈？数据库/网络/计算/存储？
- 有哪些稳定性风险？单点故障/没有降级/没有监控？

**业务维度：**
- 当前用户量和增速如何？
- 核心业务流程的复杂度如何？
- 未来半年到一年的业务方向是什么？

**团队维度：**
- 团队规模和技术栈是什么？
- 有多少人熟悉当前系统？
- 团队有能力维护更复杂的架构吗？

### 5.2 确定演进的优先级

不要试图同时解决所有问题。按以下标准排优先级：

1. **业务影响大的优先**：直接影响用户体验或收入的问题（比如支付失败率高）优先于内部效率问题。
2. **风险高的优先**：可能导致系统宕机或数据丢失的问题优先于"不好用但能用"的问题。
3. **成本低收益高的优先**：加个索引就能解决的性能问题，比拆服务优先级更高。

### 5.3 渐进式演进的操作方法

**方法一：搭便车。** 在做业务需求的过程中顺带完成架构改进。比如这次需求要改订单模块的代码，顺便把订单模块的边界理清楚。

**方法二：抽时间。** 每个迭代预留 10-20% 的时间做技术改进。不需要大的审批，但要有明确的目标和验证标准。

**方法三：专项。** 对于较大的架构改造（比如分库分表），需要立项并获得业务支持。但项目范围要尽可能小——只做最痛的部分，不要趁机做"顺便优化"。

## 6. 业务案例

### 6.1 案例：一个电商系统的五年演进

**第一年：单体 + 单库**

三个开发，一个 Spring Boot 项目，一个 MySQL 数据库。所有功能在一个代码库里。部署在一台 4 核 8G 的服务器上。日活几百人。

这个阶段的架构选择完全正确。快速迭代，快速验证业务。

**第二年：模块化 + 缓存 + 读写分离**

团队扩大到八人。代码按业务模块重新组织：order、payment、product、user 四个包，模块之间通过接口调用。数据库加了 Redis 缓存和 MySQL 主从复制。日活几万人。

关键改动：把之前混在一起的订单和支付逻辑拆分为两个模块。虽然还在同一个代码库里，但两个模块的数据表已经各自独立，通过服务层接口交互。

**第三年：核心服务拆分**

团队十五人，分成了订单组和商品组。订单组负责订单、支付、退款，商品组负责商品、库存、搜索。

把单体拆成了两个服务：交易服务和商品服务。各自有独立的数据库和部署。引入了消息队列做异步通信。日活十几万人。

**第四年：数据库拆分 + 更多服务化**

订单表数据量过亿，做了分库分表。支付模块从交易服务中拆出来成为独立的支付服务。用户服务也独立出来。引入了配置中心和链路追踪。日活几十万人。

**第五年：平台化**

公司开了新业务线。用户服务升级为用户平台，支付服务升级为支付平台，新业务线直接接入。引入了 API 网关做统一的认证和路由。日活百万人。

在这五年中，没有一次"推倒重来"。每次架构改动都是在当前的基础上做增量改进，解决当前最痛的问题。

## 7. 架构判断清单

- 如果系统的架构是三年前设计的、从那以后没做过任何调整，而业务规模已经增长了数倍，说明架构演进滞后了。
- 如果团队经常讨论"要不要推倒重来"，说明要么架构确实滞后了，要么团队对渐进式改进的方法不熟悉。
- 如果新人加入后两周以上才能独立开发一个小功能，说明系统的复杂度可能已经超出了当前架构的管理能力。
- 如果团队不到十人但维护着超过五个独立服务，说明可能过早服务化了。
- 如果系统的最大瓶颈在数据库，优先考虑数据层面的优化（索引、缓存、读写分离），而不是急着拆服务。
- 如果做架构改造时说不清楚"为什么现在做、不做会怎样"，说明改造的动机可能不够充分。

## 8. 本章小结

架构不是设计出来的，而是在业务发展过程中逐步演进出来的。一个好的架构不是一步到位的完美设计，而是在每个阶段都做出了合适的选择——不超前，也不滞后。

演进的关键不是技术能力，而是判断力：知道什么时候该动、什么时候该等。改得太早是浪费，改得太晚是救火。最好的时机通常是：系统开始出现痛点，但还没到出大问题的程度。

对工程师来说，理解架构演进意味着两件事。第一，不要在系统早期追求完美架构，先让业务跑起来，在跑的过程中发现真正的问题。第二，不要等到系统病入膏肓才想到改造，在日常开发中保持对系统健康度的关注，发现信号就小步改进。架构能力最终体现在：系统在持续变化中，始终保持着可控和可演进的状态。
