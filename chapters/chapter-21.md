# 第 21 章：中台、平台与能力复用

## 1. 从一个真实问题开始

一家公司有三条业务线：电商、到店、企业采购。三条业务线各自有独立的技术团队。

电商团队做了一套用户系统，支持手机号注册、微信登录、收货地址管理。到店团队也做了一套用户系统，支持手机号登录和位置信息。企业采购团队又做了一套，支持企业账号和员工子账号。

三套用户系统，三个数据库，三种登录逻辑。

当公司想做"跨业务线的会员体系"时发现：同一个手机号在三个系统里对应三个不同的用户 ID，积分不通、等级不通、优惠券不通。想要打通，意味着三个团队的用户数据要做映射和迁移——工作量巨大。

CTO 说："我们需要一个统一的用户中心。"

于是成立了一个平台团队，用半年时间建了一个"用户中台"。统一了注册登录、用户画像、会员等级、积分体系。三条业务线逐步接入。

但上线后，新的问题出现了：

- 电商团队需要在用户资料里加一个"默认收货地址"字段，提了需求给平台团队，排期要等两周；
- 到店团队需要用户支持"到店签到"功能，但平台团队说"签到是业务功能，不属于用户中台"；
- 企业采购团队需要"企业组织架构"和"员工审批链"，平台团队觉得这是 B 端功能，通用用户中台不做。

三条业务线都觉得"用户中台不好用"——它太通用了，满足不了各自的特殊需求；但它又太核心了，绕不过去。

这就是平台化的典型困境：**统一了基础能力，但也成了瓶颈。**

## 2. 问题为什么会变复杂

### 2.1 重复建设的代价

没有平台的时候，每个业务线自己造轮子。短期看效率高——不用等别人，自己说了算。但长期看代价很大：

- 相同的功能被实现多次，开发成本浪费；
- 每套系统的质量参差不齐，有的做了安全加固，有的没做；
- 数据孤岛无法打通，做不了跨业务的分析和运营；
- 公司想做统一体验（比如统一登录、统一消息推送）时，发现底层不通。

### 2.2 过度统一的代价

有了平台之后，另一种代价出现了：

- 平台团队成了瓶颈。所有业务线的需求都要排平台团队的队，平台团队的排期决定了所有业务线的节奏；
- 平台为了"通用"而做了大量的抽象，业务线为了使用平台的通用接口，需要在自己这边做适配和转换；
- 平台的迭代节奏和业务线不同步。平台追求稳定和通用，业务线追求快速响应和灵活定制，两者经常冲突；
- 当平台出了 bug 或性能问题，所有业务线都受影响。

### 2.3 什么时候该做平台、什么时候不该

这是很多公司纠结的问题。太早做平台，业务还不稳定，平台的抽象很可能是错的。太晚做平台，各业务线已经各自发展，统一的成本很高。

一个实用的判断框架是看三个条件：

**第一，这个能力是否在多个业务线中被需要且核心逻辑相似？** 用户注册登录、消息推送、支付——这些在多个业务线中需要，且核心逻辑相似（虽然细节不同）。适合平台化。

**第二，这个能力的变化频率是否远低于使用它的业务？** 用户认证的核心流程一年变不了几次，但使用它的业务功能每周都在变。这种"稳定的底层 + 多变的上层"结构适合平台化。

**第三，团队是否有能力维护一个平台？** 平台需要专门的团队来维护、需要清晰的接口设计、需要版本管理、需要监控和运维。如果公司总共就十几个工程师，专门抽人做平台可能不划算。

## 3. 核心概念解释

### 3.1 平台、中台、能力沉淀的区别

这三个词经常混用，但在实践中含义不同。

**能力沉淀**：把某个重复出现的功能提取为一个可复用的模块或库。比如把短信发送逻辑封装成一个 SDK。这是最轻量级的复用方式，不需要独立的团队。

**平台**：把一类能力封装成一个独立的服务，有自己的接口、数据存储和运维。比如消息平台、支付平台。平台有专门的团队维护，通过 API 为业务方提供服务。

**中台**：中台是一个更大的概念，通常指"把企业的核心业务能力抽象为一组平台化的服务"。比如用户中台、交易中台、数据中台。中台的目标不只是技术复用，还包括业务能力的标准化和产品化。

在实际操作中，很多公司说的"建中台"，做的其实是"建平台"。真正的中台需要业务标准化的支撑——如果三条业务线的业务模式差异很大，强行统一到一个中台里，反而会成为束缚。

### 3.2 平台化的本质

平台化的本质是：**把稳定、高频、可复用的能力，从业务系统中剥离出来，变成一个独立的、有清晰接口的服务。**

关键词有三个：

**稳定**：这个能力的核心逻辑不经常变。如果一个能力每周都在改，它不适合平台化，因为每次改动都会影响所有使用方。

**高频**：这个能力被多个场景频繁使用。如果只有一个业务方用，那直接做在业务系统里就好，不值得独立出来。

**可复用**：多个业务方对这个能力的需求大体一致，只是在细节上有差异。如果每个业务方的需求完全不同，强行统一反而增加复杂度。

### 3.3 平台和业务的关系

平台和业务之间是"服务-消费"的关系，但这个关系需要仔细管理：

- **平台提供标准能力，业务做差异化。** 比如支付平台提供标准的支付和退款能力，但"满减优惠""分期付款"这些业务规则由业务系统负责。
- **平台接口要稳定，但可以通过扩展机制支持差异化。** 比如消息平台提供标准的发送接口，同时支持自定义模板和自定义回调。
- **平台的变更需要考虑向后兼容。** 平台改接口不能随意改，因为有多个调用方。需要版本管理和平滑过渡。

## 4. 常见误区

### 4.1 "先建平台后建业务"

有些公司在业务还没跑通的时候就开始建平台。理由是"以后肯定要用，提前建好"。

问题是：业务还没稳定时，你不知道平台应该抽象成什么样。等业务跑了一两年，发现平台的抽象和业务实际需求对不上，改平台的成本比从头做还高。

**平台应该从业务中"长出来"，而不是在业务之前"设计好"。** 正确的节奏是：先让业务跑起来，在两三个业务线中发现重复的能力，验证了确实有复用价值后，再抽取为平台。

### 4.2 平台承载了太多业务逻辑

当平台团队为了"服务好业务方"，把越来越多的业务逻辑放进平台时，平台就从"通用能力"变成了"业务系统"。

比如支付平台本来只做支付和退款。后来某个业务线说"帮我算一下手续费"，支付平台加了手续费计算。另一个业务线说"帮我做分账"，支付平台又加了分账。再后来"帮我对账""帮我生成财务报表"……支付平台变成了半个财务系统。

**平台应该只做通用的技术能力，不做特定的业务逻辑。** 如果某个功能只有一个业务方需要，它不应该放在平台里。

### 4.3 平台团队和业务团队脱节

平台团队如果不了解业务，设计出来的接口和能力可能不贴合实际需求。平台团队如果只听某一个业务线的需求，设计可能过于特化。

好的做法是：平台团队和主要业务方保持定期沟通，理解各业务方的共性需求和差异需求。平台的能力规划由业务需求驱动，而不是由技术趣味驱动。

### 4.4 忽视平台的运维成本

平台一旦上线，就有持续的运维压力：

- 所有业务方的请求都经过平台，平台的稳定性要求比任何单个业务系统都高；
- 平台的接口变更需要考虑所有调用方的兼容性；
- 平台出了问题，影响面是全局的。

如果公司没有足够的人力来持续维护平台，宁可不做。一个没人维护的平台比没有平台更糟糕。

### 4.5 把所有东西都平台化

有些公司觉得"平台化是好事"，然后把所有能力都做成平台：用户平台、商品平台、订单平台、库存平台、营销平台、消息平台、权限平台、日志平台、配置平台……

结果是平台太多，业务开发一个功能要对接七八个平台。每个平台都有自己的接口风格、错误码规范、文档格式。集成成本比自己实现还高。

**平台化应该有选择地做。** 只有真正高频复用、核心通用的能力才值得平台化。其他能力做成团队内部的公共模块或 SDK 就够了。

## 5. 设计方法

### 5.1 判断一个能力是否值得平台化

用以下问题来评估：

1. **有多少个业务方需要这个能力？** 至少两个以上才有复用价值。
2. **各业务方对这个能力的核心需求有多相似？** 如果相似度低于 70%，平台化的抽象成本可能大于收益。
3. **这个能力的变化频率如何？** 核心逻辑稳定、只是配置不同的能力适合平台化。核心逻辑频繁变化的不适合。
4. **有没有团队能长期维护这个平台？** 至少需要 2-3 人的专职团队。
5. **不做平台的替代方案是什么？** 是共享代码库、是 SDK、还是各自实现？评估替代方案的成本。

### 5.2 平台的接口设计原则

**接口要简单、稳定、自描述。**

- 接口的入参和出参要有清晰的语义，调用方不需要理解平台内部实现；
- 接口要做版本管理。新版本上线后，旧版本至少保留一个迭代周期；
- 接口的错误码要有明确含义，调用方可以根据错误码做不同的处理；
- 提供 SDK 封装常用的调用逻辑，降低接入成本。

**平台要提供扩展机制，而不是把所有需求都做进核心。**

比如消息平台的核心能力是"发送消息"。不同业务方对消息模板、发送策略、失败处理的需求不同。平台可以提供模板引擎和回调机制，让业务方自定义模板和处理失败的逻辑，而不是在平台里为每个业务方硬编码一套逻辑。

### 5.3 平台化的演进路径

**阶段一：共享代码**

在多个项目中使用同一个 library/SDK。比如把短信发送逻辑封装成一个 Maven 包，各项目引用。成本最低，但各项目独立部署，数据不通。

**阶段二：独立服务**

把能力抽取为一个独立服务，通过 API 提供能力。数据集中存储和管理。需要专人维护。

**阶段三：平台产品化**

在独立服务的基础上，提供管理后台、监控大盘、自助接入流程、文档中心。平台从"技术组件"变成"内部产品"。

不一定要一步到位走到阶段三。根据实际需要，在某个阶段停下来也是合理的。

## 6. 业务案例

### 6.1 案例：支付中心的平台化演进

一家公司最初只有电商业务，支付逻辑写在订单服务里。后来开了到店业务（扫码支付）和企业服务（企业转账），三个业务线各自对接支付渠道。

**痛点显现：**
- 微信支付和支付宝的对接代码被复制了三份，每份都略有不同；
- 渠道升级 SDK 时，三个团队各自升级，进度不一致；
- 财务要对账，需要从三个数据库里分别拉数据；
- 风控策略没有统一，电商做了限额检查，到店没做。

**第一步：抽取公共 SDK**

把支付渠道的对接代码封装成一个公共 SDK，三个业务线引用同一个 SDK。解决了代码重复和渠道升级的问题，但数据还是各自存储。

**第二步：建立支付服务**

把支付的核心能力（创建支付单、发起支付、处理回调、退款）抽取为一个独立的支付服务。三个业务线通过 API 调用支付服务。支付数据集中存储，对账变得简单。

支付服务的接口设计：

```
POST /payments/create      -- 创建支付单
POST /payments/pay          -- 发起支付
POST /payments/callback     -- 处理回调（内部回调，由支付渠道触发）
POST /payments/refund       -- 发起退款
GET  /payments/{id}         -- 查询支付状态
```

**第三步：增加管理能力**

- 支付服务加了统一的风控检查（限额、频率、黑名单）；
- 加了运营后台，可以查看支付数据、手动处理异常订单；
- 加了对账模块，自动比对渠道数据和平台数据；
- 加了多渠道路由，可以根据配置自动选择渠道。

**平台边界的管理：**

支付平台只做"支付和退款"。以下需求被拒绝放入支付平台：
- "帮我计算手续费"——这是财务逻辑，不是支付逻辑；
- "帮我做分账"——分账规则和业务强相关，不够通用；
- "帮我出财务报表"——这是报表系统的职责。

这些需求由业务系统自己处理，或者由专门的财务系统处理。支付平台只提供原始的支付和退款数据。

## 7. 架构判断清单

- 如果同一个能力在三个以上的项目中被重复实现，说明有平台化的候选。
- 如果把一个能力平台化后，只有一个业务方使用，说明平台化过早或不必要。
- 如果平台的接口经常因为某个业务方的特殊需求而修改，说明平台的抽象层次不对，可能混入了业务逻辑。
- 如果业务方觉得"接平台比自己做还慢"，说明平台的接入成本太高或平台团队的响应速度太慢。
- 如果平台团队不了解任何一个业务方的业务流程，说明平台和业务脱节了。
- 如果公司有十个以上的"平台"，但每个平台只有一个人维护，说明平台化过度了。
- 如果撤掉平台后业务方能在一周内用其他方式替代，说明这个平台的不可替代性不够，可能不需要独立存在。

## 8. 本章小结

平台化和中台建设解决的是能力复用的问题。当多个业务线需要相同的底层能力时，平台可以避免重复建设、统一数据和标准、降低维护成本。

但平台化不是免费的。它引入了新的成本：平台团队的人力、接口设计和版本管理的复杂度、业务方对平台的依赖和等待、平台变更影响所有调用方的风险。

做不做平台、什么时候做、做到什么程度，取决于业务的实际需要：有多少业务方需要、核心逻辑有多相似、变化频率如何、团队是否有能力维护。答案不是固定的，需要在业务发展的过程中持续评估和调整。

一个好的平台应该让业务方觉得"用起来比自己做更方便"。如果业务方的普遍感受是"对接平台比自己做还复杂"，那要么是平台的抽象不对，要么是平台化本身不适合当前阶段。
