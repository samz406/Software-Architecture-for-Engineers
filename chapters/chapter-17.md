# 第 17 章：单体架构：不是落后，而是简单有效

## 1. 从一个真实问题开始

一个创业团队拿到了天使轮融资，开始做一个 B 端 SaaS 产品。CTO 之前在大厂做过微服务架构，于是在项目启动时就规划了一套"完整"的技术方案：

- 用户服务、权限服务、订单服务、支付服务、通知服务——五个微服务。
- 每个服务一个 Git 仓库、一个数据库、一个 Docker 镜像。
- 加上 API 网关、服务注册中心（Nacos）、配置中心、链路追踪（Jaeger）、日志收集（ELK）。

团队一共四个后端工程师。

三个月后的状况：

- 产品的核心功能才做了 60%。因为大量时间花在了搭建和维护基础设施上——服务之间的调用链路配置、Docker 编排、日志收集配置、注册中心的运维。
- 联调成本很高。一个完整的功能涉及三个服务的协调开发。工程师 A 改了用户服务的接口，工程师 B 的订单服务联调失败，来回沟通花了两天。
- 本地开发环境很重。启动所有服务需要 16G 内存。新人加入后花了三天才把开发环境跑起来。
- 服务之间的事务一致性问题频繁出现。创建订单时要调用户服务和支付服务，任何一个调用失败都可能导致数据不一致。

投资人问进展，CTO 说"基础架构搭得差不多了"。投资人问"用户能用的功能做了多少？"

后来团队做了一个痛苦的决定：把五个微服务合并回一个单体。合并后开发效率提升了两倍多，三周内完成了之前一个月的功能量。

这不是个别案例。在项目早期，**微服务带来的复杂度远超它解决的问题**，而单体架构在很多场景下是更好的选择。

## 2. 问题为什么会变复杂

### 2.1 "微服务"成了默认选项

不知道从什么时候开始，"做微服务"变成了技术团队的默认选择。招聘 JD 里写着"微服务架构经验"，技术分享讲的是"我们如何做微服务拆分"，架构评审的第一个问题是"服务怎么拆"。

这种氛围导致单体架构被视为"落后的""临时的""早晚要换的"。很多团队在不需要微服务的阶段就上了微服务，付出了不必要的复杂度成本。

### 2.2 微服务的隐性成本

微服务架构引入的不只是"拆分"，还有一整套基础设施和运维要求：

- **服务治理**：注册发现、配置中心、健康检查。
- **通信**：服务间 RPC/HTTP 调用、序列化/反序列化、超时重试。
- **数据一致性**：跨服务的事务如何保证？分布式锁？消息队列？补偿机制？
- **可观测性**：分布式链路追踪、集中日志收集、统一监控。
- **部署**：多个服务的独立打包、部署、回滚。CI/CD 流水线的复杂度成倍增加。
- **测试**：端到端测试需要启动所有依赖的服务。Mock 和契约测试的维护成本。

这些成本在大团队（几十人以上）中可以被分摊，但在小团队中就是沉重的负担。

### 2.3 单体的优势被低估了

单体架构有很多被忽视的优势：

- **开发简单**：所有代码在一个项目里，IDE 可以全局搜索、全局重构、一键调试。
- **部署简单**：一个包、一个进程、一次部署。回滚也是一次操作。
- **调用简单**：模块之间是进程内的方法调用，不需要网络通信、序列化、服务发现。
- **事务简单**：一个数据库、一个事务，天然保证一致性。
- **调试简单**：一条调用链在一个进程里，日志在一个文件里，断点可以从头跟到尾。

## 3. 核心概念解释

### 3.1 什么是单体架构

单体架构（Monolithic Architecture）是指整个应用作为一个部署单元运行。所有的功能模块——用户管理、订单处理、支付、通知——都在同一个进程中，共享同一个数据库。

单体不等于"所有代码混在一起"。一个好的单体应该是**模块化的单体**——代码按业务模块组织，模块之间通过接口交互，只是部署在同一个进程中。

### 3.2 模块化单体

模块化单体是单体架构的最佳实践。它保留了单体的简单性，同时通过代码层面的模块化避免了"大泥球"问题。

```
com.example.app/
    order/          -- 订单模块
        api/        -- 对外接口
        service/    -- 内部逻辑
        repository/ -- 数据访问
    payment/        -- 支付模块
        api/
        service/
        repository/
    user/           -- 用户模块
        api/
        service/
        repository/
    shared/         -- 共享基础设施
        config/
        util/
```

模块化的关键规则：
- 每个模块有明确的对外接口（api 包），其他模块只能通过接口访问。
- 模块内部的类（service、repository）不对外暴露。
- 模块之间不直接共享数据库表。虽然物理上在同一个数据库中，但逻辑上每个模块管理自己的表。

### 3.3 单体的适用条件

单体架构适合以下场景：

**团队规模小**：10 人以下的后端团队。大家在同一个代码库里工作，沟通成本低，不需要通过服务拆分来隔离团队。

**业务还在探索期**：产品还没有找到 PMF（Product-Market Fit），功能在频繁调整。这时候需要的是快速迭代的能力，而不是完善的基础设施。

**业务复杂度可控**：系统的核心业务逻辑不至于让一个代码库膨胀到无法管理的程度。通常 20-30 万行代码以内的系统，单体是完全可以管理的。

**流量和数据量可控**：单库单表能支撑当前和短期未来的数据量。不需要分库分表或独立扩容某个模块。

### 3.4 单体什么时候不够用

当以下情况出现时，说明单体可能需要向服务化演进：

- **团队超过 15-20 人**，在同一个代码库里频繁冲突，合并代码成为痛点。
- **某个模块需要独立扩容**。比如订单查询的流量是其他功能的十倍，需要单独加机器。
- **发布耦合严重**。改了一个小功能需要发布整个系统，发布频率被拖慢。
- **代码库超过 30 万行**，新人理解成本很高，模块边界开始模糊。

## 4. 常见误区

### 4.1 "单体就是技术债"

单体不是技术债。技术债是在当前选择中为未来埋下了成本。如果当前阶段单体是最合适的选择，它就不是债。

一个设计良好的模块化单体，在需要拆服务的时候可以相对平滑地拆分——因为模块边界已经划清了，拆分只是把模块从进程内通信改为网络通信。

真正的技术债是：一个没有模块化的单体，代码混成一团，到了需要拆分的时候发现拆不开。所以关键不是"用不用单体"，而是"单体内部有没有良好的模块化"。

### 4.2 "单体不能横向扩展"

单体可以横向扩展。部署多个实例，前面加负载均衡器，就是横向扩展。一个单体应用部署十个实例，处理能力是一个实例的接近十倍。

单体不能做的是**按模块独立扩容**。如果订单查询需要十个实例的处理能力，但用户管理只需要两个，单体没法区分——只能所有模块一起扩到十个实例。

但如果系统还没到需要按模块独立扩容的规模，这个限制根本不是问题。

### 4.3 "微服务一定比单体好"

微服务解决的是**大规模团队协作和独立扩展**的问题。它不解决代码质量、架构设计、业务理解的问题。

一个代码混乱的单体，拆成微服务后不会变得清晰——它只是变成了多个代码混乱的小项目，加上分布式通信的复杂性。

如果模块边界在单体里都划不清楚，拆成微服务只会更混乱。

### 4.4 过早拆分

有些团队在系统第一版就按微服务架构设计。五个人的团队维护八个服务。每个人要维护一两个服务，同时还要理解其他服务的接口和数据模型。

过早拆分的代价：
- 基础设施投入过大，挤占了业务开发时间。
- 服务间通信引入了不必要的复杂度。
- 分布式事务问题在业务还不复杂时就出现了。
- 团队在服务治理上花的时间比写业务代码还多。

### 4.5 "先微服务，将来好维护"

有人说"现在就微服务，将来扩展的时候就不用重构了"。但"将来"什么时候来不确定，而微服务的运维成本现在就要付出。

更好的策略是：现在用模块化单体快速迭代，同时在代码层面保持良好的模块边界。等到确实需要拆分的时候，从模块化单体拆出微服务的成本比从零搭微服务架构低得多。

## 5. 设计方法

### 5.1 如何设计一个好的单体

**第一步：按业务模块划分代码结构。** 不是按技术层（controller/service/dao）划分，而是按业务领域（order/payment/user）划分。每个业务模块内部再按技术层组织。

**第二步：定义模块间的接口。** 每个模块对外暴露一组 Facade 接口。其他模块通过 Facade 调用，不直接调用内部类。

**第三步：数据归属清晰。** 虽然所有表在同一个数据库里，但每张表有明确的属主模块。其他模块不直接读写不属于自己的表。

**第四步：为将来的拆分留余地。** 模块之间通过接口而不是直接依赖来交互。如果将来需要把某个模块拆成独立服务，只需要把接口从进程内调用改为网络调用。

### 5.2 单体的代码质量维护

单体最大的风险不是"不能扩展"，而是"代码腐化"。随着代码量增长，模块边界逐渐模糊，依赖关系变得混乱。

**定期检查模块依赖。** 用工具（如 ArchUnit）自动检测是否有跨模块的非法依赖。比如"支付模块不能直接访问订单模块的内部类"。

**代码审查关注边界。** 每次 Code Review 时检查：这个改动有没有跨越模块边界？如果跨了，是必要的还是可以通过接口解决？

**控制模块大小。** 如果一个模块的代码量超过了其他模块的三倍以上，考虑拆分。

## 6. 业务案例

### 6.1 案例：一个创业团队从微服务回归单体

一个做 SaaS CRM 的团队，初始架构是五个微服务。四个后端开发，半年时间基础设施搭建消耗了 40% 的人力，核心功能进展缓慢。

回归单体后的变化：
- 开发效率提升约两倍。一个功能从需求到上线的时间缩短了 60%。
- 本地开发环境从"启动五个服务 + 依赖组件"变成"启动一个 Spring Boot 应用"。
- 数据一致性问题消失了。之前跨服务的事务问题需要消息队列和补偿机制，现在一个数据库事务搞定。
- 调试时间大幅减少。之前追踪一个问题要看五个服务的日志，现在看一个日志文件就行。

但他们也做了一件重要的事：在合并回单体时，保持了模块化的结构。每个原来的微服务变成了单体里的一个模块，模块之间通过接口交互。这样将来如果需要再拆，有清晰的边界可以拆。

### 6.2 案例：一个中型系统的单体演进路线

一个电商后台管理系统。初始四个人，用 Spring Boot 做了一个单体应用。

**第一年（4 人）**：单体 + 单库。功能模块：商品管理、订单管理、用户管理、报表。所有模块在一个代码库里，代码量约 5 万行。运行得很好。

**第二年（8 人）**：模块化单体。代码按模块重新组织，每个模块有自己的 Facade 接口。开始出现模块间的编译依赖问题，用 Maven 多模块结构强制隔离。代码量约 12 万行。

**第三年（15 人）**：把订单模块拆成独立服务。原因是订单查询的流量是其他功能的五倍，需要独立扩容。拆分过程比较顺利，因为模块边界在第二年就已经划清了——订单模块的 Facade 接口直接变成了 RPC 接口。

**第四年（20 人）**：支付模块也独立出去了。理由是支付对接外部渠道，稳定性要求特别高，需要独立部署和独立发布。其他模块仍然在单体中。

这种渐进式的演进方式比一开始就微服务要稳妥得多。每次拆分都是在确实需要的时候才做，而且因为有良好的模块边界，拆分成本可控。

## 7. 架构判断清单

- 如果团队不到 10 人，且没有明确的独立扩容需求，单体大概率是更好的选择。
- 如果团队花在基础设施维护上的时间超过了业务开发时间的 30%，说明架构对当前阶段来说太重了。
- 如果一个完整功能的开发需要协调三个以上的服务团队，在小团队中说明拆分过度了。
- 如果系统是单体但代码没有按模块化组织，说明需要先做模块化，而不是急着拆微服务。
- 如果产品还在频繁调整方向，单体的灵活性比微服务的"正确性"更重要。
- 如果单体的某个模块的流量是其他模块的五倍以上，可以考虑把这个模块独立出去。

## 8. 本章小结

单体架构不是落后的、临时的、需要被替换的架构。在团队规模小、业务还在探索、复杂度可控的阶段，单体架构是最高效的选择。它简单、快速、易于调试和部署。

单体架构的关键不是"把所有代码放在一起"，而是在一个部署单元中保持良好的模块化。模块边界清晰的单体，在需要拆分时可以平滑演进为微服务。模块边界混乱的单体，不管怎么拆都是泥潭。

对工程师来说，选择架构方案时不应该问"什么架构最先进"，而应该问"什么架构最适合当前阶段"。用简单的方案解决当前的问题，把复杂的方案留到真正需要的时候——这本身就是一种架构能力。
