# 第 2 章：架构到底解决什么问题？

## 1. 从一个真实问题开始

周一上午，产品提了一个看起来不大的需求：订单列表增加“是否可开发票”标记，并支持按该条件筛选。

看起来只是加个字段。结果一评估，事情很快变味：

- 可开发票依赖订单状态、支付状态、退款状态；
- 部分历史订单没有完整支付渠道信息；
- 开票资格在 B 端和 C 端规则不同；
- 财务系统已有一套离线判定逻辑；
- 客服后台和用户端都要改，且口径必须一致。

最后这个需求不是改一处，而是牵动了订单、支付、退款、财务、客服五条链路。真正的问题不是“字段怎么加”，而是：**系统是否有能力承接跨域规则变化**。

这就是本章要回答的核心：架构究竟在解决什么问题？

## 2. 问题为什么会变复杂

### 2.1 表面是功能，背后是系统性问题

工程师常把复杂性归因到“需求变态”。但很多时候，需求只是触发器，真正复杂的是系统内部结构：

- 规则散落在多个服务和脚本中；
- 同一业务概念在不同系统里定义不一致；
- 接口只传数据，不传语义和约束；
- 历史兼容逻辑无限延期。

于是每次变更都像拆盲盒：你不知道会碰到哪些隐含依赖。

### 2.2 五类高频复杂性来源

本书把架构要解决的问题归纳为五类：

1. **复杂性问题**：系统规模上来后，没人能完整理解结构；
2. **变化问题**：需求持续变化，改动成本失控；
3. **协作问题**：多团队并行开发，边界与责任不清；
4. **质量问题**：性能、稳定性、安全性没有工程化保障；
5. **演进问题**：今天能跑，但没有未来升级路径。

这五类问题不是分开的。真实项目里它们会互相放大。

### 2.3 为什么“能跑”不等于“可持续”

很多系统在上线初期都能跑，问题在于后续：

- 需求新增速度 > 结构治理速度；
- 团队人数增长速度 > 共识形成速度；
- 历史数据累积速度 > 模型演进速度。

当这三个不等式长期成立，系统就会从“可维护”滑向“靠经验维持”。

## 3. 核心概念解释

### 3.1 架构的目标不是功能交付，而是系统可控

功能交付关心“这个需求能否上线”。
架构关心“上线后系统是否仍可理解、可修改、可扩展、可恢复”。

所以架构价值是“控制系统行为的不确定性”。

### 3.2 用五个视角理解架构问题

#### 视角一：结构可理解性

任何新同学加入后，能否在有限时间内理解核心流程与边界？如果不能，系统已进入高认知成本状态。

#### 视角二：变化承接能力

需求变更时，是否总是牵连多模块、多团队、多表联动？牵连面越大，说明系统缺少局部演进能力。

#### 视角三：协作摩擦成本

接口是否稳定？责任是否明确？跨团队问题能否快速定位 owner？协作摩擦长期高企，会直接拖慢业务速度。

#### 视角四：质量保障机制

性能、稳定性、安全性是否是“靠人盯”还是“靠机制守”？靠人盯意味着不可复制。

#### 视角五：演进路径清晰度

是否有明确的“下一阶段如何升级”的路线，比如从单体到模块化、从同步耦合到事件协作。没有路径，系统只能被动补丁。

### 3.3 架构工作的本质输出

架构不是输出一张图，而是输出四类可执行结果：

- 清晰边界（谁负责什么）；
- 稳定契约（系统如何协作）；
- 关键约束（哪些事情不能做）；
- 演进计划（下一步怎么改）。

## 4. 常见误区

### 误区一：把架构问题等同于中间件选型

“我们上了 MQ、Redis、ES，就有架构了。”这是一种常见错觉。中间件只是工具，无法替你定义边界和职责。结构问题不解决，工具只会放大复杂性。

### 误区二：只讨论功能，不讨论非功能

很多评审只看“流程对不对”，很少问：

- 峰值流量如何处理？
- 下游超时如何降级？
- 幂等和重试怎么设计？
- 数据回溯与审计如何支持？

这些非功能问题，往往才是线上事故来源。

### 误区三：把历史兼容当永久方案

“先加个兼容分支，后面再清理。”如果没有明确下线时间，所谓兼容会变成永驻逻辑。几年后谁也不敢删，系统越来越重。

### 误区四：追求统一，忽略差异

有些团队热衷“一个模型走天下”。结果是 B 端、C 端、渠道侧规则硬塞在同一抽象里，模型名义统一，语义却越来越混乱。

### 误区五：发生故障后只修 bug，不改机制

比如某次重复回调导致重复退款，修复方式只是“加 if 判断”。下次出现顺序颠倒、消息重放、超时补偿，又会在别处炸。真正要改的是幂等键、状态机和补偿机制。

## 5. 设计方法

本章给出一套“架构问题识别与应对”方法，适合在需求评审和方案设计阶段使用。

### 5.1 五问法：先识别问题类型

拿到需求后，先问：

1. 这是功能增量，还是规则重构？
2. 变更影响哪些边界？是否跨团队？
3. 哪些非功能指标会受影响（性能、稳定性、安全、可审计）？
4. 历史兼容要持续多久？退出条件是什么？
5. 这次改动是否为下一步演进创造条件，还是继续透支系统？

### 5.2 架构风险清单（评审必过）

- 是否有单点关键路径无降级方案；
- 是否存在跨服务共享数据库写入；
- 是否定义幂等键与重复处理策略；
- 是否给出回滚与灰度方案；
- 是否明确监控指标与告警阈值；
- 是否补充 ADR 记录关键取舍。

### 5.3 取舍模板（避免空谈）

每个关键决策至少写清：

- 目标：要解决什么问题；
- 方案 A/B：核心差异；
- 约束：时间、人力、风险、历史包袱；
- 选择理由：为什么此刻选这个；
- 后果：接受哪些代价；
- 触发条件：何时需要重审决策。

这套模板能把“观点争论”变成“上下文决策”。

## 6. 业务案例

继续用“可开发票标记”需求，看两种不同推进方式。

### 6.1 方式一：字段驱动

团队直接在订单表加字段 `invoiceable`，并在多个入口分别更新：

- 下单时写初值；
- 支付成功后改值；
- 退款后再改值；
- 财务任务夜间修正。

短期上线快，但很快出现口径不一致：

- 前台显示可开票，财务判定不可开；
- 退款回滚后字段未更新；
- 历史任务覆盖了新规则结果。

### 6.2 方式二：规则驱动 + 状态派生

改用“核心状态 + 规则引擎函数”策略：

- 不直接持久化最终资格字段（或仅做缓存快照）；
- 以订单状态、支付状态、退款状态、渠道类型作为输入；
- 统一由资格判定模块计算，并对外提供稳定接口；
- 前台、客服、财务共用同一判定结果与规则版本。

并配套：

- 规则版本号，支持回溯；
- 关键变更写审计日志；
- 历史数据回放验证新规则影响范围。

这套方案初期工作量更大，但它解决的是“规则一致性与演进能力”，而不只是“加个字段”。

### 6.3 关键收益

- 规则变更集中管理，减少散点修改；
- 口径一致，跨团队沟通成本下降；
- 支持历史回溯，降低财务争议；
- 为后续“按地区/渠道差异化开票”留出扩展位。

## 7. 架构判断清单

- 如果一个需求评估时主要时间花在“找影响面”，说明结构可理解性不足。
- 如果需求变更总依赖“某位老同学经验”，说明知识未工程化沉淀。
- 如果同一业务概念在多个系统里定义不同，先统一语言再开发。
- 如果上线方案没有回滚与监控，说明风险管理缺位。
- 如果每次故障都修局部而不补机制，系统会重复付学费。
- 如果方案只讲功能流程，不讲约束与后果，不算完整架构方案。

## 8. 本章小结

第一，架构要解决的不是“做一个高级系统”，而是让系统在持续变化里保持可控。可控意味着你知道系统为何这样设计、出了问题如何定位、下一步如何演进。

第二，复杂性、变化、协作、质量、演进这五类问题，会在真实项目中交织出现。工程师必须学会先识别问题类型，再选择策略，而不是上来就讨论技术名词。

第三，架构工作最怕两件事：一是只谈概念不落地，二是只顾眼前不看后果。前者会让方案变成文档摆设，后者会让系统变成技术债放大器。

第四，从今天开始，你可以用“五问法 + 风险清单 + 取舍模板”做最小架构实践。它不需要复杂流程，但能明显提高团队决策质量。

第五，后续章节会继续展开：如何做抽象、怎么划边界、如何建模状态、如何设计接口与一致性机制。你会看到，架构不是玄学，它是一套可训练、可复用、可验证的工程能力。
