# 第 4 章：抽象能力：从具体到本质的提炼

## 1. 从一个真实问题开始

产品经理提了一个需求："优惠券增加一个'仅限新用户使用'的功能。"

工程师看了一下代码，优惠券表 coupon 已经有二十多个字段了。他加了一个 is_new_user_only 的布尔字段，在核销逻辑里加了一段判断，测试通过，上线了。

两周后，产品说："再加一个'仅限会员使用'。"工程师又加了一个 is_member_only 字段。

一个月后："加一个'指定商品类目可用'。"加了 allowed_category_ids 字段。

再后来："加一个'满 200 减 50'的门槛。"加了 min_amount 字段。

再后来："加一个'仅限周末可用'。"加了 available_days 字段。

半年过去，优惠券表有了三十多个字段，核销方法里有二十多个 if-else 判断。每次加新的使用条件，工程师要：
- 在数据库加字段；
- 在创建接口加参数；
- 在核销逻辑里加判断分支；
- 在运营后台加表单项；
- 在券详情页加展示逻辑。

改动散落在五个地方，每次都走一遍全链路。

后来一个资深工程师接手了这块代码。他没有继续加字段，而是做了一件事：把所有使用条件抽象为"规则"。一个优惠券可以关联多条规则，每条规则有类型和参数。核销时遍历所有规则，逐条校验。

之后再加新的使用条件，只需要新增一个规则类型的实现。数据库结构不用改，核销主流程不用动，运营后台的表单可以根据规则类型动态渲染。

这个资深工程师做的事情就是**抽象**——从一堆具体的字段和判断中，提炼出"使用规则"这个统一的概念。

## 2. 问题为什么会变复杂

### 2.1 需求是一个一个来的，但系统是一个整体

产品经理不会一次告诉你"优惠券将来会有二十种使用条件，请设计一个通用的规则引擎"。需求是一个一个来的，每次看起来都很小：加个字段、加个判断、加个参数。

但这些"小需求"在代码里是累加的。第一个 if-else 没问题，第五个开始不舒服，第十个的时候代码已经不可维护了。

问题在于：如果每次都按"最快实现这个具体需求"的方式写代码，系统会变成一堆具体实现的堆叠，没有结构，没有弹性。

### 2.2 缺乏抽象的系统，修改成本线性增长

一个没有抽象的系统，每次新增功能的成本大致相同甚至递增。因为每次修改都要直接接触具体的实现细节。

有了好的抽象之后，新增功能的边际成本会下降。新增一种优惠券规则，只需要实现一个新的规则类，不需要动主流程。这就是抽象的收益——**用结构换效率**。

但抽象也有成本。建立抽象需要理解多个具体场景的共性，需要设计合理的扩展点，需要团队理解和遵循这个抽象。过早抽象（还没有足够的具体场景就开始抽象）或者过度抽象（把简单的东西复杂化）都是常见的问题。

### 2.3 抽象能力是架构能力的起点

架构设计的核心工作是划分职责和管理复杂性。而划分职责的前提是识别出系统中的核心概念——什么东西是独立变化的、什么东西是稳定的、什么东西可以被统一对待。这些识别工作，就是抽象。

一个不会抽象的工程师，面对复杂需求只能堆代码。一个会抽象的工程师，能在堆代码之前先找到合适的结构，让代码长在结构上。

## 3. 核心概念解释

### 3.1 什么是抽象

抽象就是**忽略不重要的差异，提炼出共同的结构**。

"仅限新用户使用"和"仅限会员使用"和"满 200 可用"，具体逻辑完全不同。但从结构上看，它们都是"优惠券的一个使用前置条件"。把它们统一建模为"使用规则"，就是一次抽象。

抽象的对象可以是：
- **数据**：多个字段抽象为一个概念（多个使用条件 → 规则列表）；
- **行为**：多段相似的逻辑抽象为一个接口（多种通知方式 → 通知策略）；
- **流程**：多个类似的流程抽象为一个模板（多种订单处理 → 订单流程引擎）；
- **关系**：多种依赖关系抽象为一个交互模式（多个下游 → 事件订阅）。

### 3.2 三个层次的抽象

**代码层面的抽象**：用接口、抽象类、策略模式等把具体实现隐藏在统一的契约后面。这是工程师最熟悉的抽象层次。

比如支付渠道的抽象：

```java
public interface PaymentChannel {
    PayResult pay(PayRequest request);
    RefundResult refund(RefundRequest request);
    PayStatus query(String transactionId);
}

public class AlipayChannel implements PaymentChannel { ... }
public class WechatPayChannel implements PaymentChannel { ... }
public class UnionPayChannel implements PaymentChannel { ... }
```

调用方不关心具体走哪个渠道，只面对 PaymentChannel 接口。新增一个渠道，只需要实现这个接口。

**模型层面的抽象**：从业务概念中提炼出核心对象和关系。这比代码抽象更重要，因为它决定了代码的整体结构。

比如上面优惠券的例子，从"一堆字段"到"券模板 + 规则列表"的模型转变。或者从"订单表有一个 status 字段表示所有状态"到"订单状态、支付状态、物流状态分别管理"的模型转变。

**架构层面的抽象**：从系统的职责中提炼出独立的能力单元。比如把"发短信、发邮件、发站内信、发 App 推送"抽象为"消息平台"。把"注册、登录、Token 管理、权限校验"抽象为"认证授权服务"。

三个层次从下到上，影响范围越来越大，改动成本也越来越高。代码层面的抽象可以随时重构，架构层面的抽象一旦确定就很难推翻。

### 3.3 什么时候该抽象

一个实用的经验是**三次法则**：当同一种模式出现三次的时候，考虑抽象。

第一次出现——直接实现，不要过度设计。
第二次出现——注意到相似性，但可以先复制。
第三次出现——该抽象了，因为你已经看到了足够的具体场景来判断共性是什么。

太早抽象的风险是：你只见过一两个场景，提炼出来的"共性"可能是错的。等真正不同的第三个场景出现时，之前的抽象反而成了束缚。

太晚抽象的代价是：具体实现散落各处，重构的工作量变大。

### 3.4 好的抽象和坏的抽象

**好的抽象**有几个特征：
- 概念直觉：听到名字就知道它是什么。"使用规则"比"条件校验器链式处理模块"好懂。
- 稳定接口：抽象的接口不经常变。新增具体实现时不需要改接口定义。
- 对称性：同一抽象下的各个实现，使用方式一致，不需要区别对待。

**坏的抽象**也有特征：
- 泄漏：调用方需要知道抽象背后的具体实现才能正确使用。比如接口定义了 pay() 方法，但调用方在调用前需要根据渠道类型设置不同的参数——这说明抽象不够干净。
- 过度：把本来简单的事情包了七八层，只有作者自己看得懂。两个类就能解决的问题，搞出一套框架。
- 投机：基于猜测的未来需求做抽象，而不是基于已有的多个具体场景。结果是抽象和实际需求对不上。

## 4. 常见误区

### 4.1 把"起一个接口名"当作抽象

有些工程师认为定义了一个 interface 就是做了抽象。但如果接口的方法签名里塞满了特定业务的参数，或者接口只有一个实现，那这个接口只是一层无意义的间接层，不是真正的抽象。

真正的抽象是从多个具体场景中发现共性模式。接口只是表达抽象的一种手段，不是抽象本身。

### 4.2 为了复用而抽象

"这段逻辑将来可能别的地方也用到，提前抽一个工具类吧。"这是很多工程师抽象的动机。

问题是"将来可能"很多时候不会发生。就算发生了，需求可能和你当初预想的不一样。为未发生的复用做抽象，是一种投机行为。

正确的顺序是：先发现重复，再消除重复。不要先预防重复。

### 4.3 过度抽象导致简单问题复杂化

有个团队做消息通知功能。需求很明确：订单状态变化时发短信通知用户，目前只有短信一种渠道。

工程师设计了一套"消息中间件"：消息抽象层、渠道适配层、模板引擎、路由策略、重试队列、消息持久化。代码量从预计的几百行变成了几千行。

结果是：上线后只用了短信渠道，模板也只有三种，路由策略永远返回"短信"。那套抽象大部分是空转的复杂性。

**如果当前只有一个场景，不需要抽象。** 直接写最简单的实现，等第二个、第三个场景出现时再提炼。

### 4.4 抽象层次不对

有些抽象抽到了太底层，把实现细节暴露给了调用方。有些抽象抽到了太高层，在使用时需要传入大量的配置和回调才能工作。

一个好的判断标准是：调用方在使用抽象时，需要了解多少内部细节？了解的越少，抽象越好。

比如文件存储的抽象，好的接口是 `upload(file)` 和 `download(fileId)`。坏的接口是 `upload(file, storageType, region, bucket, accessKey, retryPolicy, compressionAlgorithm)`——这说明抽象没有把复杂性封装住。

### 4.5 抽象之后不维护

抽象建立之后，需要持续维护。业务在变化，最初的抽象可能逐渐不适用。但很多团队建立了抽象之后就不管了，新的需求不断往抽象的实现里塞特殊逻辑，直到抽象名存实亡。

如果一个抽象的实现类里有大量的 `if (type == xxx)` 分支来处理特殊情况，说明这个抽象已经被破坏了。要么重新设计抽象，要么承认这些场景不适合用同一个抽象来处理。

## 5. 设计方法

### 5.1 识别抽象机会

在日常开发中，以下信号提示你可能需要抽象：

- **同样的模式出现了三次**：三种支付渠道、三种通知方式、三种导出格式。
- **新增功能的改动总是散落在多个相同的地方**：每次加一种优惠条件，都要改数据库、改接口、改核销逻辑、改前端。
- **两个看起来不同的东西，在结构上非常相似**：订单取消和退款，流程不同，但都涉及"状态回退 + 资源释放 + 通知"。
- **一段代码的核心逻辑很短，但周围的适配和判断很长**：说明核心逻辑和具体场景没有分开。

### 5.2 提炼抽象的步骤

**第一步：列出所有具体场景。**

以优惠券的使用条件为例：
- 仅限新用户
- 仅限会员
- 满 X 元可用
- 指定商品类目可用
- 指定时间段可用
- 单笔限用一张

**第二步：找出共性结构。**

所有这些条件都是"在核销优惠券时做的一项检查"。每个检查的输入是当前订单信息和用户信息，输出是"通过/不通过"。

**第三步：定义抽象接口。**

```java
public interface UsageRule {
    /**
     * 校验优惠券在当前上下文下是否可用
     * @param context 包含订单信息、用户信息、时间等
     * @return 校验结果，不通过时包含原因
     */
    CheckResult check(UsageContext context);
}
```

**第四步：实现具体场景。**

```java
public class NewUserOnlyRule implements UsageRule {
    public CheckResult check(UsageContext context) {
        if (!context.getUser().isNewUser()) {
            return CheckResult.fail("该优惠券仅限新用户使用");
        }
        return CheckResult.pass();
    }
}

public class MinAmountRule implements UsageRule {
    private Money minAmount;
    
    public CheckResult check(UsageContext context) {
        if (context.getOrder().getAmount().lessThan(minAmount)) {
            return CheckResult.fail("订单金额未达到使用门槛");
        }
        return CheckResult.pass();
    }
}
```

**第五步：在主流程中使用抽象。**

```java
public class CouponService {
    public void redeem(Coupon coupon, UsageContext context) {
        // 遍历所有规则，逐条校验
        for (UsageRule rule : coupon.getRules()) {
            CheckResult result = rule.check(context);
            if (!result.isPassed()) {
                throw new BusinessException(result.getReason());
            }
        }
        // 校验通过，执行核销
        coupon.markAsUsed(context.getOrder().getId());
    }
}
```

新增一种使用条件时，只需要：新增一个 UsageRule 的实现类，在运营后台配置该规则类型。主流程不需要任何改动。

### 5.3 检验抽象质量的标准

做完抽象之后，用以下问题检验：

1. **新增一个具体场景需要改几个地方？** 好的抽象应该只需要改一个地方（新增一个实现类）。
2. **调用方需要了解多少具体实现的细节？** 越少越好。
3. **抽象的名字是否能被团队成员一听就懂？** 如果需要长时间解释，可能太复杂了。
4. **现有的具体实现是否都能自然地放进这个抽象？** 如果有的实现需要大量的适配和变通才能塞进去，说明抽象可能不对。

## 6. 业务案例

### 6.1 案例：支付渠道的抽象演进

第一版实现：代码里直接写了支付宝和微信支付的对接逻辑。两段代码有大量重复——构造请求、发送请求、处理响应、记录日志、处理异常。但具体的参数格式和签名方式不同。

第二版：有人提议"抽一个支付接口"。于是定义了 PaymentChannel 接口。但接口设计有问题——方法签名是 `pay(Map<String, String> params)`，所有参数都塞在一个 Map 里。调用方需要知道每个渠道需要哪些 key，而不同渠道的 key 完全不同。这是一个"泄漏的抽象"，接口虽然统一了，但调用方的代码并没有简化。

第三版：重新设计了抽象。定义了统一的 PayRequest（包含金额、订单号、商品描述等业务字段）和 PayResult（包含渠道流水号、状态等）。每个渠道的实现负责把通用的 PayRequest 转换为自己渠道的格式。调用方只需要构造 PayRequest，不需要知道具体渠道的参数格式。

从第二版到第三版的变化不大，但抽象的质量差别很大。关键区别在于：**抽象的接口应该用业务语言定义（金额、订单号），而不是用技术语言定义（参数 Map）。**

### 6.2 案例：从字段到规则引擎的演进

回到优惠券的例子。团队在第一年用字段方式应付了十几种使用条件。第二年改为规则引擎模式后，新增使用条件的开发时间从平均两天降到了半天。

但规则引擎也带来了新的问题：
- 规则之间的优先级和冲突怎么处理？比如一张券同时配了"满 200 可用"和"指定类目可用"，是 AND 关系还是 OR 关系？
- 规则的参数怎么存储？JSON 字段？关联表？
- 运营后台怎么让非技术人员配置规则？

这些问题在设计抽象时就要考虑。团队的做法是：
- 规则默认是 AND 关系（所有规则都通过才可用），如果需要 OR 关系，用一个 CompositeRule（组合规则）来包装；
- 规则参数用 JSON 存储在规则表的 params 字段里，每种规则类型负责解析自己的 JSON 结构；
- 运营后台按规则类型动态渲染表单，每种规则类型注册自己的表单配置。

这个案例说明一个事实：**抽象不是一次完成的，它会随着业务场景的丰富而逐步完善。** 第一版的抽象不需要完美，只需要比"堆字段"好就行。后续再根据实际使用中遇到的问题迭代。

## 7. 架构判断清单

- 如果一个表有超过 30 个字段且还在持续增长，说明可能需要把部分字段抽象为独立的模型（如规则表、扩展属性表）。
- 如果一个核心方法里有超过 10 个 if-else 分支处理不同的业务场景，说明需要把这些分支抽象为策略或规则。
- 如果新增一种支持的类型（支付渠道、通知方式、导出格式）需要修改三个以上的文件，说明抽象不够。
- 如果接口定义用了 `Map<String, Object>` 作为参数或返回值，说明抽象层次不对。
- 如果一个抽象只有一个实现且半年内看不到第二个，说明可能是过度抽象。
- 如果团队成员说不清某个抽象概念的含义，说明这个抽象要么不够好，要么缺少文档。

## 8. 本章小结

抽象是架构设计的第一能力。它不是高深的理论，而是一种日常的工程习惯：在写具体实现之前，先看看有没有更好的结构。

好的抽象能降低系统的修改成本——新增功能时只需要在已有的结构上扩展，而不是每次都从头到尾改一遍。但抽象不是越多越好。过早抽象、过度抽象、投机性抽象，都会让系统变得更复杂而不是更简单。

对工程师来说，培养抽象能力的方法是：多看具体场景，多找共性模式，多练"等一等再抽象"的耐心。等你见过三个具体场景之后再提炼共性，比你在第一个场景就开始预测未来要靠谱得多。抽象能力不是一种天赋，而是在足够多的具体实践中训练出来的直觉。
