# 第 18 章：微服务架构：拆分之后，复杂性去了哪里？

## 1. 从一个真实问题开始

一个做了四年的电商系统，团队从五人扩到了三十人。单体应用的代码量超过了四十万行。CTO 决定做微服务拆分。

拆分方案很清晰：按业务域拆成订单服务、商品服务、用户服务、支付服务、库存服务、营销服务。每个服务独立仓库、独立数据库、独立部署。

半年后拆分完成了。但新的问题也来了：

- 原来在单体里一个数据库事务就能搞定的"下单扣库存"，现在涉及订单服务和库存服务两个服务的协调。消息队列 + 补偿机制的代码量比原来的业务逻辑还多。
- 原来查订单详情一条 SQL 带几个 join 就搞定了，现在要调订单服务拿订单、调商品服务拿商品信息、调用户服务拿收货地址、调支付服务拿支付状态。四次 RPC 调用，一次网络抖动整个页面就超时。
- 原来排查一个问题看一个日志文件，现在一个请求跨六个服务，日志散落在六台机器上。不上链路追踪根本查不了问题。
- 原来一个人改一个需求提交一次代码，现在一个需求可能涉及三个服务、三个仓库、三次代码审查、三次部署。

技术负责人在复盘会上说了一句话："微服务不是让复杂性消失了，而是让它从代码搬到了基础设施和运维上。"

## 2. 问题为什么会变复杂

### 2.1 微服务解决了什么问题

微服务架构要解决的核心问题是：**大规模团队在同一个系统上的协作效率**。

当团队超过十五到二十人，在同一个代码库里工作的冲突会越来越多：代码合并冲突、发布排队、改一个模块不小心影响另一个模块。微服务通过把系统拆分为独立的服务，让每个团队有自己的代码库、自己的部署节奏、自己的技术选型。

另一个问题是**独立扩容**。系统的不同部分负载不同——订单查询的流量远大于商品管理。微服务允许高负载的服务单独加机器，低负载的服务保持小规模。

### 2.2 微服务引入了什么问题

但微服务不是免费的。它把单体中的"模块间方法调用"变成了"服务间网络调用"，由此引入了一系列分布式系统的问题：

**网络不可靠**：服务间的调用可能超时、丢失、重复。需要重试、超时、幂等、熔断。

**数据不一致**：每个服务有自己的数据库，跨服务的操作无法用单个事务保证一致性。需要最终一致性方案、补偿机制、对账。

**排查困难**：一个请求经过七八个服务，出了问题不知道是哪个服务的问题。需要分布式链路追踪。

**运维复杂**：从管理一个应用变成管理几十个应用。部署、监控、日志、配置都要乘以服务数量。

### 2.3 拆分粒度是最大的难题

拆太粗：两个无关的业务绑在一起，还是单体的老问题。
拆太细：服务间调用太频繁，一个请求跨十个服务，性能和可靠性都成问题。

没有标准答案。拆分粒度取决于业务的耦合程度、团队的规模和组织结构、运维的能力。

## 3. 核心概念解释

### 3.1 微服务的拆分原则

**按业务域拆分**：按业务领域划分服务边界，而不是按技术层。"订单服务""支付服务"是按业务域拆的。"数据库服务""缓存服务"是按技术层拆的——后者通常不是好的拆分方式。

**高内聚低耦合**：拆分后，每个服务内部高度内聚（只做一类事），服务之间低耦合（交互最少化）。

**数据独占**：每个服务拥有并管理自己的数据。其他服务不直接访问它的数据库，只通过 API。

**独立部署**：每个服务可以独立开发、测试、部署，不需要和其他服务协调。

### 3.2 服务间通信

**同步通信（RPC/HTTP）**：服务 A 调用服务 B 的接口，等待 B 返回结果。简单直观，但 A 和 B 在运行时强耦合——B 挂了 A 也受影响。

**异步通信（消息队列）**：服务 A 发送消息到队列，服务 B 从队列消费。A 和 B 在运行时解耦——B 暂时不可用不影响 A。但引入了消息的可靠投递、顺序、重复等问题。

选择同步还是异步的标准：
- 需要立即拿到结果的用同步：下单时查库存够不够。
- 不需要立即拿到结果的用异步：下单后发送通知。
- 一个操作触发多个下游的用异步：支付成功后通知库存、积分、通知等多个服务。

### 3.3 服务治理

微服务架构需要一套基础设施来管理服务之间的交互：

**服务注册与发现**：服务启动时注册自己的地址，调用方通过注册中心发现目标服务。常用：Nacos、Consul、Eureka。

**配置中心**：集中管理各服务的配置，支持动态修改不需要重启。常用：Nacos、Apollo。

**API 网关**：统一的请求入口，做路由、认证、限流、日志。常用：Spring Cloud Gateway、Kong。

**链路追踪**：跟踪一个请求在多个服务之间的完整调用链路，定位性能瓶颈和错误。常用：Jaeger、Zipkin、SkyWalking。

**集中日志**：把所有服务的日志收集到一起，方便搜索和分析。常用：ELK（Elasticsearch + Logstash + Kibana）。

这些基础设施是微服务架构的"税"。没有它们，微服务就是一盘散沙。

### 3.4 微服务的数据管理

每个服务有自己的数据库。这解决了数据归属的问题，但引入了新的挑战：

**跨服务查询**：订单详情页需要订单信息、商品信息、用户信息。在单体里是一个 join 查询，在微服务里需要分别调三个服务再在内存中组装。

解决方案：
- API 组合：由前端或 BFF（Backend For Frontend）层分别调用各服务并组装。
- 数据冗余：订单服务中冗余存储商品快照和用户地址快照，查询时不需要调其他服务。
- CQRS：读和写分离。写操作走各自的服务，读操作走一个专门的查询服务（从各服务同步数据）。

**跨服务事务**：下单需要创建订单 + 扣库存 + 冻结优惠券。在单体里是一个事务。在微服务里需要 Saga 模式（一系列本地事务 + 补偿操作）或消息驱动的最终一致性。

## 4. 常见误区

### 4.1 "先拆再说"

有些团队在还不确定业务边界的时候就开始拆服务。结果拆完发现边界不对——两个服务之间的调用比服务内部的调用还频繁。

正确的做法是：**先在单体中把模块边界划清楚，确认边界合理后再考虑拆分。** 在代码层面验证过的边界，搬到服务层面才不会出问题。

### 4.2 拆了微服务但共享数据库

把代码拆成了独立服务，但所有服务还是读写同一个数据库。这样拆分的意义大打折扣——服务在数据层面仍然是耦合的。一个服务改了表结构，其他服务可能受影响。

如果业务确实需要多个服务访问同一份数据，应该明确一个服务作为数据的"属主"，其他服务通过 API 获取数据。

### 4.3 "每个功能一个服务"

把服务拆得太细：用户注册一个服务、用户登录一个服务、用户信息管理一个服务。结果是几十个小服务，每个只有几百行代码，大部分逻辑都在服务间的通信和协调上。

拆分的粒度应该是**业务域**级别，不是**功能**级别。用户注册、登录、信息管理都属于"用户域"，应该在一个服务中。

### 4.4 忽视运维能力

微服务架构对运维能力的要求远高于单体。没有自动化部署、没有集中监控、没有链路追踪，微服务就是运维噩梦。

如果团队没有运维自动化的能力（或者没有 DevOps 工程师），不建议上微服务。

### 4.5 团队结构和服务不匹配

康威定律说：系统的架构倾向于反映组织的沟通结构。如果把系统拆成了五个服务，但还是一个大团队负责所有服务，那拆分的收益会大打折扣——因为协作模式没变。

理想状态是：每个服务由一个小团队（3-8 人）负责，团队对自己的服务有完整的开发、测试、部署、运维权限。

## 5. 设计方法

### 5.1 什么时候应该拆微服务

满足以下条件中的多个时，考虑拆分：

1. **团队超过 15 人**，代码合并冲突频繁，发布排队严重。
2. **某个模块需要独立扩容**，但单体无法按模块扩容。
3. **不同模块的发布节奏差异大**，一个模块每天发布，另一个模块每周发布，互相制约。
4. **模块之间的技术栈需求不同**，一个模块适合 Java，另一个适合 Python。
5. **运维团队有能力支撑多服务的自动化部署、监控和故障处理。**

### 5.2 拆分的步骤

**第一步：在单体中划清模块边界。** 确保模块之间通过接口交互，不直接依赖内部实现。

**第二步：选择第一个要拆出去的模块。** 选择变化频率最高、或者需要独立扩容、或者影响面最独立的模块。

**第三步：把模块拆为独立服务。** 独立仓库、独立数据库、独立部署。模块的 Facade 接口变为 RPC/HTTP 接口。

**第四步：搭建服务治理基础设施。** 服务注册、配置中心、链路追踪、集中日志。可以在第一个服务拆出去的同时搭建。

**第五步：逐步拆出更多服务。** 每次拆一个，验证没问题后再拆下一个。不要一次全拆。

### 5.3 服务粒度的判断

问自己三个问题：

1. **这两个功能总是一起修改吗？** 如果是，放在一个服务里。如果各自独立变化，考虑拆分。
2. **这两个功能之间每个请求都需要同步调用吗？** 如果是，放在一个服务里。频繁的同步调用跨网络会带来延迟和可靠性问题。
3. **这个服务是否有独立的团队负责？** 如果一个人同时负责三个服务，拆分的意义不大。

## 6. 业务案例

### 6.1 案例：电商系统从单体到微服务的演进

**第一年**：单体。五个人，一个 Spring Boot 项目。订单、商品、用户、支付全在一起。

**第二年**：模块化单体。团队到了十人。代码按模块重新组织，模块间通过 Facade 接口交互。引入了 Maven 多模块隔离编译依赖。

**第三年**：拆出第一个服务。团队到了十五人。订单查询流量远大于其他功能，把订单模块拆为独立的订单服务。同时搭建了 Nacos + Spring Cloud Gateway + SkyWalking 基础设施。

**第四年**：继续拆分。支付独立为支付服务（对接外部渠道，稳定性要求高）。用户和商品仍然在主单体中——它们的变化频率低，暂时不需要独立部署。

到第四年，系统是"两个微服务 + 一个核心单体"的混合架构。不是所有模块都拆成了微服务——只有确实需要的才拆了。

**关键教训**：不要追求"全面微服务化"。只在有明确收益的地方拆分，其他部分保持在单体中。混合架构是现实中最常见的合理形态。

## 7. 架构判断清单

- 如果团队不到 10 人就上了微服务，大概率是过早拆分，需要评估是否应该合并回单体。
- 如果微服务之间每个请求都需要同步调用三个以上的服务，说明拆分粒度可能太细。
- 如果拆了微服务但没有链路追踪和集中日志，说明排查问题会非常困难。
- 如果一个需求需要同时修改三个以上的服务并协调发布，说明服务拆分的边界可能不对。
- 如果多个微服务共享一个数据库，说明数据层面的拆分没做到位。
- 如果服务数量超过了团队人数的两倍，说明可能维护不过来。

## 8. 本章小结

微服务不是银弹。它解决了大规模团队协作和独立扩容的问题，但引入了分布式系统的复杂性——网络通信、数据一致性、运维成本。这些复杂性不会凭空消失，它从代码层面搬到了基础设施和运维层面。

选择微服务的前提是：团队规模和业务复杂度已经到了单体无法有效管理的阶段，并且团队有能力承担微服务的运维成本。如果这两个条件不满足，模块化单体是更好的选择。

从单体到微服务的演进应该是渐进的：先模块化 → 拆出最需要独立的模块 → 逐步拆出更多模块。不要一步到位，不要追求"全面微服务化"。混合架构——一部分微服务、一部分单体——在大多数公司的大多数阶段都是最务实的选择。
