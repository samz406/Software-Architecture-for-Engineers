# 第 25 章：架构决策记录（ADR）

## 1. 从一个真实问题开始

一个电商系统的订单模块在三年内经历了三个技术负责人。第三任负责人接手后，发现订单状态存储了两份——MySQL 里一份，Redis 里一份。两份数据偶尔会不一致，导致用户看到的订单状态和实际不符。

他想把 Redis 里的订单状态去掉，只用 MySQL 作为唯一数据源。但团队里的老员工说："当年加 Redis 是有原因的，好像是因为查询太慢。"

"好像"——没人确切知道原因。

他翻了代码的 Git 历史，找到了三年前的那次提交。提交信息是"add redis cache for order status"。为什么加的？当时的查询慢到什么程度？试过其他方案吗？有没有考虑过一致性问题？

都不知道。

他不敢随便去掉 Redis 缓存——万一当年加的原因现在仍然存在呢？也不确定应该保留——因为一致性问题确实在造成线上 bug。

最终他花了两周时间做性能测试和风险评估，才确认可以安全地去掉 Redis 缓存（因为三年前缺失的联合索引后来已经加上了，查询性能不再是问题）。

**如果三年前做这个决策的人记录了为什么加 Redis 缓存、当时的性能数据、考虑过的替代方案，这两周的调研完全不需要。**

## 2. 问题为什么会变复杂

### 2.1 决策的上下文会丢失

每个架构决策都是在特定上下文中做出的：当时的业务规模、技术约束、团队能力、时间压力。

但上下文不会自动保存。代码记录了"做了什么"，但不记录"为什么这么做"。Git 提交信息通常只有一行，记录的是改动内容而不是决策理由。架构文档如果有的话，记录的是当时的方案，但随着代码演进文档往往没有同步更新。

三个月后，做决策的人可能自己都忘了当时的考虑。三年后换了人，决策的上下文就完全丢失了。

### 2.2 不知道为什么就不敢改

系统中有很多"看起来不合理但没人敢动"的设计。一个冗余的缓存层、一个奇怪的异步处理流程、一个看不懂为什么存在的中间表。

不知道当初为什么这么做，就无法判断现在是否可以改。改了可能没事，也可能引发一个当年已经踩过的坑。

这导致系统中积累越来越多的"不敢动的代码"，直到不得不做一次大规模重构。

### 2.3 重复踩坑

团队 A 两年前尝试过方案 X，发现有性能问题放弃了。但这个失败的尝试没有记录下来。两年后团队 B 面对类似问题，又选了方案 X，又踩了同样的坑。

被放弃的方案和被选中的方案同样值得记录——知道什么路走不通，和知道什么路能走通一样重要。

## 3. 核心概念解释

### 3.1 什么是 ADR

ADR（Architecture Decision Record）是一种轻量级的文档格式，用于记录架构决策。每个 ADR 记录一个决策：做了什么选择、为什么、考虑过哪些替代方案、有什么后果。

ADR 的核心特点是**轻量**。不是写一份详细的技术方案，而是用一页纸记录一个决策的关键信息。

### 3.2 ADR 的格式

一条 ADR 通常包含以下内容：

```markdown
# ADR-001: 订单状态查询使用 MySQL 联合索引，不使用 Redis 缓存

## 状态
已采纳（2024-01-15）

## 背景
订单列表查询 P95 响应时间 2800ms，原因是 orders 表缺少
(user_id, order_status, created_at) 的联合索引。
此前通过 Redis 缓存订单状态来绕过这个问题，但引入了
MySQL 和 Redis 数据不一致的问题（过去三个月出现了 12 次
用户投诉订单状态不对）。

## 决策
给 orders 表添加联合索引 (user_id, order_status, created_at)，
去掉订单状态的 Redis 缓存。订单列表查询直接走 MySQL。

## 考虑过的替代方案

### 方案 B：保留 Redis 缓存，修复一致性问题
在订单状态变更时同步更新 Redis，加上定时对账任务修复不一致。
放弃原因：增加了系统复杂度（多了同步逻辑和对账任务），
且无法完全消除不一致的时间窗口。

### 方案 C：订单查询走 Elasticsearch
把订单数据同步到 ES，查询走 ES。
放弃原因：当前数据量（8000 万）MySQL 加索引后完全能应对，
引入 ES 增加了运维成本和数据同步复杂度。等数据量超过 3 亿时再考虑。

## 后果
- 订单列表查询 P95 从 2800ms 降到 50ms。
- 消除了 Redis 和 MySQL 的数据不一致问题。
- 去掉了 Redis 缓存相关的代码（约 300 行），降低了维护成本。
- 新增联合索引导致写入性能下降约 5%（可接受）。
- 当数据量超过 3 亿时可能需要重新评估（预计一年半后）。
```

### 3.3 ADR 的关键要素

**状态**：已采纳、已废弃、已替代（被另一条 ADR 替代）。状态让读者知道这个决策是否仍然有效。

**背景**：做这个决策时面对的问题和约束。这是最重要的部分——没有背景，决策就是一个没有根基的结论。

**决策**：做了什么选择。简明扼要地描述。

**考虑过的替代方案**：考虑了哪些其他方案、为什么放弃。这部分的价值往往比"决策"本身更大——它告诉后来者什么路已经走过了、为什么走不通。

**后果**：这个决策带来的结果，包括正面的和负面的。让后来者知道这个决策的实际效果。

### 3.4 ADR 和架构文档的区别

架构文档描述的是一个系统的**整体设计**——模块划分、数据模型、核心流程。它是一个快照，记录某个时间点系统"是什么样的"。

ADR 记录的是一个**具体的决策**——为什么这么选、放弃了什么。它是一条记录，解释系统"为什么是这样的"。

架构文档回答"系统长什么样"。ADR 回答"系统为什么长这样"。两者互补。

## 4. 常见误区

### 4.1 "太小的决策不值得记录"

哪些决策值得记录？一个简单的判断标准：**如果三个月后有人可能问"为什么这么做"，就值得记录。**

"为什么订单表用 user_id 而不是用 phone 做分区键？"——值得记录。
"为什么变量命名用 camelCase？"——不值得记录，这是团队编码规范。

### 4.2 "写 ADR 太费时间"

一条 ADR 写二十分钟到半小时。和做这个决策时的讨论时间相比，这个投入很小。

而且 ADR 不需要精心打磨的文笔。它是给团队内部看的技术笔记，不是给外部展示的报告。写清楚背景、决策、替代方案就够了。

### 4.3 只记录成功的决策

被放弃的方案同样重要。如果只记录"选了方案 A"，后来者可能会想"为什么不用方案 B？"然后花时间去评估方案 B，最后发现当年已经评估过了。

记录被放弃的方案和放弃的原因，能避免团队重复踩坑。

### 4.4 写了但没人看

ADR 写了但存在一个没人知道的角落，等于没写。

ADR 应该放在项目仓库中（比如 `docs/adr/` 目录），随代码一起版本控制。新人 onboarding 时应该阅读关键的 ADR。在讨论相关改动时应该引用已有的 ADR。

### 4.5 过于正式

有些团队把 ADR 搞得很正式——要审批流程、要签字、要归档。这样的重流程会让大家不愿意写。

ADR 应该是低门槛的：工程师做了一个架构决策，花二十分钟写一条 ADR，提交到仓库。不需要审批，但鼓励 Code Review 时一起看。

## 5. 设计方法

### 5.1 什么时候写 ADR

在做出架构决策后尽快写，趁上下文还在记忆中。以下场景建议写 ADR：

- 选择了某个技术方案而放弃了其他方案（缓存策略、消息队列选型、数据模型设计）。
- 做了一个影响多个模块的设计决策（接口设计、数据归属、模块边界）。
- 做了一个有明确取舍的决策（性能 vs 一致性、简单 vs 灵活）。
- 否决了一个看起来合理的方案（别人可能会问为什么不选这个方案）。
- 解决了一个困扰团队的技术难题（后来者可能会遇到同样的问题）。

### 5.2 ADR 的编号和管理

ADR 按时间顺序编号：ADR-001、ADR-002、ADR-003。编号一旦分配不会重用。

目录结构建议：

```
docs/
  adr/
    ADR-001-order-query-index-vs-cache.md
    ADR-002-payment-channel-routing-strategy.md
    ADR-003-member-point-separate-from-growth.md
    README.md  (ADR 索引，列出所有 ADR 和状态)
```

README 作为索引：

```markdown
# 架构决策记录

| 编号 | 标题 | 状态 | 日期 |
|-----|------|------|------|
| 001 | 订单查询用索引不用缓存 | 已采纳 | 2024-01-15 |
| 002 | 支付渠道路由策略 | 已采纳 | 2024-02-01 |
| 003 | 积分和成长值独立核算 | 已采纳 | 2024-03-10 |
```

### 5.3 ADR 的演进

ADR 一旦写好，内容本身不应该修改（它是一个历史记录）。如果后来决策变了，写一条新的 ADR，引用并替代旧的 ADR。

```markdown
# ADR-007: 订单查询迁移到 Elasticsearch

## 状态
已采纳（2025-06-01）。替代 ADR-001。

## 背景
ADR-001 决定订单查询走 MySQL 联合索引。当时数据量 8000 万，
MySQL 能应对。现在订单数据量已达 4 亿，联合索引的查询性能
开始下降，P95 回升到 1.2 秒。

## 决策
订单查询迁移到 Elasticsearch。通过 Canal 同步 MySQL 数据
到 ES，查询走 ES，写入仍走 MySQL。
```

这样就形成了完整的决策链：ADR-001 说"用索引不用缓存"，ADR-007 说"索引不够了，迁移到 ES"。后来者能看到整个演进过程。

## 6. 业务案例

### 6.1 案例：积分系统的 ADR 实践

一个团队在做会员积分系统时，开始实践 ADR。以下是他们在三个月内记录的几条 ADR：

**ADR-003：积分和成长值独立核算**

背景：产品需求中"积分"和"成长值"概念容易混淆。积分可以消耗（兑换商品），成长值只增不减（用于计算等级）。

决策：积分和成长值用独立的账户和流水表。消费产生积分的同时产生成长值，但两者金额可以不同（成长值 = 消费金额，积分 = 消费金额 × 等级系数）。

被放弃的方案：积分等于成长值（用一套数据）。放弃原因：积分消耗后成长值不应该减少，一套数据无法表达这两个不同的生命周期。

**ADR-004：积分余额允许为负数**

背景：用户退款时需要回退积分。但用户可能已经花掉了这些积分。

决策：允许积分余额为负。用户下次获取积分时自动抵扣负数部分。

被放弃的方案一：余额不足时不回退积分。放弃原因：财务要求积分账目准确，不回退会导致对不上。

被放弃的方案二：退款时如果积分已消耗，用现金补偿。放弃原因：增加了退款流程的复杂度，且涉及资金操作需要额外审批。

**ADR-005：等级升级实时触发，降级月度批处理**

背景：用户消费后应该立刻看到等级提升（体验好）。但降级需要看上月整体消费情况（业务规则）。

决策：成长值增加时实时检查是否触发升级。降级由月度定时任务处理——月底检查每个用户上月消费是否达到当前等级的保级门槛。

被放弃的方案：全部实时计算。放弃原因：降级条件是"上月累计消费低于阈值"，实时计算需要在每次成长值变更时都汇总上月数据，查询开销大且逻辑复杂。

**效果**：六个月后团队来了两个新人。他们通过阅读这几条 ADR，半天内就理解了积分系统的核心设计思路和关键取舍。而之前没有 ADR 的模块，新人需要一周左右才能理解。

## 7. 架构判断清单

- 如果团队成员经常说"不知道当初为什么这么设计"，说明缺少决策记录。
- 如果系统中有"没人敢动"的代码，往往是因为不知道当初的设计意图。
- 如果团队在讨论方案时反复评估已经被否决过的方案，说明否决的记录没有保留。
- 如果新人理解系统设计需要超过一周时间，ADR 可以显著加速 onboarding。
- 如果架构决策只存在于某个人的脑子里，那个人离职后知识就丢失了。
- 如果团队已经有超过十条 ADR，但没有索引或分类，说明管理方式需要改进。

## 8. 本章小结

ADR 是一种低成本、高回报的实践。每条 ADR 花二十分钟，但能节省后来者几天甚至几周的调研时间。

ADR 记录的核心信息是：做了什么决策、在什么背景下做的、考虑过什么替代方案、为什么放弃了它们、决策的后果是什么。其中"背景"和"被放弃的方案"是最有价值的部分——它们记录了决策的上下文和边界。

开始写 ADR 不需要特别的工具或流程。在代码仓库里建一个 `docs/adr/` 目录，每次做架构决策时写一个 markdown 文件。从今天开始就能做。
