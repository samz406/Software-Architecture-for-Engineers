# 第 33 章：统一账号与集团平台架构设计

## 1. 从一个真实问题开始

一家互联网公司做了三年，从最初的一个电商 App 扩展到了三个产品线：电商、内容社区、线下门店 SaaS。三个产品线各自有自己的用户系统——用户在电商注册一个账号、在社区注册一个账号、在门店系统又注册一个账号。

问题逐渐暴露：

- 用户在电商 App 上买了东西，想在社区分享购物体验，但两个 App 的账号不通。用户要重新注册社区账号，没有人愿意。
- 运营想做跨产品线的营销——"电商用户在社区发帖可以获取双倍积分"。但两边的用户 ID 不通，没办法判断"这个社区用户是不是电商用户"。
- CTO 想看一个统一的用户画像："某个用户在三个产品线上分别做了什么"。但数据散在三个数据库里，没有关联关系。
- 三套用户系统各自维护注册、登录、找回密码的逻辑。同样的功能写了三遍，bug 也修了三遍。

老板说："统一用户账号，一个账号登录所有产品。" 

听起来简单——把三个用户表合成一个不就行了？实际操作后发现远没有那么简单：三个系统的用户 ID 格式不同（一个用自增 ID、一个用 UUID、一个用手机号）、密码加密方式不同、用户名唯一性规则不同。

**统一账号不是"合并数据库"，而是一个涉及身份认证、权限体系、数据迁移、多产品协同的系统性工程。**

## 2. 问题为什么会变复杂

### 2.1 历史数据的包袱

三个系统各自运行了一两年，加起来有几百万用户数据。这些数据的格式不同、规则不同、质量参差不齐。

比如手机号：电商系统要求必填手机号，内容社区允许只用邮箱注册，门店系统用的是门店老板的手机号。统一后"用户的手机号"到底以哪个为准？

### 2.2 各产品线的自主性和统一性的矛盾

统一账号需要各产品线改造登录流程，把认证逻辑从自己的系统切换到统一的账号中心。但各产品线有自己的排期和优先级——"我们下个月在做大促，没时间改登录"。

而且各产品线对"账号"的需求不完全一样：电商需要收货地址和支付信息、社区需要昵称和头像、门店需要企业信息和门店关联。统一不意味着所有产品线的用户信息模型完全一致。

### 2.3 稳定性要求极高

账号系统是所有产品的基础设施。如果账号中心挂了，三个产品线的用户全部无法登录。这对系统的可用性要求远高于单个产品线的用户系统。

## 3. 核心概念解释

### 3.1 统一账号的核心模型

统一账号系统的核心概念是分层：**账号层**和**产品档案层**。

账号层管理用户的身份认证信息——用户 ID、登录凭证（手机号、邮箱、第三方账号）、密码。所有产品线共享。

产品档案层管理用户在各产品线的业务信息——电商的收货地址、社区的昵称头像、门店的企业信息。各产品线独立管理。

```sql
-- 统一账号（账号层）
CREATE TABLE account (
    account_id BIGINT PRIMARY KEY,           -- 全局唯一用户ID
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

-- 登录凭证（支持多种登录方式）
CREATE TABLE account_credential (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    account_id BIGINT NOT NULL,
    credential_type VARCHAR(20) NOT NULL,    -- PHONE/EMAIL/WECHAT/ALIPAY
    credential_value VARCHAR(200) NOT NULL,  -- 手机号/邮箱/openid
    password_hash VARCHAR(200),              -- 密码登录时使用
    verified BOOLEAN DEFAULT FALSE,
    created_at DATETIME NOT NULL,
    UNIQUE INDEX uk_credential (credential_type, credential_value),
    INDEX idx_account (account_id)
);

-- 产品线关联（记录用户开通了哪些产品线）
CREATE TABLE account_product (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    account_id BIGINT NOT NULL,
    product_code VARCHAR(20) NOT NULL,       -- ECOM/COMMUNITY/STORE
    product_user_id VARCHAR(64),             -- 在该产品线的原始用户ID
    activated_at DATETIME NOT NULL,
    UNIQUE INDEX uk_account_product (account_id, product_code)
);
```

### 3.2 单点登录（SSO）

统一账号的核心能力是单点登录——用户在一个产品线登录后，访问其他产品线不需要再次登录。

常见的 SSO 实现方式：

**基于 Token 的 SSO**：用户在账号中心登录后获取一个全局 Token。访问各产品线时带上这个 Token，产品线向账号中心验证 Token 的有效性。

**基于 OAuth 2.0 的 SSO**：账号中心作为 OAuth 授权服务器，各产品线作为客户端。用户在产品线 A 点击登录，跳转到账号中心授权页面，授权后携带 authorization code 回到产品线 A，产品线 A 用 code 换取 access token。

对于同一家公司的多个产品线，基于 Token 的 SSO 更简单。OAuth 2.0 更适合有外部第三方接入的场景。

### 3.3 权限模型

统一账号通常需要统一的权限管理。常用的权限模型是 RBAC（基于角色的访问控制）：

```
用户 → 角色 → 权限

示例：
用户"张三" 
  → 角色"电商运营" → 权限"查看订单""修改商品价格"
  → 角色"社区管理员" → 权限"删除帖子""封禁用户"
```

权限的层次：

**平台级权限**：适用于所有产品线的权限，比如"用户管理""权限管理"。

**产品级权限**：只在某个产品线内有效的权限，比如"电商-查看订单""社区-删除帖子"。

**数据级权限**：控制用户能看到哪些数据，比如"只能看到自己门店的数据""只能看到自己部门的订单"。

### 3.4 数据迁移策略

三个系统的用户数据合并到统一账号是最困难的一步。常见的策略：

**手机号匹配**：三个系统中使用同一个手机号注册的用户，认为是同一个人，合并为一个统一账号。

**增量迁移**：不一次性迁移所有数据。用户下次登录时触发迁移——在老系统验证身份后，自动在统一账号中创建或关联记录。

**双写过渡期**：迁移期间，登录验证同时查统一账号和老系统。如果统一账号中没有找到，去老系统查。查到后自动同步到统一账号。

## 4. 常见误区

### 4.1 "合并数据库就是统一账号"

把三个用户表 merge 到一张表，不等于建成了统一账号系统。统一账号需要的是统一的认证中心、统一的登录流程、完善的凭证管理。

### 4.2 一步到位

试图在一个大项目中把所有产品线同时切换到统一账号。这样风险太大——任何一个产品线的切换出问题，都可能阻塞整个项目。

应该一个产品线一个产品线地切换。先选用户量最小的产品线做试点，验证没问题后再推广。

### 4.3 忽视老用户的体验

迁移后老用户原来的密码不能用了（因为加密方式换了），要求所有用户重新设置密码。这会导致大量用户流失。

应该兼容老的加密方式。用户用老密码登录时，在后台验证通过后自动用新的加密方式重新加密存储。用户无感知。

### 4.4 账号中心做成了全能系统

把用户的所有信息都放在账号中心——收货地址、购物偏好、社区昵称、门店配置。账号中心变成了一个越来越胖的单点。

账号中心只管身份和认证。业务信息由各产品线自己管理。账号中心提供 account_id，各产品线通过 account_id 关联自己的业务数据。

### 4.5 不考虑账号中心的可用性

账号中心挂了，所有产品线都登不了。但在设计时没有做缓存、没有做降级。

应该在各产品线本地缓存 Token 验证的公钥或 Token 信息。账号中心短暂不可用时，本地缓存仍能验证已有 Token，已登录的用户不受影响。只有新登录会失败。

## 5. 设计方法

### 5.1 统一账号的建设步骤

**第一步：建设账号中心**。独立的服务，提供注册、登录、Token 验证、密码管理接口。先不接入任何产品线，只是搭建好基础设施。

**第二步：新产品线直接接入**。如果公司有新的产品线在开发中，直接接入统一账号，不再自建用户系统。这是最简单的接入方式。

**第三步：选一个老产品线试点迁移**。选用户量最小的产品线，做增量迁移。用户登录时自动迁移到统一账号。设置一个过渡期（比如三个月），过渡期内两套系统并行。

**第四步：推广到其他产品线**。试点成功后，按计划迁移其他产品线。

**第五步：下线老系统**。所有产品线迁移完成且过渡期结束后，下线老的用户系统。

### 5.2 账号中心的稳定性设计

| 措施 | 说明 |
|-----|------|
| 多机部署 | 至少三个实例，负载均衡 |
| Token 缓存 | 各产品线本地缓存 Token 验证信息，减少对账号中心的调用 |
| 降级方案 | 账号中心不可用时，已登录用户可通过本地缓存继续使用 |
| 限流保护 | 登录接口限流，防止被暴力破解 |
| 数据备份 | 账号数据是核心资产，主从备份 + 异地备份 |

## 6. 业务案例

### 6.1 案例：三个产品线的统一账号建设

**背景**：电商 App 300 万用户、内容社区 50 万用户、门店 SaaS 2 万用户。

**第一阶段（2个月）：建设账号中心**

搭建独立的账号服务，实现注册、登录、Token 管理。技术栈和现有系统一致（Spring Boot + MySQL）。Token 使用 JWT，各产品线可以自己验证 Token 不需要每次调账号中心。

**第二阶段（1个月）：门店 SaaS 试点**

门店 SaaS 用户量最小（2 万），风险最低。采用增量迁移方式——用户登录时如果在统一账号中没有记录，则自动用门店系统的手机号和密码创建统一账号，并关联产品线。

过渡期内门店系统同时支持老登录方式和新登录方式。

试点一个月后，98% 的活跃用户已经迁移到统一账号。

**第三阶段（2个月）：内容社区迁移**

社区有一个特殊问题：很多用户只用微信登录，没有绑定手机号。统一账号要求至少有一个手机号或邮箱作为主凭证。

方案：社区用户用微信登录时，如果统一账号中没有该微信 openid 对应的记录，自动创建一个"仅有微信凭证"的账号。后续引导用户补绑手机号（但不强制）。

**第四阶段（3个月）：电商迁移**

电商是最大的产品线，迁移的挑战在于：电商的很多内部系统（客服系统、风控系统、数据平台）都直接查 user 表。迁移后这些系统也需要改成调账号中心的接口。

方案：先迁移登录流程（用户感知的部分），内部系统暂时通过一个同步机制——账号中心的数据变更实时同步到电商的老 user 表。等内部系统逐步改造完成后，再下线同步机制。

**结果**：八个月完成全部迁移。用户可以一个账号登录三个产品线。跨产品线的营销活动（电商用户在社区发帖得积分）顺利上线。用户注册流程从"三次注册"变成"一次注册、三处可用"。

## 7. 架构判断清单

- 如果公司有两个以上的产品线且用户有交叉使用的需求，应该开始规划统一账号。
- 如果统一账号的迁移方案要求所有用户同一时间切换，风险太大——应该用增量迁移。
- 如果账号中心挂了所有产品线都无法使用，说明缺少缓存和降级机制。
- 如果各产品线的业务数据（收货地址、社区昵称）都存在账号中心，说明账号中心承担了不该承担的职责。
- 如果迁移后老用户必须重新设置密码才能登录，说明密码兼容方案没做好。
- 如果统一账号建设预计超过一年，考虑分阶段——先做最核心的 SSO，其他功能后续迭代。

## 8. 本章小结

统一账号是多产品线公司的基础设施。它解决的核心问题是：用户只需要一个身份，就能使用公司的所有产品。

建设统一账号的关键是分层（账号层 vs 产品档案层）、渐进（一个产品线一个产品线迁移）、兼容（老用户无感知迁移）。

统一账号不是一个纯技术项目。它涉及多个产品线的协调、历史数据的治理、用户体验的保障。技术上不难，难的是在不影响现有业务正常运行的前提下完成迁移——这考验的是工程能力和项目管理能力。
