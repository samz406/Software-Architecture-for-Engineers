# 第 11 章：数据架构：系统的骨架

## 1. 从一个真实问题开始

一个运行了两年的电商系统，产品提了一个需求：在订单详情页展示"本次订单使用的优惠信息"。需求不复杂，展示三行字：用了什么优惠券、优惠了多少钱、优惠券的有效期。

工程师评估后说需要两周。产品很意外："就展示三个字段的事，为什么要两周？"

工程师解释了原因：

- 优惠券信息在 coupon 表里，但 coupon 表和 order 表之间没有直接关联。当初下单时把 coupon_id 存在了 order 的一个 JSON 扩展字段里（和其他十几个杂七杂八的信息混在一起）。要解析这个 JSON 才能拿到 coupon_id。
- 拿到 coupon_id 后去 coupon 表查，发现优惠券核销后状态变成了"已使用"，但优惠金额没有单独存储——需要从 coupon_template 表拿优惠规则，再根据订单金额重新计算。
- coupon_template 表存了六种优惠类型（满减、折扣、固定金额……），每种类型的计算逻辑不同。但这个计算逻辑在代码里，不在数据库里。展示页面如果要算优惠金额，需要在查询时重新调用优惠计算逻辑。
- 问题是：用户下单时的商品价格和现在的商品价格可能不一样（商品后来涨价了）。用现在的价格重新计算优惠，结果会和实际不符。

根本原因是：**下单时没有把优惠信息作为订单快照保存下来**。数据模型设计时只关注了"当前操作需要什么数据"，没有考虑"将来查询和追溯需要什么数据"。

## 2. 问题为什么会变复杂

### 2.1 数据模型决定了系统的天花板

代码可以重构——函数改名、类拆分、逻辑重写。但数据模型的重构远比代码重构困难。因为数据是有积累的：两亿条订单记录、五千万条用户数据、三年的交易流水。改数据模型意味着迁移这些数据，而数据迁移是技术项目中风险最高的操作之一。

一个好的数据模型能支撑业务长期演进。一个有缺陷的数据模型会在每次新需求中持续产生额外成本——要么用 hack 绕过去，要么花大代价做数据迁移。

### 2.2 数据模型的常见问题

**大宽表**：一张表几十个字段，什么信息都往里塞。订单表里既有订单信息、又有用户信息、又有商品信息、又有优惠信息。改任何一块信息都要动这张表。

**万能 JSON 字段**：为了避免频繁加字段，搞一个 ext_info 的 JSON 字段，什么扩展信息都往里塞。结果是：这个 JSON 里有什么、各个字段什么含义，只有写入方知道。查询时没法建索引、没法做 SQL 过滤。

**缺少快照**：关联数据只存了 ID（coupon_id、product_id），没有存快照。将来查询时要实时计算或实时查关联表，而关联数据可能已经变了。

**数据归属不清**：多个模块共享一张表，表的字段既有 A 模块写的、又有 B 模块写的。改表结构时不知道会影响谁。

### 2.3 数据模型和业务模型的错位

最常见的数据设计方式是"需求来了就加字段"。来一个需求加一个字段，来一个需求加一张表。表结构完全由需求驱动，没有经过业务建模。

结果是：表结构反映的是需求的堆叠顺序，而不是业务的内在结构。一张 order 表有六十个字段，但工程师说不清楚哪些字段属于"订单核心信息"、哪些属于"快照信息"、哪些属于"扩展信息"。

## 3. 核心概念解释

### 3.1 数据模型设计的基本原则

**原则一：核心实体独立建表。** 订单、支付单、退款单、优惠券——每个核心业务对象有自己的表。不要把多个业务对象的信息混在一张宽表里。

**原则二：关联关系用 ID，核心信息存快照。** 订单和商品的关联用 product_id，但商品名称、价格、规格在下单时应该作为快照存储在订单项表中。因为商品信息会变（改名、调价），但订单记录的是下单那一刻的交易信息。

**原则三：表的字段归属要清晰。** 每个字段应该有明确的"属主模块"。如果一个字段在两个模块中都被写入，说明数据归属不清。

**原则四：为查询场景预留设计。** 数据不只是存进去，还要查出来。设计表结构时要考虑主要的查询场景，确保有合适的索引支持。

### 3.2 宽表 vs 范式化

**范式化**设计是把数据拆到最小粒度，消除冗余。一个纯范式化的设计中，商品名称只存在 product 表中，订单项表只存 product_id，查订单时要 join product 表获取商品名称。

好处是数据不冗余，修改一处即可。坏处是查询时需要 join 多张表，在数据量大时性能差。

**宽表**设计是在一张表中存储尽量多的信息，减少 join。订单项表里直接存了商品名称、价格、图片 URL。

好处是查询快，不需要 join。坏处是数据冗余，商品改名后历史订单里的名称不会自动更新。

实际项目中通常是混合使用：

- 核心业务表按范式设计，保证数据一致性。
- 查询密集的场景用宽表或物化视图，用空间换时间。
- 需要追溯历史的数据存快照（如订单中的商品快照），这种冗余是有意为之的。

### 3.3 索引设计

索引设计不是"哪个字段查得多就加个索引"这么简单。

**联合索引的顺序很重要。** `(user_id, created_at)` 的联合索引可以支持"查某用户的订单按时间排序"，但不能支持"查某时间段的所有订单"。要根据实际查询场景来设计索引顺序。

**索引不是越多越好。** 每个索引都要在写入时维护。一张表有十个索引，每次 insert 就要更新十个索引，写入性能会明显下降。

**覆盖索引能显著减少 IO。** 如果查询只需要索引中的字段，不需要回表，查询性能会好很多。比如"查某用户的订单数量"，如果有 user_id 的索引，只需要扫描索引就能 count。

### 3.4 数据的生命周期

不同的数据有不同的生命周期和访问模式：

**热数据**：最近三个月的订单、当前有效的优惠券。查询频率高，需要快速访问。存在主库，可能还要缓存。

**温数据**：三个月到一年前的订单。偶尔查询，可以接受稍慢的响应。可以归档到单独的历史表或从库。

**冷数据**：一年以上的订单。极少查询，但不能丢（合规或审计需要）。可以归档到对象存储或数据仓库。

数据归档策略需要和业务确认。哪些数据需要在线查询？查询的延迟要求是多少？保留多久？这些问题影响存储方案的选择。

### 3.5 分库分表

当单表数据量超过一定规模（通常是千万到亿级），性能会显著下降。分库分表是应对数据量增长的手段。

**分表**：一张大表按某个维度拆成多张小表。比如订单表按用户 ID 取模分成 16 张表。

**分库**：在分表的基础上，把不同的表放到不同的数据库实例上。分散数据库的 IO 和 CPU 压力。

分库分表的关键决策是**选择分片键**。常见的选择：

- 按用户 ID 分：适合用户维度的查询（"查某用户的订单"）。但按时间维度的查询（"查今天的所有订单"）需要查所有分片。
- 按时间分：适合时间维度的查询（"查某天的订单"）。但按用户维度的查询需要查所有分片。
- 按订单 ID 分：查询单个订单很快，但按用户或时间维度查都不方便。

没有完美的分片键。选择分片键时要明确：主要的查询场景是什么？可以接受哪些场景的查询变慢？

## 4. 常见误区

### 4.1 "先跑起来再说"

项目初期表结构设计草率——字段命名不规范、缺少注释、关联关系不清晰、没有考虑查询场景。理由是"先跑起来再优化"。

但数据模型不像代码那样容易重构。表里积累了数据之后，改字段名要做数据迁移，改表结构要考虑在线 DDL 的风险，拆表要考虑数据一致性。

数据模型的设计应该在项目初期就认真对待。不需要一步到位，但核心表的结构要合理。

### 4.2 "加个字段就行"

每次新需求都往现有表里加字段，直到一张表有五六十个字段。

问题是：字段越多，表的行宽越大，单页能存储的行数越少，扫描效率越低。而且很多字段可能是某个特定场景才用的（比如退款相关的字段只在退款场景有值），对于不涉及退款的订单，这些字段全是 null，浪费存储空间。

更好的做法是：核心字段放在主表，扩展字段按场景拆到独立表。订单核心信息一张表，订单扩展信息一张表，退款信息一张表。

### 4.3 "用 JSON 字段解决一切扩展性问题"

JSON 字段灵活，不需要改表结构就能存储各种数据。但 JSON 字段的问题是：

- 不能建普通索引（MySQL 5.7+ 支持 JSON 索引但有限制）。
- 查询条件复杂（JSON_EXTRACT 的性能不如普通字段查询）。
- 字段含义不明确（没有 schema 约束，什么数据都能往里塞）。
- 数据一致性无法在数据库层面保证。

JSON 字段适合存储真正的"扩展信息"——不需要查询过滤、不需要建索引、结构灵活多变的数据。核心业务字段不应该放在 JSON 里。

### 4.4 过早分库分表

数据量还不大（几百万级别），就开始分库分表。结果是：

- 跨分片的查询和聚合变得复杂。
- 分布式事务难以保证。
- 运维复杂度大幅增加。

分库分表是解决数据量问题的最后手段，不是第一选择。在分库分表之前，先尝试：索引优化、查询优化、读写分离、热数据缓存、历史数据归档。很多时候这些手段就够用了。

### 4.5 忽视数据一致性

两张表之间有关联关系，但没有用事务保证一致性。比如创建订单时先插入 order 记录，再插入 order_item 记录。如果插入 order_item 时失败了，order 记录已经存在但没有 item——数据不一致。

在单库中，用数据库事务保证一致性。跨库或跨服务时，用消息队列 + 补偿机制保证最终一致性。但无论用哪种方式，一致性方案在设计时就要考虑，不能事后补。

## 5. 设计方法

### 5.1 数据模型设计步骤

**第一步：识别核心业务对象。** 从业务流程中提取核心实体：订单、订单项、支付单、退款单、优惠券等。

**第二步：确定对象之间的关系。** 一对一、一对多、多对多。订单和订单项是一对多。订单和优惠券是多对多（一个订单可以用多张券，一张券可以关联多个订单——但通常一张券只能用一次）。

**第三步：确定每个对象的核心字段。** 区分：核心业务字段（必须有的）、快照字段（需要留痕的）、扩展字段（可选的）。

**第四步：为主要查询场景设计索引。** 列出核心查询场景（按用户查订单、按时间查订单、按状态查订单），设计对应的索引。

**第五步：考虑数据的生命周期。** 数据量预估、归档策略、是否需要分库分表。

### 5.2 订单数据模型示例

```sql
-- 订单主表（核心信息）
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(32) NOT NULL UNIQUE,
    user_id BIGINT NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    discount_amount DECIMAL(10,2) DEFAULT 0,
    payable_amount DECIMAL(10,2) NOT NULL,
    order_status VARCHAR(20) NOT NULL,  -- CREATED/CONFIRMED/CANCELLED
    pay_status VARCHAR(20) NOT NULL,    -- UNPAID/PAYING/PAID/REFUNDED
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    INDEX idx_user_created (user_id, created_at),
    INDEX idx_status_created (order_status, created_at)
);

-- 订单项表（包含商品快照）
CREATE TABLE order_items (
    id BIGINT PRIMARY KEY,
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    sku_id BIGINT NOT NULL,
    -- 以下是快照字段，记录下单时的商品信息
    product_name VARCHAR(200) NOT NULL,
    sku_name VARCHAR(200) NOT NULL,
    unit_price DECIMAL(10,2) NOT NULL,
    quantity INT NOT NULL,
    subtotal DECIMAL(10,2) NOT NULL,
    INDEX idx_order_id (order_id)
);

-- 订单优惠信息表（快照）
CREATE TABLE order_promotions (
    id BIGINT PRIMARY KEY,
    order_id BIGINT NOT NULL,
    promotion_type VARCHAR(20) NOT NULL,  -- COUPON/FULL_REDUCE/MEMBER_DISCOUNT
    promotion_id BIGINT,
    promotion_name VARCHAR(200),
    discount_amount DECIMAL(10,2) NOT NULL,
    INDEX idx_order_id (order_id)
);
```

关键设计决策：
- 商品信息在 order_items 中做了快照，不依赖 product 表的当前数据。
- 优惠信息独立为 order_promotions 表，不塞在订单主表的 JSON 字段里。
- 索引按主要查询场景设计：按用户查、按状态查。

## 6. 业务案例

### 6.1 案例：订单 + 支付的数据结构演进

**第一阶段**：支付信息存在订单表里。order 表有 pay_channel、pay_transaction_id、pay_time 等字段。

问题：一个订单可能多次尝试支付（第一次微信支付失败，换支付宝支付成功）。但 order 表只能存一条支付信息。

**第二阶段**：支付信息独立为 payment 表。一个订单可以关联多条支付记录。

```sql
CREATE TABLE payments (
    id BIGINT PRIMARY KEY,
    payment_no VARCHAR(32) NOT NULL UNIQUE,
    order_id BIGINT NOT NULL,
    amount DECIMAL(10,2) NOT NULL,
    channel VARCHAR(20) NOT NULL,
    status VARCHAR(20) NOT NULL,
    channel_transaction_id VARCHAR(64),
    paid_at DATETIME,
    created_at DATETIME NOT NULL,
    INDEX idx_order_id (order_id),
    INDEX idx_channel_transaction (channel_transaction_id)
);
```

**第三阶段**：退款也独立为 refund 表，和 payment 关联。一笔支付可以多次部分退款。

这个演进过程说明：数据模型不是一步到位的，但每一步都应该清晰地表达业务关系。支付从 order 表的几个字段，演进为独立的 payment 表和 refund 表，反映了支付作为独立业务概念的实际地位。

### 6.2 案例：从单表到读写分离

订单表数据量到了三千万，高峰期数据库 CPU 经常 80% 以上。分析后发现：70% 的查询是读操作（用户查订单列表、客服查订单详情、运营看报表），30% 是写操作（创建订单、更新状态）。

方案一（读写分离）：增加 MySQL 从库，读操作走从库，写操作走主库。主从延迟通常在 1 秒以内，对于订单列表查询是可接受的。对于需要强一致的场景（支付成功后立即查询），仍然走主库。

方案二（缓存 + 归档）：热数据（最近三个月的订单）放 Redis 缓存，冷数据（三个月以上）归档到历史表。归档后主表数据量从三千万降到五百万，性能压力大幅降低。

团队选择了两个方案都做。读写分离解决日常的读压力，缓存 + 归档解决数据量增长的问题。分库分表留到下一个阶段——数据量过亿的时候再考虑。

## 7. 架构判断清单

- 如果一张表超过 40 个字段，说明可能需要拆分为核心表和扩展表。
- 如果关联查询需要 join 超过三张表，说明数据模型可能需要适度冗余或做物化视图。
- 如果新增一个展示字段需要实时计算而不是直接查询，说明缺少必要的数据快照。
- 如果一张表的数据量超过五千万且查询性能下降明显，需要考虑读写分离、缓存或数据归档。
- 如果 JSON 扩展字段的内容需要作为查询条件，说明这些字段应该提升为独立字段。
- 如果不同模块直接读写同一张表，说明数据归属不清晰，需要明确属主模块。
- 如果做数据迁移时不知道某个字段的含义或者哪些代码在使用它，说明数据字典缺失。

## 8. 本章小结

数据架构是系统的骨架。代码可以随时重写，但数据模型一旦积累了大量数据，修改的成本就很高。在项目初期花时间设计合理的数据模型，比后期做数据迁移要划算得多。

数据模型设计的核心是：让表结构反映业务结构。核心业务对象独立建表，关联关系用 ID 加快照，按查询场景设计索引，按数据生命周期规划存储策略。

不需要在项目初期就做分库分表这样的重型方案。先把数据模型设计好、索引建好、查询优化做好。等数据量真正到了瓶颈时再考虑分库分表。正确的顺序是：先优化查询 → 读写分离 → 缓存 → 归档 → 分库分表。每一步都在前一步不够用的时候才走。
