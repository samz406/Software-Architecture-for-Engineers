# 第 22 章：如何做一次完整的架构设计

## 1. 从一个真实问题开始

产品经理提了一个需求："我们要做一个会员体系，支持等级、积分、权益、成长值。"

技术负责人把需求拆给了三个工程师。第一个工程师从数据库表开始设计——会员表、积分流水表、等级配置表。第二个工程师从接口开始设计——查积分、加积分、查等级。第三个工程师从页面开始设计——会员中心页的各个模块。

两周后联调，发现三个人的设计对不上：

- 积分流水表记录了每次积分变更，但没有"冻结积分"的概念。而权益兑换需要先冻结积分，确认后再扣减。
- 等级配置表只支持"按累计积分升级"，但产品说"消费金额也算成长值"。成长值和积分是两套独立的数值体系，但第一个工程师以为成长值就是积分。
- 接口设计中，"查积分"返回的是总积分，但前端需要的是可用积分、冻结积分、即将过期积分三个数字。

结果又花了两周返工。

问题不在于三个工程师的能力不行，而在于**没有做一次完整的架构设计就开始干活了**。每个人基于自己对需求的理解各干各的，缺少一个全局的、统一的设计来对齐大家的认知。

## 2. 问题为什么会变复杂

### 2.1 需求不等于设计

产品给的需求是"做什么"——做会员体系。但架构设计要回答的是"怎么做"以及"为什么这么做"。

从需求到设计之间有一大段空白地带需要填：
- 积分和成长值是同一个概念还是两个？（业务建模问题）
- 积分有没有过期？过期的粒度是什么？（状态和生命周期问题）
- 等级升降是实时计算还是定时批量跑？（性能和一致性取舍）
- 积分操作要不要支持撤销？退款时积分怎么回退？（异常流程问题）

这些问题如果不在设计阶段回答清楚，就会变成开发阶段的"惊喜"。

### 2.2 缺少全局视角

三个工程师各自在自己的局部范围内做了合理的设计，但没有人从全局视角看整个系统。局部最优的组合不等于全局最优。

数据模型、接口设计、业务流程、异常处理——这些必须作为一个整体来设计，不能分开设计再拼凑。

### 2.3 设计的时机被忽视

有些团队把"架构设计"等同于"写架构文档"，觉得是一个纯文档工作，能省就省。实际上架构设计的核心产出不是文档，而是**一组经过推演和验证的关键决策**。文档只是把这些决策记录下来。

跳过设计直接写代码，表面上节省了几天时间，但返工的成本往往是设计时间的三到五倍。

## 3. 核心概念解释

### 3.1 架构设计的完整流程

一次完整的架构设计通常包含以下步骤：

**第一步：理解业务目标和背景**

不是理解"做什么功能"，而是理解"为什么要做这个"。会员体系是为了提升用户留存还是为了增加客单价？这个业务目标会影响架构决策。

如果是提升留存，积分和等级的体验要做得很实时，用户做了一个操作立刻看到积分变化。如果主要是为了增加客单价，重点在权益和优惠的灵活配置，积分延迟几分钟到账不影响体验。

**第二步：识别核心场景**

不是把所有功能列一遍，而是找出最关键的三到五个场景，把它们的流程画清楚。

会员体系的核心场景：
1. 用户消费后获取积分和成长值
2. 积分兑换权益（优惠券、实物奖品）
3. 等级升降和权益变更
4. 退款导致的积分回退

每个核心场景都要画出完整的流程，包括正常路径和异常路径。

**第三步：分析质量属性**

功能需求决定"做什么"，质量属性决定"做到什么程度"。

- 性能：积分查询的QPS会有多大？大促期间积分发放的峰值？
- 一致性：积分余额能不能出现负数？能容忍多久的延迟？
- 可用性：积分服务挂了，影不影响下单？
- 可扩展性：将来要对接外部积分商城，现在的设计能扩展吗？

**第四步：明确约束和限制**

约束是不能违反的边界条件：
- 技术约束：团队用 Java + MySQL，不引入新的技术栈。
- 资源约束：两个人，一个月上线。
- 业务约束：积分操作必须有完整的流水记录，财务要审计。

约束会排除一些"理论上更好"的方案。两个人一个月，就不可能做独立的积分微服务加完整的事件溯源。

**第五步：设计系统结构**

基于前面的分析，设计模块划分、数据模型、核心流程和接口。这是架构设计的主体部分，下一节详细展开。

**第六步：识别风险和制定应对**

找出设计中的高风险点，提前想好应对方案：
- 积分并发操作会不会超发？→ 用乐观锁 + 余额校验。
- 积分过期批量处理会不会太慢？→ 按用户分片处理，控制每批数量。
- 将来积分规则变复杂怎么办？→ 积分计算逻辑抽象为规则引擎接口，当前用简单实现。

**第七步：规划演进路径**

第一版做什么，后续版本怎么迭代。不要把所有功能塞到第一版，但要确保第一版的架构能承载后续扩展。

### 3.2 系统结构设计的要点

系统结构设计是架构设计的核心。它需要回答三个问题：

**模块怎么分？** 会员体系可以分为积分模块、等级模块、权益模块。模块之间通过接口交互。积分模块不关心等级怎么算，等级模块不关心积分怎么发。

**数据怎么存？** 积分要不要独立的数据库？积分流水表预计多大？用什么分页方式？成长值快照存在哪？

**流程怎么走？** 用户下单支付后，积分怎么发放？是同步发还是异步发？发放失败怎么处理？退款时怎么回退？

### 3.3 推演和验证

设计完成后不是直接开始写代码。需要用核心场景推演一遍：

拿"用户支付后获取积分"这个场景走一遍：
1. 支付成功事件到达积分模块。
2. 积分模块根据订单金额和积分规则计算积分。
3. 增加用户积分余额，写入积分流水。
4. 增加成长值。
5. 检查成长值是否触发升级。

走到第 5 步发现问题：升级逻辑在等级模块，但成长值在积分模块更新。等级模块怎么知道成长值变了？是积分模块通知等级模块，还是等级模块定时检查？

这种推演能发现设计中的漏洞——在写代码之前发现。

## 4. 常见误区

### 4.1 "设计就是画架构图"

很多工程师理解的架构设计就是画几个方框、连几条线。画完图觉得设计就完成了。

架构图只是设计的可视化表达，不是设计本身。一张架构图回答不了："积分余额用什么锁？""积分流水表按什么字段分表？""退款回退积分如果原订单的积分已经被用了怎么办？"

这些具体的决策才是架构设计的核心内容。

### 4.2 "先做再说，边做边调"

小功能可以边做边调。但涉及数据模型和模块边界的设计，一旦代码写下去了，改的成本很高。

积分流水表设计错了，上线后已经有了一百万条数据，要改表结构就要做数据迁移。模块边界划错了，两个模块的代码互相依赖了，要拆开就是一次重构。

### 4.3 "我自己想清楚了就行"

架构设计不是一个人的事。即使主设计者想得很清楚，如果没有和团队其他成员对齐，每个人写出的代码可能基于不同的假设。

上面会员体系的例子就是典型：三个人各自理解不同，因为缺少一次对齐的环节。

### 4.4 设计太理想化

有些设计者喜欢设计"完美"的方案——高度抽象、充分灵活、面面俱到。但忽略了团队只有两个人、工期只有一个月的现实。

好的架构设计是在约束下找到最优解，不是忽略约束设计理想方案。

### 4.5 只设计正常流程

设计时只考虑了"用户支付成功 → 发积分 → 升等级"的正常路径。没有考虑：
- 发积分时积分服务挂了怎么办？
- 用户在积分到账前退款了怎么办？
- 同一个订单的支付回调重复到达怎么办？

异常流程的设计往往比正常流程复杂，但也更容易被忽视。

## 5. 设计方法

### 5.1 设计的产出物

一次完整的架构设计，最终应该产出以下内容（不一定都写成正式文档，但团队内部要对齐）：

**1. 业务背景和目标**：为什么要做、做到什么程度。

**2. 核心场景的流程图**：三到五个关键场景的完整流程，包含正常和异常路径。

**3. 模块划分和依赖关系**：哪些模块、模块之间怎么交互、数据归属。

**4. 数据模型**：核心表结构、关键字段、索引设计。

**5. 接口定义**：模块间的核心接口，入参出参、错误码。

**6. 关键决策和取舍**：选了方案 A 而不是方案 B，原因是什么。

**7. 风险和应对**：已知的风险点和应对方案。

**8. 分期计划**：第一期做什么、第二期做什么。

### 5.2 设计会议的组织

架构设计不应该是一个人闭关三天写文档，而是一个逐步深化的过程：

**第一轮（1-2小时）**：主设计者介绍业务背景、核心场景、初步方案。团队提问、挑战、补充。

**中间（1-3天）**：主设计者根据讨论结果深化设计，完善数据模型和接口定义，推演核心场景。

**第二轮（1-2小时）**：走读详细设计，重点讨论数据模型、异常处理、风险点。对齐实现方案。

两轮讨论足以覆盖大多数中等规模的设计。复杂系统可能需要更多轮，但也不要变成无限讨论。

### 5.3 设计的深度

不是每个需求都需要完整的架构设计。判断标准：

**需要完整设计的**：
- 新系统或新模块的设计。
- 涉及数据模型变更的功能。
- 跨团队或跨服务协作的功能。
- 预计开发超过两周的功能。

**简单设计即可的**：
- 在已有模块上增加字段或接口。
- 不涉及数据模型变更的功能调整。
- 单人一两天能完成的小功能。

## 6. 业务案例

### 6.1 案例：会员积分体系的架构设计过程

**第一步：理解业务背景**

产品目标是提升用户复购率。通过积分和等级体系激励用户持续消费。第一期上线积分获取和查询、等级计算、基础权益配置。积分商城和外部积分兑换放到第二期。

约束：两个后端工程师，一个月工期。技术栈是现有的 Spring Boot + MySQL 单体应用。

**第二步：识别核心场景并推演**

场景一：消费获取积分

```
支付成功事件 → 积分模块接收 → 计算积分（订单金额 × 积分比例）
→ 增加积分余额 → 写入积分流水 → 增加成长值 → 触发等级检查
```

推演中发现的问题：
- 积分比例可能因用户等级不同而不同（高等级用户获取更多积分）。这意味着积分模块依赖等级模块。设计决策：积分模块调用等级模块查询当前等级，获取积分倍率。
- 同一个支付事件可能重复到达。设计决策：积分流水表加唯一索引 `(order_id, type)`，防重复。

场景二：退款回退积分

```
退款成功事件 → 积分模块接收 → 查询该订单的积分发放记录
→ 扣减积分余额 → 写入负数积分流水 → 扣减成长值 → 触发等级检查
```

推演中发现的问题：
- 如果用户积分已经花掉了，余额不够扣怎么办？设计决策：允许积分余额为负数。在用户下次获取积分时自动抵扣。另一种方案是"不扣了就算了"——和产品讨论后选择方案一，因为财务需要积分账目准确。

场景三：等级升降

```
成长值变更 → 查询等级阈值配置 → 判断是否触发升降级
→ 更新用户等级 → 记录等级变更日志 → 触发权益变更
```

推演中发现的问题：
- 降级是实时的还是周期性的？产品说降级按自然月结算——月底检查上个月的消费是否达标。设计决策：升级实时触发，降级用月度定时任务处理。

**第三步：模块划分**

```
会员模块（member）
├── 积分子模块：积分发放、扣减、查询、过期处理
├── 等级子模块：等级计算、升降级、等级配置管理
└── 权益子模块：权益配置、权益发放、权益查询
```

模块在代码层面用包隔离，通过 Facade 接口交互。不做微服务拆分——两个人一个月的工期不支持。

**第四步：数据模型设计**

```sql
-- 用户积分账户
CREATE TABLE member_point_account (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL UNIQUE,
    available_points INT NOT NULL DEFAULT 0,   -- 可用积分
    frozen_points INT NOT NULL DEFAULT 0,      -- 冻结积分
    total_earned BIGINT NOT NULL DEFAULT 0,    -- 累计获取
    total_consumed BIGINT NOT NULL DEFAULT 0,  -- 累计消耗
    version INT NOT NULL DEFAULT 0,            -- 乐观锁
    updated_at DATETIME NOT NULL
);

-- 积分流水
CREATE TABLE member_point_log (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    points INT NOT NULL,               -- 正数=获取，负数=扣减
    type VARCHAR(32) NOT NULL,         -- EARN/CONSUME/EXPIRE/REFUND
    biz_type VARCHAR(32) NOT NULL,     -- ORDER/EXCHANGE/MANUAL
    biz_id VARCHAR(64) NOT NULL,       -- 关联的业务ID（如订单号）
    expire_at DATETIME,                -- 过期时间
    created_at DATETIME NOT NULL,
    UNIQUE KEY uk_biz (biz_id, type, biz_type)
);

-- 用户等级
CREATE TABLE member_level (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL UNIQUE,
    current_level INT NOT NULL DEFAULT 0,
    growth_value BIGINT NOT NULL DEFAULT 0,
    level_updated_at DATETIME,
    version INT NOT NULL DEFAULT 0
);
```

**第五步：关键设计决策记录**

| 决策 | 选择 | 放弃的方案 | 原因 |
|-----|------|-----------|------|
| 积分和成长值分离 | 两套独立数值 | 积分=成长值 | 积分可消耗，成长值只增不减（或按月衰减） |
| 积分余额可为负 | 允许负数 | 余额不足时不回退 | 财务要求积分账目准确 |
| 升级实时/降级月结 | 混合模式 | 全部实时 | 降级逻辑复杂（看上月消费），实时计算成本高 |
| 模块化单体 | 包级别隔离 | 独立微服务 | 两人一月工期不支持微服务基础设施 |

**效果**：设计评审花了两个半天（约8小时），后续开发中没有出现需要返工的结构性问题。一个月内完成了第一期上线。

## 7. 架构判断清单

- 如果开发进行到一半发现数据模型要大改，说明设计阶段对核心场景的推演不够。
- 如果团队成员对同一个概念（如"积分"和"成长值"）理解不一致，说明设计阶段缺少概念对齐。
- 如果只画了架构图没有推演过核心场景的完整流程，设计大概率有漏洞。
- 如果设计中没有提到任何异常场景的处理方案，说明只考虑了正常路径。
- 如果设计方案只有一个、没有备选方案和取舍说明，说明可能没有充分思考过替代方案。
- 如果设计的第一版方案包含了所有功能，说明没有考虑分期和优先级。
- 如果设计文档超过了二十页但团队没有当面讨论过，说明对齐方式有问题——设计需要讨论，不只是阅读。

## 8. 本章小结

架构设计不是画图，也不是写文档。它是在动手写代码之前，通过分析业务、推演场景、设计结构、识别风险，形成一组关键决策的过程。

完整的架构设计流程是：理解业务目标 → 识别核心场景 → 分析质量属性 → 明确约束 → 设计系统结构 → 推演验证 → 识别风险 → 规划演进路径。

不是每个需求都需要走完整流程，但涉及新系统设计、数据模型变更、跨团队协作的需求，花几天时间做一次完整的架构设计，远比"边做边改"高效。设计阶段发现一个问题的成本，是开发阶段的十分之一，是上线之后的百分之一。
