# 第 14 章：性能架构：从慢 SQL 到系统吞吐

## 1. 从一个真实问题开始

上线两年的订单系统，运营反馈"订单列表页越来越慢了"。用户打开"我的订单"要等三四秒，高峰期甚至超时。

工程师看了下接口日志，发现这个接口的响应时间 P95 已经到了 2.8 秒。拉出慢查询日志一看，罪魁祸首是这条 SQL：

```sql
SELECT * FROM orders 
WHERE user_id = ? AND order_status IN ('PAID','SHIPPED','COMPLETED') 
ORDER BY created_at DESC 
LIMIT 20;
```

这条 SQL 在 EXPLAIN 中显示走了全表扫描。为什么？

查了下索引，orders 表有一个 user_id 的单列索引。但 SQL 里有 order_status 的过滤条件和 created_at 的排序。MySQL 选择了 user_id 索引来过滤，但排序需要 filesort，数据量一大就很慢。

工程师加了一个联合索引 `(user_id, order_status, created_at)`，响应时间从 2.8 秒降到了 50 毫秒。

问题解决了？没有。两周后运营又反馈"订单搜索接口慢了"。这次是客服后台的订单搜索——按手机号搜索订单。这个查询需要先查 user 表拿到 user_id，再查 orders 表。两个查询加起来又超过了 2 秒。

再后来，"订单导出功能慢了"——一次导出 10 万条订单，直接 OOM。

然后是"大促期间整个系统变慢了"——不是某一条 SQL 的问题，而是数据库连接池被打满了，所有请求都在排队。

**性能问题就像打地鼠——修好一个冒出另一个。** 根本原因是系统没有从整体上考虑性能架构，只是在问题出现后逐个修补。

## 2. 问题为什么会变复杂

### 2.1 性能问题的特点：不是 bug，是渐变

Bug 是突然出现的——昨天还好好的，今天就报错了。性能问题是渐变的——系统上线时响应很快，数据量从一万涨到一百万时开始变慢，涨到一千万时变得不可接受。

这种渐变导致性能问题经常被忽视。"响应时间从 100ms 变成了 300ms"——慢了点，但还能用。等变成 3 秒的时候才被重视，但这时候可能已经需要大的改造才能解决。

### 2.2 性能瓶颈的层次

一个请求的响应时间是多个环节叠加的结果：

- **网络层**：DNS 解析、TCP 连接、数据传输。
- **应用层**：代码逻辑的执行时间、锁等待、线程调度。
- **数据层**：SQL 执行时间、连接池等待、磁盘 IO。
- **外部依赖**：调用其他服务的网络延迟和等待时间。

性能优化的第一步是**定位瓶颈在哪一层**。优化一个不是瓶颈的环节，对总体性能没有帮助。

### 2.3 单点优化 vs 系统优化

把一条慢 SQL 优化到毫秒级是单点优化。但如果每个请求要执行十条 SQL、调两个外部服务、做三次缓存查询，即使每个环节都很快，总时间也不短。

系统级的性能优化关注的是：减少不必要的操作、并行化可以并行的操作、异步化不需要同步的操作。

## 3. 核心概念解释

### 3.1 性能指标

**响应时间（RT）**：一个请求从发出到收到响应的时间。通常关注 P50（中位数）、P95（95%的请求在这个时间内完成）和 P99。P50 代表典型用户体验，P95/P99 代表长尾用户体验。

**吞吐量（QPS/TPS）**：每秒处理的请求数。QPS 关注查询，TPS 关注事务（写操作）。

**并发数**：同时处理的请求数量。并发数 = QPS × 平均响应时间。如果 QPS 是 1000，平均 RT 是 100ms，那么并发数约为 100。

这三个指标相互关联：响应时间变长 → 并发请求堆积 → 吞吐量下降 → 更多请求排队 → 响应时间进一步变长。这是性能恶化的典型连锁反应。

### 3.2 数据库性能优化

数据库是大多数系统的性能瓶颈所在。

**索引优化**：最基本也是效果最明显的优化。常见的索引问题：
- 缺少索引，走了全表扫描。
- 联合索引的列顺序不对。`(user_id, created_at)` 能支持"查某用户最近的订单"，但 `(created_at, user_id)` 不能高效地支持这个查询。
- 索引失效：对索引列做函数运算（`WHERE YEAR(created_at) = 2024`）、隐式类型转换、使用 `!=` 或 `NOT IN` 等。

**查询优化**：
- 避免 `SELECT *`，只查需要的字段。减少数据传输和内存消耗。
- 避免大 offset 分页。`LIMIT 100000, 20` 需要扫描前十万条再取二十条。用游标分页代替：`WHERE id > last_id LIMIT 20`。
- 避免在循环中查数据库（N+1 问题）。批量查一次代替循环查 N 次。

**连接池**：数据库连接是稀缺资源。连接池太小会导致请求排队等待连接。连接池太大会导致数据库端的连接数过多。通常的经验值：连接池大小 = 2 × CPU 核数 + 磁盘数。

### 3.3 缓存

缓存是性能优化最常用的手段。把频繁读取、不经常变化的数据放在内存中，避免每次都查数据库。

**缓存适合的场景**：
- 读多写少：商品详情、用户信息。
- 数据变化后可以接受短暂的不一致：订单列表（缓存延迟几秒更新）。
- 计算成本高的结果：报表数据、排行榜。

**缓存不适合的场景**：
- 数据实时性要求高：账户余额、库存数量。
- 数据更新频率和读取频率差不多：实时计数器。

**缓存的常见问题**：

**缓存穿透**：查询一个不存在的数据，缓存中没有，每次都打到数据库。解决方法：缓存空值（设置较短的过期时间），或用布隆过滤器拦截。

**缓存击穿**：热点数据的缓存过期后，大量请求同时打到数据库。解决方法：用互斥锁让只有一个请求去加载缓存，其他请求等待。

**缓存雪崩**：大量缓存同时过期。解决方法：缓存过期时间加随机偏移。

### 3.4 异步化

把不需要同步返回结果的操作改为异步执行。

同步方式：用户下单 → 锁库存 → 冻结优惠券 → 创建支付单 → **发通知** → **记日志** → **更新统计** → 返回结果。整个链路 500ms。

异步方式：用户下单 → 锁库存 → 冻结优惠券 → 创建支付单 → 返回结果。发通知、记日志、更新统计放到消息队列异步处理。链路缩短到 200ms。

判断哪些操作可以异步化的标准：
- 用户不需要看到这个操作的结果（通知、日志、统计）。
- 这个操作允许延迟处理（积分发放、报表更新）。
- 这个操作的失败不影响主流程的成功（发通知失败不应该导致下单失败）。

### 3.5 批处理和削峰填谷

**批处理**：把多次小操作合并为一次批量操作。比如库存扣减，不是每个订单单独扣，而是积攒 100ms 内的扣减请求，合并为一条 SQL 批量扣减。

**削峰填谷**：用消息队列缓冲突发流量。大促期间每秒一万个下单请求打到订单服务，订单服务只能处理三千 QPS。加一个消息队列，请求先进队列，订单服务按自己的能力从队列中消费。用户下单后看到的是"订单处理中"，实际订单在队列中排队等待处理。

## 4. 常见误区

### 4.1 "不慢就不优化"

等到用户投诉才开始优化，往往需要大的改造。更好的做法是在日常监控中关注性能趋势。响应时间从 100ms 涨到了 300ms——虽然还在可接受范围内，但趋势不对，应该排查原因。

### 4.2 "加缓存解决一切"

缓存不是银弹。缓存引入了一致性问题——数据更新后缓存中可能还是旧数据。缓存还引入了运维成本——Redis 集群的内存、故障恢复、数据淘汰策略。

在加缓存之前，先看看 SQL 优化和索引优化能不能解决问题。很多时候加个索引就够了，不需要引入缓存的复杂度。

### 4.3 "优化一条 SQL 就够了"

把最慢的 SQL 优化了，整体还是慢。因为可能有十条"不算慢但也不快"的 SQL，每条 100ms，加起来就是 1 秒。

性能优化要看全链路，不能只看单点。用链路追踪（Tracing）工具查看一个请求在各个环节的耗时分布，找到整体瓶颈。

### 4.4 过早优化

系统 QPS 才 100，就开始做分库分表、做多级缓存、做异步化。这些优化引入了大量复杂度，但系统根本用不上。

**先用简单方案满足当前需求。** 等到性能真的成为瓶颈时再优化，而且优化的方向要基于实际的性能数据，不是猜测。

### 4.5 优化了局部、拖垮了全局

为了优化一个查询的性能，在表上加了六个索引。结果写入性能下降了 50%，因为每次 insert 要维护六个索引。

性能优化要考虑全局影响。读性能和写性能是矛盾的——加索引提升读性能但降低写性能。缓存提升查询性能但增加数据不一致风险。

## 5. 设计方法

### 5.1 性能优化的优先级

按成本从低到高排列：

1. **SQL 和索引优化**：成本最低，效果往往最明显。大部分性能问题可以在这一步解决。
2. **代码逻辑优化**：减少不必要的数据库查询、消除 N+1 问题、减少序列化/反序列化。
3. **缓存**：对读多写少的场景效果好，但引入了一致性和运维复杂度。
4. **异步化**：把非核心操作从主链路中剥离，缩短响应时间。
5. **读写分离**：读压力大时，用从库分担读请求。
6. **分库分表**：数据量太大时的最后手段，复杂度最高。

**优先尝试低成本的方案。** 很多团队跳过 1-3 直接上 5-6，是因为觉得"加索引太简单了，不够酷"。但工程的目标是用最简单的方案解决问题。

### 5.2 性能容量规划

在性能问题出现之前做好容量规划：

- 当前系统的核心指标是多少？（QPS、RT、数据量）
- 按业务增长趋势，半年后/一年后这些指标会到多少？
- 系统在什么水位会到达瓶颈？（数据库 CPU 80%？连接池 90%？）
- 到达瓶颈前需要做什么准备？（加索引？加缓存？加读从库？）

容量规划不需要精确到个位数，但要有大致的判断，避免被性能问题打个措手不及。

## 6. 业务案例

### 6.1 案例：订单列表查询的优化过程

**阶段一：索引优化**

订单列表查询走了全表扫描，加联合索引后响应时间从 2.8s 降到 50ms。但客服后台的搜索接口还是慢——按手机号搜索需要 join user 表和 orders 表。

优化方案：在 orders 表中冗余存储 user_phone 字段（脱敏后），避免 join。

**阶段二：缓存**

用户打开"我的订单"页面的频率很高，但订单数据更新频率相对低。加 Redis 缓存，缓存用户最近的 100 条订单，过期时间 5 分钟。

缓存更新策略：订单状态变化时删除该用户的缓存。下次查询时重新加载。

**阶段三：大数据量导出**

运营要导出 10 万条订单，一次性查出来 OOM。改为流式导出：用游标分页查数据库，每查 1000 条就写入 CSV 文件流，不在内存中积累全量数据。导出任务放到后台异步执行，完成后通知运营下载文件。

**阶段四：大促削峰**

大促期间下单 QPS 是平时的十倍。订单服务扛不住，数据库连接池被打满。方案：在下单入口加消息队列做削峰。用户点击下单后，请求先进入队列，返回"订单处理中"。后端按处理能力消费队列，创建订单后推送通知。同时对订单服务做横向扩容。

## 7. 架构判断清单

- 如果核心接口的 P95 响应时间超过 1 秒，需要排查和优化。
- 如果数据库的慢查询日志每天超过 100 条，说明索引或查询设计有问题。
- 如果数据库 CPU 在非高峰期就超过 60%，需要规划扩容或优化。
- 如果一个查询接口执行了超过 5 条 SQL，需要考虑合并查询或使用缓存。
- 如果系统在平时运行正常但每到高峰期就出问题，说明缺少削峰措施和容量规划。
- 如果加了缓存但缓存命中率低于 50%，说明缓存策略不对（可能缓存了不常被查询的数据）。
- 如果性能优化的方案比原始功能的代码还复杂，需要重新评估是否过度优化。

## 8. 本章小结

性能问题不是 bug，而是随着数据量和流量增长逐渐显现的系统瓶颈。好的性能架构不是一开始就做最复杂的优化，而是按成本从低到高依次尝试：先优化 SQL 和索引，再考虑缓存和异步化，最后才是读写分离和分库分表。

性能优化最重要的前提是**知道瓶颈在哪**。靠猜测做优化往往事倍功半。用监控数据和链路追踪定位到具体的瓶颈点，然后有针对性地优化，是最高效的做法。

对工程师来说，性能意识应该贯穿日常开发：写 SQL 时想一下走不走索引，做查询时想一下数据量大了会怎样，调外部接口时想一下超时了怎么办。这些日常的思考比出了问题后的紧急优化要有效得多。
