# 第 28 章：系统重构：在飞行中换发动机

## 1. 从一个真实问题开始

一个运行了四年的订单系统，当初按单体架构设计，数据库是一个大的 MySQL 实例，所有表都在一个库里。四年间业务增长了十几倍，订单表已经有两亿条记录，高峰期数据库 CPU 经常打到 80%。

团队提出了重构方案：做分库分表，把订单数据按用户 ID 分到 16 个库里。同时把订单服务从单体中拆出来，变成独立部署的微服务。

方案在技术上是合理的。但执行起来才发现问题有多大：

- 系统每天还在跑业务，不能停机。重构必须在系统持续运行的情况下完成；
- 旧系统有上百个查询接口，很多查询不是按用户 ID 来的（比如按时间范围查、按商家查、按状态查），分库分表后这些查询怎么办？
- 旧系统的订单表有 60 多个字段，很多字段的含义连老员工都说不清楚。新系统的数据模型要不要重新设计？
- 迁移两亿条历史数据需要多久？迁移过程中如果有新订单进来怎么办？
- 如果新系统上线后发现问题，怎么回滚？数据已经在新库里了，回到旧库数据怎么同步？

最终这次重构花了八个月。其中方案设计一个月，开发三个月，数据迁移和验证两个月，灰度切换和收尾两个月。中间还经历了两次回滚。

大型系统重构的难度，从来不在"新系统怎么设计"——那是最简单的部分。难在"怎么从老系统平滑过渡到新系统、同时不影响线上业务"。

## 2. 问题为什么会变复杂

### 2.1 重构面对的是正在运行的系统

重构不是做一个新项目。新项目可以从零开始、自由设计、按自己的节奏开发。重构面对的是一个每天有真实流量、真实数据、真实用户在使用的系统。

这意味着：
- 不能停机。业务不会因为技术团队要重构就暂停营业；
- 新旧系统需要共存一段时间。你不能一夜之间把所有流量切到新系统；
- 数据要迁移。历史数据不能丢，还要保证迁移过程中的数据一致性；
- 出了问题要能回滚。新系统上线后发现 bug，必须能快速退回旧系统。

### 2.2 重构的风险是累积的

小的重构——改个函数名、提取个方法、调整个类结构——风险很低，因为影响范围小，测试容易覆盖。

大型重构——换数据库、拆服务、改数据模型、改核心流程——风险是累积的。每个环节都有风险：数据迁移可能遗漏数据、新系统可能有逻辑差异、灰度切换可能导致部分请求走新路径部分走旧路径、回滚可能数据不一致。

这些风险叠加起来，使得大型重构成为技术项目中失败率最高的类型之一。

### 2.3 重构的最大阻力往往不是技术

很多重构项目死在启动之前，不是因为技术方案不行，而是因为得不到业务的支持。

业务方的视角是：系统现在还能用，重构不产生新功能，还要占用开发资源几个月，还有上线风险。有这个时间不如多做几个业务需求。

所以推动重构的第一步不是画架构图，而是用业务能理解的语言说明白：不重构的代价是什么。比如"如果不做分库分表，两个月后大促时数据库大概率扛不住，影响的是千万级的交易额"。

## 3. 核心概念解释

### 3.1 重构的分类

根据影响范围，重构可以分为三类：

**代码级重构**：不改变系统的外部行为，只改善代码的内部结构。比如提取方法、消除重复、重命名变量。风险最低，可以在日常开发中持续进行。

**模块级重构**：改变模块的职责划分、接口设计、数据结构。比如把一个大 Service 拆分为多个小 Service，重新设计某个模块的数据模型。风险适中，需要测试覆盖。

**系统级重构**：改变系统的整体架构。比如从单体到微服务、数据库拆分、更换核心技术组件。风险最高，需要完整的迁移方案和回滚计划。

本章主要讨论系统级重构。

### 3.2 重构的核心原则

**原则一：小步前进。** 不要试图一步到位。把大的重构拆分为多个小步骤，每个步骤都是可验证、可回滚的。

**原则二：新旧并行。** 在切换之前，新旧系统要能同时运行。流量通过灰度策略逐步切到新系统，而不是一刀切。

**原则三：数据为王。** 代码重写了可以修 bug，但数据丢了或者数据不一致，修复的成本极高。数据迁移和数据验证是重构中最需要谨慎的部分。

**原则四：可回滚。** 每一步都要有回滚方案。如果新系统出了问题，能在多久内退回旧系统？回滚时数据怎么同步？

### 3.3 灰度切换

灰度切换是大型重构中最关键的技术手段。核心思想是：不是一次性把所有流量切到新系统，而是逐步增加新系统承担的流量比例。

常见的灰度策略：
- **按用户切**：先让 1% 的用户走新系统，观察一周，没问题再扩大到 10%、50%、100%；
- **按业务线切**：先切流量最小的业务线，稳定后再切大的；
- **按功能切**：先切读接口（查询），稳定后再切写接口（创建、修改）。

灰度期间需要做的事：
- 新旧系统的结果做比对（对比返回的数据是否一致）；
- 监控新系统的错误率、延迟、数据一致性；
- 准备快速回滚的开关（一个配置改动就能把流量切回旧系统）。

## 4. 常见误区

### 4.1 "先全部开发完再上线"

把新系统完整开发完，然后一次性切换。这种做法的风险是：新系统开发了三个月，上线时发现大量问题，但已经没有退路了。

更好的做法是：开发一个功能就灰度一个功能。比如先把订单查询接口切到新系统，验证没问题后再切订单创建接口。

### 4.2 "数据迁移一次搞定"

两亿条数据的迁移不可能"一次搞定"。迁移是一个持续的过程：先做全量迁移，然后做增量同步，最终在切换的瞬间追平差异。

迁移过程中最容易出问题的是增量同步——旧系统还在不断产生新数据，这些数据需要实时同步到新系统。同步有延迟、有丢失的风险，需要有对账机制来发现差异。

### 4.3 "重构就是重写"

重构不等于推倒重来。很多团队一说重构就是"用新技术从头写一遍"。但从头写意味着：丢掉了旧系统中多年积累的业务逻辑和边界条件处理。

很多"看起来没用"的 if-else，其实是在处理某个历史遗留的边界情况。重写时如果不把这些逻辑迁移过去，上线后就会出现"旧系统能处理但新系统处理不了"的问题。

更稳妥的做法是：**渐进式改造**，而不是推倒重来。能复用的逻辑就复用，只改需要改的部分。

### 4.4 忽视回滚方案

有些团队做重构时只想着"怎么上线"，没有想"上线后出问题怎么退"。

回滚方案需要回答的问题：
- 回滚时流量怎么切回旧系统？需要多长时间？
- 新系统已经产生的数据怎么同步回旧系统？
- 如果已经有部分用户在新系统上操作了，回滚后他们的数据还在吗？
- 回滚后旧系统能不能承受全量流量？（灰度期间旧系统可能已经缩容了）

### 4.5 低估验证成本

重构项目中，开发工作量通常只占 40-50%，剩下的时间花在数据迁移验证、功能回归测试、性能测试、灰度观察上。

很多团队把所有时间都排给了开发，留给验证的时间不够。结果匆忙上线，上线后发现一堆问题。

## 5. 设计方法

### 5.1 重构前的评估

在决定是否启动重构之前，需要评估几个问题：

1. **不重构的代价有多大？** 是"不舒服但能忍"，还是"再不改系统就要出大问题"？
2. **重构的范围有多大？** 能不能缩小范围，只重构最痛的部分？
3. **重构需要多少资源？** 需要几个人、几个月？需要暂停多少业务需求？
4. **风险有多高？** 涉及数据迁移吗？涉及核心交易流程吗？
5. **有没有更轻量的替代方案？** 比如不分库分表，而是做读写分离或者归档历史数据。

### 5.2 分步骤实施

以"订单系统从单库到分库分表"为例，拆分为以下步骤：

**第一步：双写（1-2 周）**

新建分库分表的目标数据库。修改写操作，同时写入旧库和新库。读操作仍然走旧库。

这一步的目的是：让新库开始积累数据，同时不影响线上读操作。

**第二步：数据迁移 + 验证（2-4 周）**

把旧库的历史数据迁移到新库。迁移完成后，用对账工具比对新旧库的数据是否一致。

对账需要覆盖：数据条数是否一致、关键字段的值是否一致、关联关系是否正确。

**第三步：灰度读切换（2-4 周）**

开始把部分读请求切到新库。先切内部使用的非核心查询接口，观察返回结果是否和旧库一致。然后逐步切到核心查询接口。

灰度期间做双读对比：同一个请求同时查新库和旧库，比较结果是否一致。不一致时记录日志并报警。

**第四步：全量读切换（1-2 周）**

所有读操作切到新库。旧库只接收写操作。

**第五步：写切换（1-2 周）**

写操作也切到新库。旧库变成只读的备份。

**第六步：下线旧库（2-4 周观察期）**

全量切换完成后，保留旧库一段时间作为备份。观察期内如果发现问题可以快速回滚。观察期结束后下线旧库。

### 5.3 数据迁移的关键实践

**全量 + 增量**：先做一次全量迁移，然后通过 binlog 监听做增量同步。这样可以在不停机的情况下完成迁移。

**对账**：迁移完成后必须做数据对账。简单的方式是按时间段抽样比对，严格的方式是全量比对。

**幂等**：增量同步可能因为网络抖动重复执行，数据写入逻辑要做幂等处理。

**熔断**：如果增量同步出现大量错误，要有熔断机制暂停同步并报警，而不是持续写入错误数据。

## 6. 业务案例

### 6.1 案例：支付系统从同步到异步的重构

一个支付系统原来是同步架构：用户点击支付 → 调用支付渠道 → 等待结果 → 返回用户。高峰期渠道响应慢时，用户要等十几秒，体验很差，而且线程池经常被打满。

重构目标：改为异步架构。用户点击支付后立即返回"支付处理中"，后台异步调用渠道，渠道回调后更新状态并通知用户。

**重构步骤：**

1. **新建异步支付链路**：保留旧的同步链路不动，新增一条异步链路。两条链路共用同一个数据库。

2. **灰度切换**：通过配置开关控制新订单走同步还是异步。先让 5% 的订单走异步，观察成功率和延迟。

3. **扩大灰度**：5% → 20% → 50% → 100%。每次扩大前确认：异步链路的成功率和同步链路持平、用户投诉率没有上升、对账没有差异。

4. **下线同步链路**：全量切到异步后，观察两周，然后下线同步链路的代码。

**遇到的问题和处理：**

- 灰度到 20% 时发现异步链路的超时处理有 bug：渠道超时后没有正确触发补偿，导致部分订单卡在"支付中"。紧急把灰度比例回退到 5%，修复后重新扩大；
- 异步架构下用户看到的状态可能有延迟，产品团队配合调整了前端的展示逻辑（显示"支付处理中，预计几秒内完成"）。

### 6.2 从案例中总结重构的关键要素

1. **新旧并行是必须的**。不管重构的是什么，都要让新旧逻辑共存一段时间，通过灰度逐步切换。

2. **灰度比例的控制要谨慎**。不要一下子从 5% 跳到 100%。每次扩大都要有充分的验证数据支撑。

3. **回滚能力是生命线**。灰度期间任何时候都能快速回退到旧系统，是重构成功的前提条件。

4. **重构的成功标准不是"新系统上线了"，而是"旧系统安全下线了"**。

## 7. 架构判断清单

- 如果一次重构无法拆成小步验证，说明方案风险过高，需要重新设计实施路径。
- 如果重构方案没有回滚计划，说明还没准备好上线。
- 如果数据迁移后没有做对账验证，说明数据质量不可信。
- 如果灰度策略是"一次性切 100%"，说明没有灰度——这是上线，不是灰度。
- 如果重构的目标不能用业务语言解释（比如"不重构的话大促扛不住"），说明推动起来会很困难。
- 如果重构的验证时间少于开发时间的一半，说明验证可能不充分。
- 如果旧系统在灰度期间已经被关停或缩容到无法承载全量流量，说明回滚能力已经丧失。

## 8. 本章小结

系统重构是技术项目中风险最高的类型。它的难度不在于"设计一个更好的系统"——给一张白纸谁都能画出漂亮架构。难度在于从当前正在运行的系统平滑过渡到新系统，同时不影响业务、不丢数据、出了问题能回滚。

控制重构风险的关键是三个字：**小步走**。把大的改造拆成小的步骤，每一步都可验证、可回滚。新旧系统并行运行，通过灰度策略逐步切换流量。数据迁移后做对账验证。出了问题能快速退回。

重构的勇气不来自技术能力，而来自对风险的充分准备。当你能清楚地回答"如果出了问题我们怎么办"时，重构就可以启动了。如果这个问题回答不清楚，那再好的技术方案也不应该贸然上线。
