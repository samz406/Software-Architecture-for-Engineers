# 第 35 章：架构师如何提升判断力

## 1. 从一个真实问题开始

两个架构师面对同一个问题——支付系统需要支持多渠道路由。

架构师 A 花了三周设计了一个"完善"的方案：引入规则引擎、支持动态配置、可以根据渠道费率、成功率、延迟等多个维度自动选择最优渠道。方案设计精巧，技术上无可挑剔。

架构师 B 花了两天设计了一个"简单"的方案：配置表里写三行——微信优先、支付宝备选、银联兜底。渠道不可用时按顺序降级。

三个月后复盘，架构师 B 的方案上线了、运行良好。架构师 A 的方案还在开发中——规则引擎的配置太复杂，运营不会用。

**架构师 A 的方案技术上更好，但架构师 B 的方案更适合当前阶段。** 只有三个渠道的时候，规则引擎是过度设计。

这种"什么时候该简单、什么时候该复杂"的判断，就是架构师的判断力。它不是靠知识积累就能获得的——你可以知道规则引擎怎么实现，但"该不该在这个场景用规则引擎"是另一回事。

## 2. 问题为什么会变复杂

### 2.1 判断力不等于知识量

知道的方案多不代表判断力强。知道十种缓存策略是知识，在具体场景中选出最合适的那一种是判断力。

有些工程师读了很多技术文章和书籍，知道各种设计模式和架构风格，但面对实际问题时仍然做出不合适的选择——因为他们的知识没有和实际场景的经验关联起来。

### 2.2 判断力需要"踩坑"来磨

最深刻的判断力来自亲身经历的失败。

"我以前在一个项目中用了规则引擎做路由，结果配置复杂度太高，运营不会用，最后还是简化成了配置表。"——这种一手的失败经验，比看十篇"规则引擎最佳实践"有用得多。

但不是每个人都有机会在足够多的项目中踩够多的坑。所以需要从别人的经验中学习——看案例、听复盘、分析失败。

### 2.3 判断力有时效性

技术在变化，业务在变化，团队在变化。三年前"在这个场景下用 MySQL 而不是 MongoDB"是对的，三年后可能因为数据模型变复杂了，MongoDB 变成了更好的选择。

判断力需要持续更新，不能固化成教条。

## 3. 核心概念解释

### 3.1 判断力的本质：在不确定中做合理的选择

架构决策的特点是：信息不完整、未来不确定、多个维度互相矛盾。

用 MySQL 还是 ES 做订单查询？MySQL 简单可靠但查询灵活性有限，ES 查询灵活但引入了数据同步复杂度。两个都有道理，没有绝对正确的答案。

判断力就是在这种不确定中做出"当前最合理的选择"的能力。注意不是"最好的选择"——"最好"要等到事后才知道。在决策时刻，能做到"合理"就已经很好了。

### 3.2 判断力的构成

判断力由三个部分组成：

**场景识别**：这个问题属于什么类型？和我之前遇到过的哪个场景类似？有什么不同？

比如看到"系统变慢了"，经验丰富的架构师会快速判断：是数据量增长导致的？是并发增加导致的？是某个下游变慢导致的？不同的根因对应不同的解决方向。

**方案库**：针对这类问题有哪些可选的方案？每个方案的适用条件和代价是什么？

这是知识的部分。但光有方案库不够——还需要知道每个方案在什么条件下有效、在什么条件下失效。

**取舍感**：在当前约束下，哪个方案是最务实的选择？

这是判断力最核心也最难的部分。它需要综合考虑技术因素、业务因素、团队因素、时间因素。

### 3.3 提升判断力的五个方法

**方法一：复盘事故和故障**

每次线上故障都是提升判断力的机会。但很多团队的故障复盘停留在"找到了根因、修了 bug"。

更深层的复盘应该问：为什么系统设计没有防住这种故障？是缺少了什么机制（超时、熔断、降级）？当初设计时有没有考虑过这种场景？如果考虑了但觉得概率低不处理，这个判断对不对？

**方法二：分析失败的系统**

成功的系统各有各的成功之道，失败的系统却有共性的问题。分析一个"为什么这个系统变成了大泥潭"，能学到的东西比分析"为什么这个系统设计得好"更多。

常见的失败模式：
- 过早的优化——系统还没有用户就做了分库分表。
- 错误的边界——把强耦合的功能拆成了两个服务。
- 忽视运维——方案设计得好但运维成本太高，团队扛不住。

**方法三：做方案对比练习**

每次做技术决策时，强迫自己列出至少两个方案，写清楚每个方案的优劣和适用条件。即使最终选了第一个方案，这个对比过程也在训练判断力。

时间长了，你会积累一个"方案 A 在条件 X 下比方案 B 好，在条件 Y 下相反"的心智模型。这个心智模型就是判断力的基础。

**方法四：理解业务约束**

很多看似"技术不合理"的决策，在业务约束下是合理的。

"为什么不做分布式事务，直接用补偿？"——因为分布式事务的性能开销在这个场景下不可接受，业务上可以容忍短暂的不一致。

理解业务约束能帮你做出更务实的判断，而不是追求技术上的"完美"。

**方法五：记录并回顾决策**

用 ADR 记录每次架构决策。半年后回顾：当初的判断对不对？结果和预期是否一致？如果重来一次，会做不同的选择吗？

这种回顾是最有效的判断力训练。因为你能看到自己的判断在现实中的验证结果。

## 4. 常见误区

### 4.1 "用最先进的技术就是好架构"

追新不等于判断力强。在团队不熟悉 Kubernetes 的情况下上 Kubernetes，不是好判断。在数据量只有几百万的时候上 ClickHouse，不是好判断。

好的判断是选择当前阶段最合适的技术，不是最先进的。

### 4.2 "别人的经验可以直接套用"

阿里的方案不一定适合你的公司。五万 QPS 下的最佳实践用在五百 QPS 的系统上可能是过度设计。

别人的经验要转化为自己的判断，需要理解"别人为什么这么做"——他面对的约束是什么？他的团队规模是什么？他的业务阶段是什么？和我的情况有什么不同？

### 4.3 "判断力靠直觉"

有人说好的架构师凭"直觉"就知道该怎么做。这种"直觉"不是天生的，而是大量经验积累后的快速模式匹配。

所谓的"直觉判断"背后是几百次的成功和失败、几十次的复盘和总结。没有这些积累，"直觉"只是"猜"。

### 4.4 追求"绝对正确"的决策

在信息不完整的情况下，不存在绝对正确的架构决策。等到信息完整的时候，往往已经来不及做决策了。

好的判断力不是"每次都做对"，而是"在不确定中做出合理的选择，并且准备好应对判断错误的情况"。

## 5. 设计方法

### 5.1 判断力的检查清单

做架构决策时，用以下问题检查自己的判断：

1. **我理解了真实的问题吗？** 还是只看到了表面现象？
2. **我考虑了几个方案？** 如果只有一个，可能没有充分思考。
3. **我的选择基于什么假设？** 这些假设成立吗？如果不成立会怎样？
4. **这个方案在当前约束下可行吗？** 团队能实现吗？时间够吗？
5. **最坏情况是什么？** 如果这个决策是错的，后果有多严重？能补救吗？
6. **半年后回看这个决策，我会后悔吗？** 有没有明显的风险被忽视了？

### 5.2 从复盘中提炼判断力

每个季度做一次技术复盘，回顾最近做过的重要技术决策：

```markdown
## 决策复盘

### 决策：订单查询用 MySQL 索引优化而不是引入 ES
- 当时的判断：数据量 8000 万，MySQL 加索引能搞定
- 实际结果：加索引后 P95 从 2.8s 降到 50ms，效果好
- 回顾：判断正确。当时数据量确实不需要 ES

### 决策：用数据库 Outbox 而不是 Kafka 做异步
- 当时的判断：团队不熟悉 Kafka，Outbox 更简单
- 实际结果：Outbox 方案可用但轮询间隔导致延迟较高
- 回顾：判断基本正确，但应该更早规划向 Kafka 迁移
```

## 6. 业务案例

### 6.1 案例：同一个团队在不同阶段做出不同判断

**第一年**：团队选择了单体架构，把所有模块放在一个 Spring Boot 应用里。

有人建议用微服务。团队判断：六个人的团队，业务还在摸索期，单体是更合适的选择。

结果：判断正确。单体架构让团队在第一年快速迭代了三十多个功能，没有被基础设施拖慢。

**第二年**：订单查询变慢了。有人建议上 ES。

团队判断：当前数据量两千万，加个联合索引够了。ES 的运维成本和数据同步复杂度在当前阶段不值得引入。

结果：判断正确。加索引后性能恢复，投入了半天时间。

**第三年**：团队到了二十人，发布冲突频繁。有人建议拆微服务。

团队判断：确实需要拆了——不是因为"微服务更先进"，而是因为发布冲突和代码合并已经成为实际的效率瓶颈。先拆订单服务和支付服务（变化最频繁的两个模块），其他保持在单体中。

结果：拆分后发布冲突减少 70%，两个团队可以独立发布。判断正确。

**三年中的三个判断有一个共同点：基于当前的实际情况做选择，而不是基于"技术趋势"或"别人都在用"。** 这就是判断力。

## 7. 架构判断清单

- 如果你做了一个架构决策但说不清楚"为什么不选另一个方案"，说明判断的过程不够充分。
- 如果你在不同的项目中总是选同一种技术方案，可能是"锤子思维"——只有一把锤子，看什么都像钉子。
- 如果你半年前做的决策现在已经证明不合适，复盘一下当时的判断过程——不是为了自责，而是为了下次做得更好。
- 如果团队里只有一个人做架构决策且从不被质疑，说明缺少多元视角——判断力在被挑战中才能提升。

## 8. 本章小结

判断力是架构师最核心的能力，也是最难培养的能力。它不等于知识量，不等于经验年限，也不等于技术深度。

判断力由三个部分组成：场景识别（这是什么类型的问题）、方案库（有哪些可选方案）、取舍感（在当前约束下哪个最合适）。

提升判断力的方法：复盘事故、分析失败系统、做方案对比练习、理解业务约束、记录并回顾决策。所有这些方法的核心都是**在实践中积累、在反思中沉淀**。

好的判断力不是"每次都做对"，而是在不确定中做出合理选择的稳定能力。
