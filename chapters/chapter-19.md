# 第 19 章：事件驱动架构：从同步调用到异步协作

## 1. 从一个真实问题开始

订单支付成功后，系统需要做八件事：

1. 更新订单状态为"已支付"
2. 扣减库存
3. 核销优惠券
4. 发放积分
5. 更新会员成长值
6. 发送短信通知用户
7. 推送站内消息
8. 更新实时销售报表

最初的实现是在支付回调的处理方法里，顺序调用八个模块：

```java
public void onPaymentSuccess(PaymentCallback callback) {
    orderService.updateStatus(callback.getOrderId(), PAID);
    inventoryService.deductStock(callback.getOrderId());
    couponService.redeemCoupon(callback.getOrderId());
    pointService.grantPoints(callback.getUserId(), callback.getAmount());
    memberService.addGrowthValue(callback.getUserId(), callback.getAmount());
    smsService.sendPaymentNotice(callback.getUserId());
    messageService.pushInAppMessage(callback.getUserId());
    reportService.updateSalesData(callback.getOrderId());
}
```

这段代码有什么问题？

**性能问题**：八个调用串行执行。每个平均 100ms，总共 800ms。支付渠道的回调通常有超时限制（比如 5 秒），如果再加几个下游，可能超时。

**可靠性问题**：短信服务挂了，整个支付回调处理失败。但短信发不出去不应该影响"支付成功"这个核心结果。

**耦合问题**：支付模块直接依赖了八个下游模块。每次新增一个"支付成功后要做的事"，都要修改支付模块的代码。但支付模块的开发者可能根本不了解积分规则和报表逻辑。

**扩展问题**：产品说"支付成功后还要触发拼团逻辑"。工程师要改支付回调代码。产品又说"支付成功后还要通知供应商发货"。又改。支付回调的方法越来越长，越来越脆弱。

后来团队做了一个改造：支付回调只做一件事——更新订单状态并发出一个 `OrderPaid` 事件。其他七件事各自监听这个事件来处理。

改造后支付模块不再依赖任何下游。新增"支付成功后的操作"不需要改支付模块——只需要新增一个事件的消费者。

## 2. 问题为什么会变复杂

### 2.1 同步调用的"扇出"问题

当一个操作需要通知多个下游时，同步调用的问题会被放大。每增加一个下游，主链路的延迟增加一点、可靠性降低一点、耦合增加一点。

这种"一对多"的通知场景在业务系统中非常常见：
- 支付成功后通知八个下游。
- 订单取消后退券、退库存、退积分。
- 用户注册后发欢迎短信、创建默认配置、初始化数据。
- 商品下架后关闭相关活动、通知商家。

### 2.2 同步调用导致"核心和非核心不分"

在上面的例子中，"更新订单状态"和"更新报表"的重要性完全不同。订单状态更新是核心操作——失败了用户的支付就白做了。报表更新是非核心操作——晚几秒甚至晚几分钟都没事。

但在同步调用模式下，它们被同等对待。报表服务挂了，和订单服务挂了的后果一样——整个支付回调失败。

事件驱动架构的一个核心价值就是**分离核心操作和非核心操作**。核心操作同步完成，非核心操作通过事件异步完成。

### 2.3 系统间的"知识负担"

同步调用意味着调用方需要知道所有下游的存在。支付模块要知道"支付成功后要调库存、调优惠券、调积分……"。这些知识本不应该属于支付模块。

事件驱动反转了这个关系：支付模块只需要知道"我支付成功了"，发出这个事件。至于谁关心这个事件、拿到事件后做什么，支付模块不需要知道。

## 3. 核心概念解释

### 3.1 什么是事件

事件是**对已经发生的事实的记录**。"订单已支付""库存已扣减""用户已注册"——这些都是事件。

事件和命令（Command）的区别：
- **命令**是"请你做某事"：`DeductStock(orderId, skuId, quantity)`
- **事件**是"某事已经发生了"：`OrderPaid(orderId, userId, amount)`

命令是有方向的——A 告诉 B 做什么。事件是广播的——A 说"这事发生了"，谁关心谁来听。

### 3.2 事件驱动的基本模式

**发布-订阅（Pub/Sub）**：事件的生产者发布事件到消息中间件（如 Kafka、RabbitMQ），消费者订阅感兴趣的事件。生产者不知道有哪些消费者，消费者也不知道其他消费者的存在。

```
支付服务 → [OrderPaid事件] → 消息中间件
                                ├→ 库存服务（扣库存）
                                ├→ 优惠券服务（核销）
                                ├→ 积分服务（发积分）
                                ├→ 通知服务（发短信）
                                └→ 报表服务（更新报表）
```

### 3.3 事件 vs 消息

在日常使用中，"事件"和"消息"经常混用。但严格来说有区别：

**事件**表达的是"发生了什么"，是一个事实的广播。一个事件可以被多个消费者独立处理。

**消息**更通用，可以是命令（"请处理这个任务"）也可以是事件。消息通常是点对点的——发给一个特定的队列，被一个消费者处理。

在实际项目中，Kafka 更适合事件驱动场景（天然支持多消费者组），RabbitMQ 更适合任务队列场景（点对点消息）。很多团队两个都用——事件用 Kafka，任务队列用 RabbitMQ。

### 3.4 可靠投递

事件驱动架构最大的挑战是：**怎么保证事件不丢？**

场景：支付成功后，先更新订单状态，再发送 `OrderPaid` 事件。如果订单状态更新成功了，但发送事件时服务崩溃了，事件就丢了。库存没扣、积分没发、用户没收到通知。

解决方案是**本地消息表**（Outbox Pattern）：

```java
@Transactional
public void handlePaymentSuccess(Long orderId) {
    // 在同一个事务中：更新订单状态 + 写消息记录
    orderRepository.updateStatus(orderId, PAID);
    outboxRepository.insert(new OutboxMessage("OrderPaid", orderId));
}

// 定时任务或CDC：读取消息记录，发送到消息中间件
@Scheduled(fixedDelay = 1000)
public void publishPendingMessages() {
    List<OutboxMessage> messages = outboxRepository.findPending();
    for (OutboxMessage msg : messages) {
        kafka.send(msg.getTopic(), msg.getPayload());
        outboxRepository.markSent(msg.getId());
    }
}
```

核心思路：事件记录和业务操作在同一个数据库事务中——要么都成功，要么都失败。事件的实际发送由定时任务或 CDC（Change Data Capture）异步完成。

### 3.5 重复消费和幂等

消息中间件通常保证"至少投递一次"（at-least-once），但不保证"只投递一次"。消费者可能收到重复的事件。

因此**消费者必须是幂等的**。处理同一个 `OrderPaid` 事件两次，效果和处理一次应该相同。

实现方式：
- 消费前检查是否已处理过（通过事件 ID 或业务唯一键）。
- 利用状态机：库存已经扣减过了（状态是"已扣减"），再次收到扣减事件就跳过。

### 3.6 消息顺序

在某些场景中，事件的处理顺序很重要。比如"订单已创建"和"订单已取消"这两个事件，如果消费者先处理了"取消"再处理"创建"，就会出问题。

保证顺序的方式：
- Kafka 中同一个 partition 内的消息是有序的。把同一个订单的事件发到同一个 partition（按 orderId 做 partition key）。
- 消费者按状态机处理：如果当前订单状态不匹配（比如要处理"取消"但订单还不存在），先暂存，等前序事件到达后再处理。

## 4. 常见误区

### 4.1 "所有调用都改成事件"

不是所有场景都适合事件驱动。下单时查库存够不够——这个操作需要立即拿到结果，不适合异步事件。

事件驱动适合的场景：
- 不需要立即拿到结果的操作。
- 一个操作触发多个下游的场景。
- 核心操作和非核心操作需要分离的场景。

不适合的场景：
- 需要同步返回结果的查询。
- 强一致性要求的操作（如扣款必须立即确认）。

### 4.2 事件内容太少

有些团队的事件只包含一个 ID：`OrderPaid { orderId: 123 }`。消费者收到事件后，需要回调订单服务查询订单详情。这导致：
- 订单服务被大量回调查询打满。
- 消费者和生产者形成了隐式的同步依赖。

更好的做法是在事件中携带消费者可能需要的核心信息：

```json
{
    "event": "OrderPaid",
    "orderId": "ORD20240115001",
    "userId": 12345,
    "amount": "99.00",
    "couponId": "CPN001",
    "items": [
        {"skuId": "SKU001", "quantity": 2}
    ],
    "paidAt": "2024-01-15T10:30:00+08:00"
}
```

消费者拿到事件就能直接处理，不需要回查。

### 4.3 忽视死信队列

消费者处理事件时可能失败（bug、下游不可用、数据异常）。如果一直重试，可能会阻塞后续消息的处理。

应该设置**死信队列**：消费失败超过重试次数后，消息进入死信队列。单独处理死信队列中的消息——人工检查、修复后重新消费。

### 4.4 忽视事件的监控

事件驱动架构中，"看不见"是最大的问题。事件发出后，消费者有没有收到？处理成功了没有？延迟多久？

需要监控：
- 事件的生产速率和消费速率。如果生产速率持续大于消费速率，消息会积压。
- 消费延迟。事件从发出到被消费的时间。
- 消费失败率。失败率上升说明消费者有问题。
- 死信队列的消息数量。不应该持续增长。

### 4.5 事件模型不设计就开写

事件的结构（包含哪些字段、字段的语义）需要提前设计。因为事件一旦发布，消费者就会依赖它的结构。后续改事件结构需要考虑所有消费者的兼容性。

事件的设计原则：
- 事件名称用过去时（OrderPaid，不是 PayOrder）。
- 包含足够的信息让消费者独立处理。
- 有版本号，方便将来演进。

## 5. 设计方法

### 5.1 从同步调用迁移到事件驱动

不需要一步到位。渐进式迁移：

**第一步**：识别"一对多"的同步调用场景。找到那些一个操作需要通知多个下游的地方。

**第二步**：引入消息中间件。先用最简单的方式——生产者发消息到 Kafka，消费者订阅。

**第三步**：把非核心下游从同步改为异步。支付成功后的八个操作，先把发短信、推消息、更新报表改成事件消费。扣库存和核销优惠券先保持同步（因为它们是核心操作）。

**第四步**：验证稳定后，逐步把更多操作改为事件驱动。

### 5.2 事件命名规范

```
{领域}.{实体}.{动作}

例如：
order.order.paid         -- 订单已支付
order.order.cancelled    -- 订单已取消
payment.refund.completed -- 退款已完成
inventory.stock.deducted -- 库存已扣减
```

统一的命名规范让团队能快速理解事件的来源和含义。

## 6. 业务案例

### 6.1 案例：订单支付成功后的异步通知链

**改造前**：

```
支付回调 → 更新订单状态(同步) → 扣库存(同步) → 核销优惠券(同步) 
         → 发积分(同步) → 更新会员(同步) → 发短信(同步) 
         → 推消息(同步) → 更新报表(同步)
总耗时：800ms
任何一个环节失败：整个回调失败
```

**改造后**：

```
支付回调 → 更新订单状态(同步) → 发出OrderPaid事件
总耗时：100ms

事件消费者（各自独立）：
├→ 库存服务：扣减库存
├→ 优惠券服务：核销优惠券
├→ 积分服务：发放积分
├→ 会员服务：更新成长值
├→ 通知服务：发短信+推消息
└→ 报表服务：更新销售数据
```

效果：
- 支付回调响应时间从 800ms 降到 100ms。
- 通知服务挂了不影响支付。
- 新增一个"支付后的操作"只需要新增一个消费者，不改支付代码。
- 各消费者独立处理、独立重试、独立扩容。

### 6.2 遇到的问题和处理

- **问题一**：积分服务消费延迟，用户支付后十分钟才看到积分。原因是积分服务的消费者线程数太少。解决：增加消费者实例和线程数。
- **问题二**：订单取消事件和支付成功事件乱序，导致"支付后立即取消"的场景中库存没有正确回退。解决：按 orderId 做 partition key 保证同一订单的事件有序。
- **问题三**：通知服务bug导致大量消息进入死信队列。解决：修复 bug 后用脚本将死信队列消息重新投递。

## 7. 架构判断清单

- 如果一个操作需要同步通知三个以上的下游，适合改为事件驱动。
- 如果非核心操作的故障会导致核心操作失败（发短信失败导致支付失败），说明核心和非核心没有分离。
- 如果新增一个"某事件发生后的操作"需要修改事件生产者的代码，说明没有真正解耦。
- 如果事件只包含一个 ID，消费者需要回查生产者才能处理，说明事件内容不够丰富。
- 如果消息队列中有大量积压消息且持续增长，说明消费者的处理能力不足。
- 如果没有死信队列或者死信队列从不被处理，说明异常消息被静默丢弃了。
- 如果无法回答"这个事件发出后被哪些消费者处理了、处理成功没有"，说明事件链路的可观测性不够。

## 8. 本章小结

事件驱动架构解决的核心问题是：一个操作需要触发多个后续操作时，如何降低耦合、提高性能、分离核心和非核心。

它不是取代同步调用的银弹——需要同步返回结果的场景仍然用同步调用。但对于"一对多通知""核心/非核心分离""下游数量会增长"的场景，事件驱动是比同步调用更好的选择。

引入事件驱动后需要关注的新问题：事件的可靠投递（本地消息表）、重复消费（幂等）、消息顺序（partition key）、异常处理（死信队列）、可观测性（监控延迟和积压）。这些问题是分布式系统的固有复杂性，不是事件驱动"制造"的问题——同步调用也有这些问题，只是表现形式不同。
