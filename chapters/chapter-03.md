# 第 3 章：从代码思维到架构思维

## 1. 从一个真实问题开始

小张是一个三年经验的后端工程师。他写代码很快，接到需求后能迅速把功能跑通，单元测试覆盖率也不错。团队里如果有紧急需求，经常会找他来赶工期。

有一次他接到一个需求：给订单系统增加"满减优惠"能力。他评估了一下，觉得不难——在订单创建的时候判断满减条件，扣减金额写入订单表的 discount_amount 字段，支付金额等于原价减去折扣。他两天就写完了，代码清晰、逻辑正确、测试通过。

一个月后，产品说要加"优惠券叠加"：一张订单可以同时使用满减和优惠券。小张继续在 OrderService 里加逻辑。

又过了两周，产品说"退款时要按比例退优惠券"。小张发现退款服务需要知道优惠券的面值、满减的比例、以及当时的叠加规则，而这些信息散落在订单表、优惠活动表和券实例表里。他不得不在退款接口里拼装这些数据，加了四五十行查询和计算。

再后来，财务要求对账时能还原每笔订单的原始优惠金额。客服要求在后台能查到"这张券为什么抵了这么多钱"。运营要求"满减规则改了之后老订单不受影响"。

每个需求单独看都合理，每次改动单独看都能跑通。但半年后回头看，优惠相关的逻辑已经扩散到了 OrderService、PaymentService、RefundService、CouponService 四个服务里。每个服务里都有一段"计算优惠金额"的逻辑，但规则不完全一样，因为是不同时期不同人写的。

小张的代码从来没有 bug——至少没有他自己能看出来的 bug。但系统开始出现"换个人就不敢改"的症状。

这个故事里的问题不是代码质量，而是**代码思维和架构思维之间的差距**。

## 2. 问题为什么会变复杂

### 2.1 代码思维的核心关注点

代码思维解决的是"怎么实现"。面对一个需求，代码思维关注的是：

- 输入是什么，输出是什么；
- 逻辑分支有哪些；
- 数据从哪里来，写到哪里去；
- 异常怎么处理；
- 性能够不够用。

这些关注点本身没有任何问题。一个工程师如果连这些都做不好，谈架构毫无意义。代码能力是架构能力的基础，不是对立面。

但代码思维有一个隐含的假设：**它默认当前需求的上下文是固定的**。小张在实现满减的时候，不会去想"三个月后退款流程需要反算优惠金额"。因为当时没有这个需求，他也不应该为一个不存在的需求做过度设计。

问题不出在小张某一次的决策上，而出在系统层面没有人在做另一层思考：**这个能力应该放在哪里？它的边界是什么？它会被谁使用？未来可能怎么变？**

### 2.2 架构问题常常伪装成代码问题

很多团队遇到"代码难改"的时候，第一反应是代码写得不好。然后开始搞代码规范、命名规则、方法长度限制。这些措施有价值，但解决不了根本问题。

上面的例子中，即便每段代码都写得非常规范，问题依然存在：

- 优惠金额计算逻辑分散在四个服务里；
- 没有一个统一的"优惠计算"能力出口；
- 退款、对账、客服查询各自拼装自己的计算逻辑；
- 新的优惠规则上线后，四个地方都需要改，但通常只改了三个。

这不是代码风格问题，而是**能力的归属问题**。优惠计算这个能力，应该有一个明确的、统一的、可被复用的位置。它的变化规则应该集中管理。它被其他模块引用时，应该通过清晰的接口，而不是复制一段逻辑。

这就是架构思维要解决的：**不是"这段逻辑怎么写"，而是"这段逻辑应该放在哪、由谁拥有、如何变化"**。

### 2.3 从"实现需求"到"管理变化"

代码思维的终点是"功能跑通"。架构思维的终点是"系统在变化中依然可控"。

工程师在职业早期，接到的任务大多是"实现某个功能"。做久了之后，会发现很多困难并不来自实现本身，而来自变化：

- 需求在变：上线后产品追加条件、改规则、加入口；
- 人在变：原来写这段代码的人离职了，新人不敢动；
- 上下游在变：依赖的系统改了接口、换了数据格式；
- 规模在变：数据量从几千涨到几百万，原来的方案跑不动了。

架构思维的核心就是应对变化。不是预测变化——没有人能预测三个月后的需求。而是让系统具备承接变化的结构：**改一个模块不需要改五个模块，加一种规则不需要在三个服务里各加一段 if-else，替换一个实现不需要重写调用方**。

## 3. 核心概念解释

### 3.1 代码思维 vs 架构思维：四个维度的差异

把代码思维和架构思维的差异拆成四个维度来看，会更容易理解。

**维度一：关注对象**

代码思维关注的是"这个方法的逻辑"。架构思维关注的是"这个能力在系统中的位置"。

比如实现用户注册功能。代码思维关注的是参数校验、密码加密、写入数据库、返回结果。架构思维会多想几步：用户注册这个能力，应该在哪个服务里？注册后要触发哪些后续动作（发欢迎邮件、初始化权限、分配默认配置）？这些动作是同步完成还是异步通知？如果注册量上来了，瓶颈会在哪里？

**维度二：决策范围**

代码思维的决策范围通常在一个方法或一个类里。用什么算法、怎么组织逻辑分支、怎么处理异常，这些都是局部决策。

架构思维的决策范围是跨模块甚至跨系统的。这个数据由谁生产、由谁消费？这个接口被多少个调用方使用？这个变化会沿着什么路径传播？这些问题一旦判断错了，影响面比一个方法写错大得多，而且修复成本也高得多。

**维度三：时间视角**

代码思维通常关注当下——当前需求、当前迭代、当前版本。

架构思维需要关注时间线。不是说要预测未来三年的需求——那是算命。而是要理解系统的生长方向：这个模块接下来大概率会往哪个方向变？当前的设计在那个方向上是可扩展的，还是会成为阻碍？

一个典型的架构判断是：当前这个表结构设计，如果半年后要支持多类型（比如从只支持实物商品到同时支持虚拟商品），能否通过加字段或加表扩展，还是需要大改？这不是过度设计，而是对数据结构变化成本的预判。

**维度四：评价标准**

代码思维的评价标准是"正确性"和"清晰性"。代码是否正确？逻辑是否清楚？可读性好不好？

架构思维的评价标准还要加上"变化适应性"和"协作友好性"。这个设计在需求变化时改动量大不大？新人加入后能不能快速理解边界和职责？两个团队并行开发时会不会互相冲突？

### 3.2 架构思维不是"想得更多"

有些工程师听到架构思维，以为就是"想得更多"——每做一个需求，都要想七八步，考虑所有可能的变化。这会导致另一个极端：过度设计。

一个三天能写完的功能，花了两周设计了一套插件系统、规则引擎和配置中心，只是因为"将来可能会变"。结果那个"将来"根本没来，或者来的时候方向和预想的完全不同。

架构思维不是想得更多，而是**想的层次不同**。它关注的不是所有可能的变化，而是三件事：

1. **这个能力的归属是否清晰**——改动时能不能找到一个确定的位置？
2. **这个能力的边界是否明确**——它和其他模块之间通过什么方式交互？
3. **这个设计的变化成本是否可控**——最可能的变化方向上，改动会波及多少模块？

这三个问题不需要花很多时间回答。对于一个有经验的工程师，在实现功能的时候多花十分钟想一下，就能避免三个月后花三天去重构。

### 3.3 什么时候需要切换到架构思维

并不是每次写代码都需要架构思维。一个内部脚本、一个临时工具、一个不会有人复用的处理函数，用代码思维搞定就够了。

但以下场景需要切换到架构思维：

**场景一：这个能力会被多个调用方使用。** 比如优惠计算、权限校验、消息发送。一旦有多个调用方，就意味着这个能力会被从不同路径访问，它的接口契约和数据一致性就变得重要。

**场景二：这个模块的变化频率高于它的调用方。** 比如营销规则变化很快，但订单流程相对稳定。如果营销规则的变化每次都需要改订单代码，说明两者的边界有问题。

**场景三：这个数据结构一旦确定就很难改。** 比如核心的订单表、支付表、用户表。这些表结构一旦上线并产生了大量数据，修改的成本（加字段、改类型、数据迁移）远高于修改代码的成本。

**场景四：这个功能涉及多个团队协作。** 一旦跨团队，接口的稳定性、变更的协调成本、数据的一致性约定就不再是一个人能控制的事。

## 4. 常见误区

### 4.1 误区一：架构思维就是先画图再写代码

有些团队推行"先设计后开发"的流程，要求开发前先画架构图、写设计文档。这个流程本身没问题，但如果只是把画图当作架构思维，就容易流于形式。

常见的问题是：图画了，文档写了，但图上的方块和线条和代码结构对不上。文档写了数据流向，但没有讲清楚边界在哪里、取舍是什么、风险点在哪。

架构思维不是一个流程节点，而是一种持续的思考习惯。它可以体现在设计文档里，也可以体现在代码 review 时问的一个问题——"这个逻辑放在这里合适吗？调用方是谁？变化时谁负责改？"

### 4.2 误区二：架构思维是架构师的事

很多工程师觉得"架构是架构师操心的，我只需要写好代码"。这在十人以下的团队里勉强说得通，因为架构师通常也写代码，能把控全局。

但在稍大一点的团队里，架构师不可能 review 每一行代码。日常开发中大量的结构决策——一个新字段加在哪张表里、一个新功能放在哪个 Service 里、一个跨模块的查询是直接调还是走消息——都是写代码的工程师在做的。

如果工程师没有架构意识，这些日常决策就会不断侵蚀系统的结构。等架构师发现问题时，代码已经写了几个月了，重构成本很高。

### 4.3 误区三：架构思维就是考虑扩展性

"要有架构思维"经常被翻译成"要考虑扩展性"，然后变成"到处留扩展点"。于是出现了这种代码：

- 一个接口只有一种实现，但套了一层策略模式；
- 一个配置只有一个值，但做了一套配置中心对接；
- 一个流程只有一条路径，但写了一个流程引擎框架。

过度追求扩展性带来的不是灵活，而是理解成本。新人看到一堆接口和抽象类，不知道哪个是真正的逻辑入口。调试问题时要跳七八层调用才能看到实际执行的代码。

架构思维不是"到处留扩展点"，而是"在最可能变化的地方留余地，在不太会变化的地方保持简单"。识别哪里会变、哪里不会变，才是难点。

### 4.4 误区四：先写对再重构

"先跑通再说，以后有时间重构"——这句话本身不算错。第一版快速出原型是合理的。

问题在于"以后有时间"几乎永远不会来。业务增长期，产品不会给你留时间重构。等业务稳定了，大家又觉得"能跑就行"。等到系统真的出了大问题，重构成本已经大到需要立项了。

更好的做法不是"先不管架构以后重构"，而是在实现功能的过程中，用很小的成本做出合理的结构决策。比如把优惠计算逻辑收拢到一个独立的类里，而不是散落在四个 Service 里——这不需要额外花多少时间，但为后续的变化留下了空间。

### 4.5 误区五：读了架构书就有架构思维

架构思维不是靠读书建立的，而是靠实践中不断反馈建立的。你可以读完 DDD 的书，知道什么是限界上下文、聚合根、领域事件。但如果没有在真实项目中尝试过、碰过壁、调整过，这些概念就只是考试知识。

架构思维的生长路径通常是：做一个决策 → 遇到变化 → 发现之前的决策导致了额外的成本 → 反思 → 下次遇到类似场景时做出更好的判断。这个循环每转一次，判断力就提升一点。

## 5. 设计方法

### 5.1 从"功能实现"到"能力归属"的思考框架

每次实现一个功能时，在写代码之前花五分钟回答这三个问题：

1. **这个能力归属于谁？** 优惠计算归属于营销模块，不是订单模块。密码加密归属于安全模块，不是用户注册接口。归属清晰了，代码放在哪里就清楚了。

2. **谁会使用这个能力？** 如果只有一个调用方，可以简单实现。如果有多个调用方，就需要考虑统一出口和接口契约。

3. **这个能力最可能的变化方向是什么？** 如果是业务规则变化（比如满减条件从 100 改成 200），变化的应该是配置或规则数据，不是代码逻辑。如果是流程变化（比如新增一个审核步骤），变化的应该是流程编排，不是具体的业务方法。

### 5.2 日常开发中的架构检查清单

以下是一份可以在日常开发中使用的检查清单，不需要每次全部检查，但在涉及新模块、新服务、新数据结构时建议过一遍：

**模块归属：**
- 这段逻辑放在当前 Service/模块里合理吗？
- 如果有另一个团队需要同样的能力，他们能找到它吗？
- 这段逻辑变化时，需要改动几个文件/模块？

**接口边界：**
- 这个方法的输入输出是否足够表达意图？
- 调用方是否需要了解内部实现才能正确使用？
- 如果内部实现变了，调用方需要改吗？

**数据结构：**
- 这张表/这个字段的含义是否明确？
- 同一个业务概念在不同地方的定义是否一致？
- 这个数据结构在最可能的扩展方向上，能否低成本扩展？

**变化成本：**
- 如果产品明天改了规则，我需要改几个地方？
- 如果这段逻辑需要被另一个服务使用，我能不能直接提供接口，还是要复制代码？

### 5.3 逐步培养架构感觉的方法

架构思维不能一步到位，但可以通过几个具体的练习逐步培养：

**练习一：读代码时追问"为什么放在这里"。** 看一段已有的代码，不要只看逻辑是否正确，想一想这段逻辑为什么放在这个类里。如果放在另一个模块里会怎样？这个位置选择带来了什么好处和什么限制？

**练习二：做需求时画一张简单的依赖图。** 不需要画完整架构图，只需要画出这次需求涉及的模块和它们之间的调用关系。如果某个模块被三四个其他模块依赖，那它的接口稳定性就很重要。如果两个模块互相调用，那可能存在边界问题。

**练习三：复盘时问"改动范围是否合理"。** 一次需求改完后，看看实际修改了多少个文件、多少个模块。如果一个"加个字段"的需求改了六个服务，说明要么需求本身跨域了（这是正常的），要么系统边界有问题（这需要记录下来）。

**练习四：参加架构评审时关注取舍。** 不只是看方案选了什么，要看方案放弃了什么、为什么放弃。好的架构决策都有代价。理解代价是什么，比知道选了什么更有价值。

## 6. 业务案例

### 6.1 案例：优惠计算能力的归属演进

让我们用一个更完整的例子来说明架构思维是如何在实际项目中起作用的。

**第一阶段：满减功能上线**

业务需求：订单金额满 100 减 10。

代码思维的做法是在 OrderService.createOrder() 方法里加一段判断：

```
if (orderAmount >= 100) {
    discountAmount = 10;
    payAmount = orderAmount - discountAmount;
}
```

这没有问题。需求简单，逻辑简单，没必要搞复杂。

**第二阶段：优惠券叠加**

业务需求：用户可以同时使用满减和优惠券。优惠券有面值限制和使用门槛。

代码思维的做法是继续在 OrderService 里加逻辑：先算满减，再算优惠券，两个优惠金额相加。

架构思维在这里会多想一步：优惠计算这个能力，已经有了满减和优惠券两种形式。接下来很可能还会有折扣、积分抵扣、会员价。这些规则的变化频率远高于订单创建流程本身。如果继续在 OrderService 里堆逻辑，这个 Service 会变成一个优惠规则的大杂烩。

此时合理的做法是把优惠计算提取为一个独立的能力单元，比如 DiscountCalculator 或 PricingService。它接受订单信息、可用优惠列表，输出优惠明细和最终金额。OrderService 只负责调用它，不关心规则细节。

**第三阶段：退款反算**

业务需求：部分退款时，按比例退还优惠金额。

如果优惠计算逻辑散落在 OrderService 里，退款服务要反算优惠金额就必须重新实现一遍计算逻辑，或者直接查订单表里的 discount_amount 字段——但这个字段可能不够细，不知道哪部分是满减、哪部分是优惠券。

如果优惠计算是一个独立的模块，它可以提供两个能力：正向计算（下单时）和反向拆分（退款时）。退款服务通过接口调用，不需要自己拼装逻辑。

**第四阶段：对账和审计**

业务需求：财务需要还原每笔订单的优惠来源和金额。

如果优惠计算有统一出口，这些数据可以在计算时就记录明细。如果逻辑散落在各处，对账时就只能从数据库里拼凑，而且可能拼不全。

### 6.2 从案例中提取架构思维的关键动作

在这个案例中，架构思维体现在几个关键判断上：

1. **识别出"优惠计算"是一个独立能力**，而不是订单创建流程的一部分。这是归属判断。

2. **预判到这个能力会有多个消费方**（下单、退款、对账、客服查询），因此需要统一出口。这是边界判断。

3. **识别出优惠规则的变化频率高于订单流程**，因此两者不应该耦合在一起。这是变化频率判断。

4. **在第二阶段而不是第一阶段做抽取**——第一阶段只有一种规则，抽取没有收益。第二阶段出现了第二种规则，抽取的时机刚好。这是时机判断。

这些判断不需要提前设计好所有可能的变化，只需要在变化出现时，能识别出"现在该调整结构了"的信号。

## 7. 架构判断清单

- 如果一段逻辑在三个以上的地方被复制粘贴，说明它缺少一个统一的能力归属。
- 如果一个 Service 类超过 2000 行，并且覆盖了多个不同的业务概念，说明它缺少边界拆分。
- 如果实现一个需求需要修改五个以上的模块，而这些模块分属不同的业务域，说明要么需求本身是跨域的（正常），要么系统的边界划分不合理（需要改进）。
- 如果一个模块的变化总是导致另一个不相关模块需要跟着改，说明两者之间存在不合理的耦合。
- 如果新加入的工程师需要三天以上才能理解一个模块的职责范围，说明模块的职责定义不够清晰。
- 如果团队中只有一两个人敢改某段代码，说明这段代码的结构已经超出了大多数人的理解范围。
- 如果每次需求评估时花在"影响分析"上的时间超过花在"实现设计"上的时间，说明系统的结构不够可预测。

## 8. 本章小结

代码思维和架构思维不是两种对立的能力，而是工程师成长路径上的两个阶段。代码思维解决"怎么把功能写对"，架构思维解决"怎么让系统在变化中保持可控"。

多数工程师在职业前两三年靠代码思维就能应对大部分工作。但随着系统规模扩大、业务复杂度上升、团队协作面增加，纯靠代码思维就会遇到瓶颈。这个瓶颈不是"写不出来"，而是"改不动、不敢改、改了不知道影响多大"。

架构思维的切换不需要等到升任架构师那一天。每次实现一个功能时，多花五分钟想想"这个能力归属于谁、边界在哪里、变化成本有多大"，就是在练习架构思维。这种思考习惯一旦建立起来，不仅写出的代码会更容易维护，对系统全局的理解也会逐步加深。

最终衡量一个工程师有没有架构思维，不是看他能不能画出漂亮的架构图，而是看他在日常开发中做的结构决策，三个月后是成了阻碍还是成了支撑。
