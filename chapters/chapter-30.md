# 第 30 章：订单系统架构设计

## 1. 从一个真实问题开始

"订单系统嘛，不就是一张订单表加几个接口？"——这是很多工程师第一次接触订单系统时的想法。

一个创业团队的第一版订单系统确实很简单：一张 orders 表，字段有 order_id、user_id、product_id、quantity、amount、status、created_at。下单接口创建一条记录，支付回调更新 status，完成。

一年后，这个"简单"的系统变成了团队里没人愿意碰的模块：

- orders 表有八十多个字段，包括商品信息、收货地址、优惠信息、支付信息、物流信息、发票信息……每次加新功能就加几个字段。
- status 字段有十五个值：CREATED、PENDING_PAY、PAYING、PAID、CONFIRMED、PICKING、PACKED、SHIPPED、DELIVERING、DELIVERED、COMPLETED、CANCELLING、CANCELLED、REFUNDING、REFUNDED。工程师说不清"CONFIRMED 和 PAID 的区别是什么"。
- 下单接口从最初的二十行代码变成了六百行，里面有库存检查、优惠计算、库存锁定、创建支付单、发送消息——全写在一个方法里。
- 退款流程涉及订单状态变更、库存回退、优惠券退回、积分回退、通知用户——在代码中散落在七个地方，有些走了有些没走，具体什么时候走哪个分支，没人说得清楚。

**订单系统看似简单，但它是电商系统中最复杂的模块。** 因为订单是业务流转的核心枢纽——上接用户和商品，下连支付和物流，旁连优惠、库存、积分、通知。

## 2. 问题为什么会变复杂

### 2.1 订单是所有业务的交汇点

用户下单的过程涉及：商品信息、价格、优惠规则、库存、收货地址。支付过程涉及：支付渠道、支付状态、金额校验。履约过程涉及：发货、物流、签收。售后过程涉及：退款、退货、换货。

这些业务不是孤立的，而是通过订单串在一起的。订单的数据模型需要承载所有这些业务的信息，订单的状态需要跟踪整个生命周期。

### 2.2 正常流程简单，异常流程复杂

下单→支付→发货→签收——正常流程就四步。但异常场景有几十种：

- 下单后未支付，超时自动取消。取消后要释放库存、退回优惠券。
- 支付过程中用户关闭了支付页面。订单是"待支付"状态，但支付渠道可能已经扣款了。
- 用户支付成功但支付回调延迟了十分钟。期间用户看到的是"待支付"，实际上钱已经扣了。
- 发货后用户申请退款。已发货的退款需要用户先退货。
- 部分退款——订单有三件商品，退了其中一件。订单状态算什么？
- 退款过程中支付渠道退款失败。需要重试还是人工处理？

每一种异常场景都需要明确的处理逻辑和状态流转规则。

### 2.3 多个维度的状态

订单不只有一个状态维度。实际上至少有三个：

- **交易状态**：待支付、已支付、已取消、已完成。
- **履约状态**：待发货、已发货、已签收。
- **售后状态**：无售后、退款中、已退款。

用一个 status 字段表达三个维度的状态，就会出现十五个值的状态爆炸。

## 3. 核心概念解释

### 3.1 订单的数据模型

一张大宽表放所有信息是最初的做法，但不可持续。合理的做法是把订单信息拆成多张表：

```sql
-- 订单主表：核心交易信息
CREATE TABLE orders (
    order_id VARCHAR(32) PRIMARY KEY,
    user_id BIGINT NOT NULL,
    order_status VARCHAR(20) NOT NULL,     -- 交易状态
    total_amount DECIMAL(10,2) NOT NULL,   -- 订单总额
    discount_amount DECIMAL(10,2) DEFAULT 0, -- 优惠金额
    pay_amount DECIMAL(10,2) NOT NULL,     -- 实付金额
    created_at DATETIME NOT NULL,
    paid_at DATETIME,
    completed_at DATETIME,
    version INT NOT NULL DEFAULT 0,
    INDEX idx_user_status_time (user_id, order_status, created_at)
);

-- 订单商品明细
CREATE TABLE order_items (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id VARCHAR(32) NOT NULL,
    sku_id VARCHAR(32) NOT NULL,
    sku_name VARCHAR(200) NOT NULL,         -- 下单时快照
    unit_price DECIMAL(10,2) NOT NULL,      -- 下单时快照
    quantity INT NOT NULL,
    item_amount DECIMAL(10,2) NOT NULL,
    INDEX idx_order (order_id)
);

-- 订单收货信息
CREATE TABLE order_addresses (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id VARCHAR(32) NOT NULL UNIQUE,
    receiver_name VARCHAR(50),
    receiver_phone VARCHAR(20),
    province VARCHAR(20),
    city VARCHAR(20),
    district VARCHAR(20),
    detail_address VARCHAR(200)
);

-- 订单支付信息
CREATE TABLE order_payments (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id VARCHAR(32) NOT NULL,
    pay_order_id VARCHAR(32) NOT NULL,      -- 支付单号
    channel VARCHAR(20) NOT NULL,            -- 支付渠道
    pay_amount DECIMAL(10,2) NOT NULL,
    pay_status VARCHAR(20) NOT NULL,
    paid_at DATETIME,
    INDEX idx_order (order_id),
    INDEX idx_pay_order (pay_order_id)
);

-- 订单状态变更日志
CREATE TABLE order_status_logs (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id VARCHAR(32) NOT NULL,
    from_status VARCHAR(20),
    to_status VARCHAR(20) NOT NULL,
    operator VARCHAR(50),                    -- 谁触发的变更
    reason VARCHAR(200),
    created_at DATETIME NOT NULL,
    INDEX idx_order (order_id)
);
```

关键设计原则：
- **快照数据下单时就存**：商品名称、价格、收货地址在下单时快照到订单中。用户后来改了地址、商家后来改了价格，不影响已创建的订单。
- **支付信息独立存储**：一个订单可能有多次支付尝试（第一次失败了，换个渠道再试），支付信息用独立的表记录。
- **状态变更有日志**：每次状态变更记录从什么状态到什么状态、谁触发的、什么原因。出了问题可以追溯。

### 3.2 订单状态机

订单状态不应该用 if-else 判断，而应该用状态机管理。状态机明确定义了每个状态可以流转到哪些状态：

```
待支付(PENDING_PAY)
  → 已支付(PAID)         [支付成功回调]
  → 已取消(CANCELLED)    [用户取消 / 超时取消]

已支付(PAID)
  → 待发货(PENDING_SHIP) [商家确认]
  → 退款中(REFUNDING)    [用户申请退款]

待发货(PENDING_SHIP)
  → 已发货(SHIPPED)      [商家发货]
  → 退款中(REFUNDING)    [用户申请退款]

已发货(SHIPPED)
  → 已签收(DELIVERED)    [物流签收]
  → 退货中(RETURNING)    [用户申请退货退款]

已签收(DELIVERED)
  → 已完成(COMPLETED)    [确认收货 / 自动确认]
  → 退货中(RETURNING)    [用户申请退货退款]

退款中(REFUNDING)
  → 已退款(REFUNDED)     [退款成功]
  → 已支付(PAID)         [退款被拒绝]
```

状态机的价值在于：任何不在规则内的状态流转都会被拒绝。"待支付"不能直接跳到"已发货"——这在 if-else 里可能因为漏判而意外发生，在状态机里直接报错。

多维度的状态建议分开管理。交易状态、履约状态、售后状态各用一个字段，不混在一起。

### 3.3 订单号设计

订单号不要用自增 ID。好的订单号设计：

```
时间戳(8位) + 机器标识(2位) + 序列号(4位) + 随机数(2位)

示例：2024011500011234AB
```

设计要点：
- 按时间递增，方便数据库按订单号排序和分页。
- 包含机器标识，分布式环境下不同机器生成的订单号不会冲突。
- 不暴露业务信息——不要让外部通过订单号推算出订单量。
- 长度适中——太短不够用，太长用户记不住。

### 3.4 下单流程的设计

下单是订单系统中最核心的流程。一个完整的下单流程：

```
用户提交订单
  → 参数校验（商品是否存在、是否下架、价格是否变化）
  → 锁定库存（预扣，支付成功后才真正扣减）
  → 计算优惠（匹配优惠券、计算折扣）
  → 冻结优惠券（防止用户同时下多个单用同一张券）
  → 创建订单记录（写入数据库）
  → 创建支付单（调支付服务）
  → 返回支付链接给用户
```

关键的设计决策：

**库存是"锁定"不是"扣减"。** 下单时只是锁定库存，表示"有人打算买但还没付钱"。支付成功后才真正扣减。如果用户不支付，超时后释放锁定的库存。

**优惠券是"冻结"不是"核销"。** 下单时冻结优惠券，支付成功后核销。下单未支付时优惠券回到可用状态。

**下单和支付是两个步骤。** 订单创建后进入"待支付"状态。用户可以选择支付渠道后发起支付。支付是一个异步过程——用户跳转到支付页面，支付完成后通过回调通知订单服务。

### 3.5 超时取消

用户下单后如果不支付，需要在一定时间后自动取消订单，释放库存和优惠券。

实现方式：

**延迟消息**：下单时发一条延迟消息到消息队列，比如延迟 30 分钟。消息到达后检查订单是否还是"待支付"状态。如果是，执行取消操作。

**定时扫描**：定时任务每分钟扫描一次"待支付且创建时间超过 30 分钟"的订单，执行取消。

延迟消息更精确（刚好 30 分钟后触发），定时扫描有时间粒度的误差（最多晚一分钟），但实现更简单。大多数场景用定时扫描就够了。

### 3.6 退款处理

退款流程比支付流程复杂得多：

```
用户申请退款
  → 检查是否可退（已发货的需要退货，已消耗的积分需要回退）
  → 创建退款单
  → 调支付服务发起退款
  → 支付服务调渠道退款
  → 退款成功回调
  → 更新订单状态
  → 回退库存
  → 解冻/退回优惠券
  → 回退积分
  → 通知用户
```

退款的挑战在于：
- 退款可能失败（渠道拒绝、余额不足），需要重试和人工介入机制。
- 部分退款的计算——退一件商品，优惠券的分摊金额怎么算？
- 退款和正向流程的冲突——用户申请退款的同时商家在发货，两个操作并发怎么处理？

## 4. 常见误区

### 4.1 一张大宽表

把所有信息塞在一张表里。八十个字段的 orders 表每次查询都 `SELECT *`，大量不需要的数据被读取出来。

应该把订单拆成主表 + 明细表 + 扩展表。查订单列表只需要主表的几个核心字段，查订单详情再关联明细表。

### 4.2 用一个 status 表达所有状态

十五个状态值混在一个字段里，工程师搞不清楚"CONFIRMED"到底代表什么。

应该把交易状态、履约状态、售后状态分开。交易状态只有四五个值（待支付、已支付、已完成、已取消），清晰明确。

### 4.3 不做数据快照

商品价格在订单创建后修改了，导致订单金额和商品页面显示的价格不一致。用户投诉："我买的时候明明是 99 元，怎么变成 109 了？"

订单中的商品信息、价格信息、收货地址必须在下单时快照保存，不依赖其他服务的实时数据。

### 4.4 支付回调不做幂等

支付渠道的回调可能重复到达。如果不做幂等处理，同一笔支付可能被处理两次——库存多扣一次、积分多发一次。

支付回调必须做幂等：检查订单状态，如果已经是"已支付"就直接返回成功，不重复处理。

### 4.5 退款逻辑散落各处

退库存的代码在库存服务、退优惠券的代码在优惠券服务、退积分的代码在积分服务——退款时分别调用这些服务。但没有一个统一的地方管理"退款需要做哪些事"。

应该有一个退款编排逻辑——退款单状态机管理退款流程，按步骤执行各个回退操作，每步都记录执行结果，失败时可以从断点重试。

## 5. 设计方法

### 5.1 订单系统的模块划分

```
订单模块
├── 订单创建：参数校验、调用库存和优惠券、创建订单记录
├── 订单查询：列表查询、详情查询、按条件搜索
├── 订单状态机：管理所有状态流转规则
├── 支付集成：创建支付单、接收支付回调
├── 超时处理：未支付订单的超时取消
└── 退款处理：退款申请、退款单管理、回退操作编排
```

订单模块不包含库存管理、优惠计算、支付渠道对接——这些是独立的模块或服务。订单模块是"调用方"和"编排者"。

### 5.2 订单与外部系统的边界

| 外部系统 | 订单调用时机 | 交互方式 | 失败处理 |
|---------|------------|---------|---------|
| 库存 | 下单时锁定，支付后扣减，取消时释放 | 同步调用 | 锁定失败则下单失败 |
| 优惠券 | 下单时冻结，支付后核销，取消时解冻 | 同步调用 | 冻结失败则下单失败 |
| 支付 | 下单后创建支付单，回调通知支付结果 | 异步回调 | 支付超时则取消订单 |
| 积分 | 支付后发积分，退款后回退积分 | 异步事件 | 失败重试，不阻塞主流程 |
| 通知 | 各环节发短信/推送 | 异步事件 | 失败不影响主流程 |

核心操作（库存、优惠券）用同步调用——失败时下单就不能成功。非核心操作（积分、通知）用异步事件——失败不影响订单主流程。

## 6. 业务案例

### 6.1 案例：订单系统从"一张表"到"结构化设计"的演进

**第一阶段：一张表打天下**

五个人的团队，业务很简单——用户买商品、支付、商家发货。一张 orders 表加二十个字段就够了。

这个阶段的选择是对的。业务简单时，简单的实现最高效。

**第二阶段：字段爆炸，开始拆表**

半年后，新增了优惠券、积分、多商品订单、发票。orders 表字段从二十个涨到六十个。查询开始变慢，因为每次都 `SELECT *` 读出了大量不需要的数据。

改造：拆分为 orders 主表 + order_items 明细表 + order_addresses 收货表 + order_payments 支付表。查列表只读主表，查详情再 join 其他表。

**第三阶段：状态混乱，引入状态机**

订单状态从 5 个涨到 15 个。出现了多次"不该出现的状态流转"——已取消的订单又被标为已发货（因为取消操作和发货操作并发执行，if-else 没有做互斥判断）。

改造：引入状态机，明确定义每个状态可以流转到哪些状态。并发操作通过乐观锁（version 字段）保护——状态变更时检查 version，不一致则重试。

**第四阶段：退款地狱**

退款成为最复杂的流程。一个退款操作要回退库存、退回优惠券、回退积分、调支付渠道退钱、通知用户。任何一步失败都需要处理。

改造：引入退款单（refund_orders 表），退款有自己的状态机。退款操作按步骤编排，每步执行后更新退款单状态。失败的步骤可以从断点重试，不需要从头开始。

**结果**：经过四个阶段的演进，订单系统从一张表变成了结构化的模块。每个阶段的改造都是在确实遇到问题后才做的，没有过早设计。

## 7. 架构判断清单

- 如果订单表超过四十个字段，需要考虑拆表。
- 如果订单状态超过十个值，需要考虑多维度状态分离。
- 如果下单接口超过三百行代码，需要拆分为多个步骤和模块。
- 如果退款流程没有独立的状态管理，一旦中间环节失败就很难恢复。
- 如果订单中引用了其他服务的实时数据（而不是快照），数据一致性迟早出问题。
- 如果支付回调没有做幂等处理，重复回调会导致数据异常。
- 如果订单系统的异常处理代码比正常业务代码还多，说明异常场景的建模需要改进（用状态机代替 if-else）。

## 8. 本章小结

订单系统的复杂性不在于"创建一条订单记录"，而在于：多维度的状态管理、与多个外部系统的协调、大量的异常场景处理、数据快照和一致性保障。

设计订单系统的关键决策包括：数据模型的拆分（主表+明细+扩展）、状态机的设计（多维度分离）、下单流程的编排（区分核心和非核心操作）、退款流程的独立管理（退款单+步骤编排）、与外部系统的交互方式（同步vs异步）。

最重要的经验是：不要在第一天就设计一个"完美"的订单系统。从简单开始，在业务增长的过程中逐步演进——数据快照、状态机、表拆分、退款编排，每一步都是在真正遇到问题后才引入的。
