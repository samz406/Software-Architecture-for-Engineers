# 第 5 章：边界能力：好的架构从划清职责开始

## 1. 从一个真实问题开始

一个电商系统，订单模块要做一个需求：用户取消订单时，如果已经用了优惠券，要把优惠券退回去。

工程师打开 OrderService，在 cancelOrder 方法里加了一段逻辑：调用 CouponService 的 returnCoupon 方法。写完测试通过了。

一周后，产品说取消订单还要释放库存。工程师又在 cancelOrder 里加了调用 InventoryService 的 releaseStock 方法。

再后来，取消订单要退积分、退会员成长值、发取消通知、更新数据报表。cancelOrder 方法变成了一个"调度中心"，直接依赖了六七个其他模块的 Service。

问题开始暴露：
- CouponService 改了 returnCoupon 的参数，OrderService 跟着改。InventoryService 改了接口签名，OrderService 又跟着改。订单组的工程师发现自己一半时间在跟着别人改代码。
- 有一次 CouponService 的接口超时了，导致整个取消订单流程失败。用户取消不了订单，投诉到了客服。
- 测试 cancelOrder 的时候需要 mock 六个外部依赖，单测写起来比业务代码还长。

技术负责人说："订单和优惠券之间的边界不清楚。"

但什么叫"边界清楚"？边界应该画在哪？怎么画？

## 2. 问题为什么会变复杂

### 2.1 边界不是画出来的，是长出来的问题

系统刚建的时候，所有代码在一个项目里，Service 之间互相调用是最自然的做法。项目小的时候没问题——三五个人在同一个代码库里，谁的代码做了什么一清二楚。

但团队和代码量增长之后，问题就出现了。OrderService 依赖了 CouponService，CouponService 依赖了 UserService，UserService 又依赖了 OrderService（查用户的最近订单）。依赖关系变成了网状，改任何一个模块都可能影响其他模块。

更严重的是数据层面的边界缺失。订单模块直接读优惠券的表来判断券状态，优惠券模块直接读订单表来统计核销数据。两个模块共享数据库表，谁也说不清"这个字段归谁管"。

### 2.2 四种边界问题

**模块边界不清**：代码层面，Service 之间互相调用，形成网状依赖。一个模块的内部改动影响其他模块。

**数据边界不清**：多个模块直接读写同一张表，或者一张表承载了多个模块的职责。改表结构时不知道会影响谁。

**服务边界不清**：拆了微服务但拆得不对。两个服务之间频繁同步调用，一个请求链路跨七八个服务，延迟和故障风险叠加。

**团队边界不清**：代码归属不明确。一个文件三个团队都在改，合并冲突频繁。出了 bug 互相推诿。

这四种边界问题通常是关联的。模块边界不清导致拆服务时拆错，拆错导致服务间调用频繁，服务间调用频繁导致联调和故障排查困难。

### 2.3 为什么边界容易被忽视

边界不清的系统在短期内往往跑得更快。不需要定义接口、不需要考虑数据归属、不需要做异步解耦——直接调用、直接读表、直接改字段，开发速度最快。

但这种"快"是以牺牲后期可维护性为代价的。当系统小的时候，网状依赖还能在脑子里理清楚。当系统大了、团队多了，网状依赖就变成了泥潭。

## 3. 核心概念解释

### 3.1 什么是好的边界

一个好的边界让模块之间的关系满足三个条件：

**可描述**：能用一句话说清楚一个模块的职责。如果说不清楚，或者需要加很多"除了……还有……"的补充，说明职责不够内聚。

**可替换**：一个模块的内部实现可以整体替换（比如换技术方案、重构内部代码），而不影响其他模块。这要求模块之间通过稳定的接口交互，而不是依赖对方的内部细节。

**可独立变化**：两个模块可以各自按照自己的节奏迭代。订单模块发新版本不需要优惠券模块配合上线。这要求模块之间的依赖是松散的。

### 3.2 模块边界

模块边界是最基础的边界，在代码层面通过包结构和访问控制来实现。

一个模块应该有：
- **明确的对外接口**：其他模块只能通过这些接口访问本模块的能力。
- **隐藏的内部实现**：内部的类、方法、数据结构不对外暴露。
- **独立的数据管理**：模块管理自己的数据，其他模块不直接读写。

在 Java 中，可以用包访问权限来实现。在模块化程度更高的项目中，可以用 Maven 模块或 Gradle 模块来强制隔离。

```
com.example.order/
    api/           -- 对外接口（其他模块可调用）
        OrderFacade.java
        OrderDTO.java
    domain/        -- 内部领域对象（包访问权限）
        Order.java
        OrderItem.java
    service/       -- 内部服务（包访问权限）
        OrderService.java
    repository/    -- 内部数据访问（包访问权限）
        OrderRepository.java
```

OrderFacade 是订单模块的对外接口。其他模块要查订单信息，调用 OrderFacade.getOrderInfo()，而不是直接注入 OrderService 或者直接读 order 表。

### 3.3 数据边界

数据边界是最容易被忽视但影响最深的边界。

**原则：每张表有且只有一个"属主模块"。** 属主模块负责这张表的读写和维护。其他模块如果需要这张表的数据，通过属主模块的接口获取，而不是直接查表。

为什么这么严格？因为表结构的变化会影响所有直接读写它的代码。如果只有属主模块直接操作这张表，改表结构时只需要改属主模块的代码。如果有五个模块都直接读这张表，改表结构就要改五个地方。

在单体应用中，数据边界可以通过编码规范来维护（"优惠券的表只有 CouponFacade 可以读写"）。在微服务中，数据边界通过数据库隔离来物理保证（每个服务有自己的数据库）。

### 3.4 服务边界

当模块拆分为独立服务之后，边界变成了网络边界。服务之间通过 API 或消息通信。

服务边界的关键问题是**拆分粒度**。拆太细：服务间调用太频繁，一个请求跨十几个服务，延迟叠加、故障率叠加。拆太粗：两个不相关的业务被绑在一起，改一个要部署整个服务。

一个实用的判断标准是：**如果两个模块之间每个请求都需要同步调用对方，它们可能不应该拆成两个服务。** 同步调用意味着强耦合——A 服务必须等 B 服务响应才能继续。这种强耦合跨网络后，可靠性和延迟都会变差。

## 4. 常见误区

### 4.1 "先写代码再划边界"

很多团队在开发初期不考虑模块边界，所有逻辑混在一起写。等系统大了再"重构划边界"。

问题是：代码越写越多，依赖越积越深，后期划边界的成本越来越高。OrderService 已经直接调用了 CouponService 的十几个内部方法，现在要改成只通过 CouponFacade 交互——这意味着十几个调用点都要改，还要保证功能不变。

更好的做法是：**在代码不多的时候就建立模块的对外接口层，哪怕现在只有一个调用方。** 成本很低（多写一个 Facade 类），但为将来保留了清晰的边界。

### 4.2 "拆服务就是划边界"

有些团队在模块边界还没理清的时候就急着拆微服务。结果拆完之后发现：两个服务之间每个请求都要互相调用好几次，因为它们的数据和逻辑本来就是耦合的。

**拆服务是物理隔离手段，不是划边界的方法。** 边界应该先在代码层面划清楚（模块化），然后在需要的时候再把模块拆成服务。如果模块之间的边界在代码层面就是混乱的，拆成服务只会让混乱变成分布式的混乱。

### 4.3 边界画得太细

另一个极端是边界画得太细。每个功能都独立一个模块，模块之间通过接口交互。结果是：写一个简单的功能要跨四五个模块，开发效率很低。

边界是为了管理复杂性，不是为了制造复杂性。**如果两个功能总是一起变化、由同一个人维护、数据高度关联，它们应该放在一个模块里。** 强行拆开只会增加不必要的间接层。

### 4.4 只划了代码边界没划数据边界

模块之间用 Facade 接口交互了，但底层还是读同一个数据库。这种"代码分了但数据没分"的状态很脆弱——有人偷偷写了一个直接查表的 SQL，边界就被绕过了。

数据边界不一定要物理分库。在单体应用中，通过编码规范和代码审查来维护数据归属也是可行的。但要有明确的规则并且严格执行。

### 4.5 忽视异步边界

很多边界问题可以通过异步解耦来解决。订单取消后要退优惠券、退积分、释放库存——如果这些都是同步调用，订单模块就强依赖了所有下游。

改为异步方式：订单取消后发出"OrderCancelled"事件，优惠券模块、积分模块、库存模块各自监听这个事件并处理。订单模块不需要知道有哪些下游，下游的处理是否成功也不影响取消操作本身。

异步解耦不是万能的——它引入了最终一致性的问题。但对于"取消订单后退券退积分"这类场景，最终一致性是完全可接受的。

## 5. 设计方法

### 5.1 如何判断边界应该画在哪

**问自己三个问题：**

**这个模块的职责能用一句话说清吗？**
- "订单模块负责订单的创建、查询、状态管理"——清楚。
- "订单模块负责订单的创建、查询、状态管理、优惠券核销、库存锁定、支付发起、消息通知"——不清楚。后面那些职责应该属于各自的模块。

**这两个模块的变化原因一样吗？**
- 优惠券规则变了 → 改优惠券模块。订单流程变了 → 改订单模块。变化原因不同，应该分开。
- 订单项变了 → 改订单模块。订单项的金额计算变了 → 也改订单模块。变化原因一致，放在一起。

**这两个模块之间的交互是什么模式？**
- 每次订单操作都要同步调用优惠券 → 耦合太紧，考虑异步解耦。
- 只在特定场景下需要对方的数据（比如退款时查原始优惠券信息）→ 通过接口查询即可。

### 5.2 划边界的具体步骤

**第一步：列出系统中所有的核心业务对象。**

订单、订单项、支付单、退款单、优惠券、库存、用户、商品、消息通知……

**第二步：按归属关系分组。**

哪些对象总是一起操作？哪些对象属于同一个业务领域？

- 订单组：订单、订单项
- 支付组：支付单、退款单
- 营销组：优惠券、促销活动
- 商品组：商品、SKU、库存

**第三步：明确每个组的对外接口。**

订单组对外提供什么能力？创建订单、查询订单、取消订单。这些是 OrderFacade 的方法。

**第四步：识别组之间的交互方式。**

订单创建时需要锁库存 → 同步调用库存模块的"锁定"接口。
订单支付成功后需要扣库存 → 异步事件，库存模块监听"支付成功"事件。
退款时需要退优惠券 → 异步事件，优惠券模块监听"退款成功"事件。

**第五步：用依赖关系图验证。**

画出模块之间的依赖方向。检查有没有循环依赖。如果 A 依赖 B，B 也依赖 A，说明边界不对——要么合并，要么通过事件解耦。

### 5.3 边界保护机制

边界划好之后，需要机制来保护它不被破坏：

- **代码层面**：用编译器强制隔离（独立模块、包访问权限）。如果做不到物理隔离，用 ArchUnit 等工具做架构规则检查。
- **数据层面**：明确每张表的属主模块。代码审查时检查是否有跨模块直接读写表的行为。
- **团队层面**：每个模块有明确的负责团队。其他团队需要本模块的能力，通过接口提需求，不自己改本模块的代码。

## 6. 业务案例

### 6.1 案例：优惠券字段影响三个模块的故事

优惠券表加了一个字段 max_discount_amount（封顶优惠金额）。本来只有营销团队需要关心这个字段。

但实际影响到了三个模块：
- **结算模块**：计算订单应付金额时要考虑优惠封顶。结算逻辑直接读了 coupon 表来获取 max_discount_amount。
- **退款模块**：退款时要按比例退还优惠金额，需要知道原始的优惠封顶值。退款代码也直接读了 coupon 表。
- **财务模块**：统计实际优惠支出时，需要知道每张券的封顶金额。财务报表的 SQL 直接 join 了 coupon 表。

三个模块都直接依赖了 coupon 表的具体字段。当营销团队把 max_discount_amount 从整数改为小数（支持更精确的金额）时，三个模块的代码都要改。

修复方案：
1. 结算模块不再直接读 coupon 表，而是在下单时通过 CouponFacade 获取优惠计算结果，结果中包含封顶金额。
2. 退款模块从订单快照中读取下单时的优惠信息，不再实时读 coupon 表。
3. 财务模块通过数据同步机制获取优惠数据（coupon 模块定时推送），不直接查 coupon 表。

改完之后，coupon 表的字段变化只影响营销模块自己。

### 6.2 案例：从同步调用到事件驱动的边界重构

订单取消流程原来是这样的：

```
cancelOrder() {
    order.setStatus(CANCELLED);
    couponService.returnCoupon(order.getCouponId());
    inventoryService.releaseStock(order.getItems());
    pointService.deductPoints(order.getUserId(), order.getPointsUsed());
    notificationService.sendCancelNotice(order.getUserId());
    reportService.recordCancellation(order);
}
```

六个同步调用，任何一个失败都导致整个取消失败。优惠券服务超时了，用户取消不了订单。

重构后：

```
cancelOrder() {
    order.setStatus(CANCELLED);
    orderRepository.save(order);
    eventPublisher.publish(new OrderCancelledEvent(order));
}
```

订单模块只负责修改订单状态和发出事件。优惠券、库存、积分、通知、报表各自监听 OrderCancelledEvent 并处理。任何一个下游的处理延迟或失败，都不影响取消操作本身。

## 7. 架构判断清单

- 如果一个需求的改动涉及三个以上的模块，说明模块之间的边界可能不对。正常情况下，一个业务需求的改动应该集中在一两个模块内。
- 如果一个模块的对外接口超过 20 个方法，说明职责可能太杂，需要进一步拆分。
- 如果两个模块之间有循环依赖（A 依赖 B，B 也依赖 A），说明边界划分不对，需要用事件解耦或提取公共依赖。
- 如果改一张表的字段需要通知三个以上的团队，说明数据边界不清。
- 如果一个服务的每个请求都需要同步调用另一个服务，说明这两个服务可能不应该拆开，或者需要改为异步交互。
- 如果测试一个模块需要 mock 超过五个外部依赖，说明该模块的依赖太多，边界需要收紧。

## 8. 本章小结

边界能力是架构设计中最基础也最重要的能力。好的边界让每个模块职责清晰、独立变化、故障隔离。坏的边界让系统变成一锅粥——改任何地方都牵一发动全身。

划边界的核心思路是：把变化原因相同的东西放在一起，把变化原因不同的东西隔开。模块内部高内聚，模块之间低耦合。数据有明确归属，交互通过稳定接口。

边界不是一次画好就不变的。随着业务发展和团队变化，边界需要调整。但调整的前提是：先有边界。一个有边界但边界不完美的系统，远好于一个完全没有边界的系统。
