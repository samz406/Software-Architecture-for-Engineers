# 第 13 章：并发与一致性：分布式系统的现实世界

## 1. 从一个真实问题开始

大促前一天，团队做了最后一轮压测。订单系统 QPS 到 5000 的时候，监控大盘出现异常：库存出现负数。

一款商品库存是 100 件，但卖出了 103 件。

工程师查了代码，扣库存的逻辑很直接：

```java
public void deductStock(Long skuId, int quantity) {
    Stock stock = stockMapper.selectById(skuId);
    if (stock.getAvailable() >= quantity) {
        stock.setAvailable(stock.getAvailable() - quantity);
        stockMapper.updateById(stock);
    } else {
        throw new BusinessException("库存不足");
    }
}
```

逻辑看起来没问题：先查库存够不够，够就扣。但在并发场景下，这段代码有经典的竞态条件。两个请求同时读到库存 = 1，都判断"够"，然后各自扣减，库存变成了 -1。

这不是一个罕见的 bug。它是并发编程中最基础的问题——**查检和操作不是原子的**。在单线程下没问题，在并发下就出问题。

## 2. 问题为什么会变复杂

### 2.1 并发不只是"多线程"

很多工程师把并发理解为"多线程编程"。但在分布式系统中，并发的来源不只是线程：

- 多个用户同时操作同一个资源（抢同一个商品的库存）。
- 同一个请求被重复投递（消息队列重试、支付回调重试）。
- 定时任务和在线请求同时操作同一份数据。
- 多个服务实例同时处理同一笔业务。

在单机上，你可以用 synchronized 或互斥锁来控制并发。在分布式环境下，代码跑在多个进程、多台机器上，单机的锁没用了。

### 2.2 一致性有多种级别

并发带来的核心问题是数据一致性。但一致性不是"要么完全一致、要么不一致"这么简单。

**强一致性**：任何时刻读到的数据都是最新写入的结果。代价最高，通常需要分布式事务或全局锁。

**最终一致性**：写入后经过一段时间（通常是毫秒到秒级），所有读取方都能读到最新值。中间可能有短暂的不一致。

**因果一致性**：如果操作 A 发生在操作 B 之前，那么任何读取方看到 B 的结果时也一定能看到 A 的结果。

不同的业务场景需要不同级别的一致性。扣库存需要强一致——不能超卖。查询订单列表可以接受最终一致——主库写入后从库有几秒延迟是可以的。发送积分可以接受最终一致——积分晚几秒到账用户感知不到。

### 2.3 分布式事务是个昂贵的选择

在单库中，一个数据库事务就能保证多个操作的一致性。但当数据分布在多个库、多个服务中时，单库事务就不够用了。

分布式事务（如两阶段提交 2PC）能保证强一致，但代价很高：
- 性能差：需要多次网络通信和协调。
- 可用性低：协调者或参与者任何一个不可用，整个事务就卡住了。
- 实现复杂：需要所有参与方都支持分布式事务协议。

所以在实际项目中，分布式事务很少使用。大部分场景用的是"本地事务 + 消息队列 + 补偿"的最终一致性方案。

## 3. 核心概念解释

### 3.1 乐观锁

乐观锁不是真正的锁，而是在更新数据时检查是否有其他人在你之前修改了这条数据。

实现方式是给表加一个 version 字段：

```sql
-- 查询时获取当前版本号
SELECT id, available, version FROM stock WHERE sku_id = 123;
-- 假设查到 available=10, version=5

-- 更新时带上版本号条件
UPDATE stock SET available = available - 1, version = version + 1 
WHERE sku_id = 123 AND version = 5;
-- 如果别人已经改了，version 不再是 5，这条 UPDATE 影响行数为 0
```

如果 UPDATE 的影响行数为 0，说明有并发修改，本次操作失败，调用方可以重试。

乐观锁适合**冲突频率低**的场景。如果大部分请求不会冲突（不同用户操作不同的数据），乐观锁效率很高。如果冲突频率高（秒杀场景，大量请求抢同一个商品），乐观锁会导致大量重试，效率反而很低。

### 3.2 悲观锁和分布式锁

悲观锁的思路是：操作数据之前先加锁，加锁成功才能操作，操作完再释放锁。

在单库中，可以用数据库的 `SELECT ... FOR UPDATE` 实现悲观锁。在分布式环境中，需要用分布式锁。

**Redis 分布式锁**是最常用的方案：

```java
// 加锁
boolean locked = redis.set("lock:stock:123", requestId, "NX", "EX", 5);
// NX: 只在 key 不存在时设置（防止重复加锁）
// EX 5: 5 秒后自动过期（防止死锁）

if (locked) {
    try {
        // 执行业务逻辑
        deductStock(123, 1);
    } finally {
        // 释放锁（需要检查是不是自己加的锁）
        if (requestId.equals(redis.get("lock:stock:123"))) {
            redis.del("lock:stock:123");
        }
    }
}
```

分布式锁的关键注意点：
- **必须设置过期时间**：防止加锁后进程崩溃导致锁永远无法释放。
- **释放时检查是否是自己的锁**：用唯一的 requestId 标识锁的持有者。
- **过期时间要大于业务执行时间**：否则业务还没执行完锁就过期了，其他请求拿到锁，出现并发问题。

### 3.3 库存扣减的正确做法

回到开头的库存超卖问题，正确的做法：

**方案一：数据库原子操作（推荐大多数场景）**

```sql
UPDATE stock SET available = available - 1 
WHERE sku_id = 123 AND available >= 1;
```

把"检查"和"扣减"合并为一条 SQL，利用数据库的行锁保证原子性。如果影响行数为 0，说明库存不足。

**方案二：Redis 原子操作（高并发场景）**

```lua
-- Redis Lua 脚本，原子执行
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
else
    return 0
end
```

在 Redis 中预扣库存，然后异步同步到数据库。适合秒杀等高并发场景。

### 3.4 幂等

幂等意味着同一个操作执行一次和执行多次的效果相同。在分布式系统中，幂等至关重要，因为：

- 网络超时后，调用方不知道服务端是否已经处理成功，最安全的做法是重试。
- 消息队列可能因为消费者 ACK 失败而重复投递消息。
- 支付渠道可能重复发送支付回调。

实现幂等的常见方案：

**业务唯一键**：在表中对业务唯一字段（如 order_no）建唯一索引。重复插入时数据库会报唯一键冲突。

**请求去重表**：专门建一张表记录已处理的请求 ID。处理前先查是否已处理。

**状态机约束**：利用状态流转的单向性保证幂等。订单状态从"待支付"到"已支付"，如果当前已经是"已支付"，直接返回成功。

### 3.5 最终一致性方案

在跨服务的场景中，用"本地事务 + 消息"实现最终一致性：

**步骤一**：在本地事务中完成核心操作 + 写消息记录。

```sql
BEGIN;
-- 核心业务操作
UPDATE orders SET status = 'PAID' WHERE id = 123;
-- 写消息记录（和业务操作在同一个事务中）
INSERT INTO outbox_messages (topic, payload, status) 
VALUES ('order-paid', '{"orderId":123}', 'PENDING');
COMMIT;
```

**步骤二**：定时任务或 CDC 机制读取消息记录，发送到消息队列。

**步骤三**：下游服务消费消息，执行自己的操作。如果失败，消息会重新投递。

**步骤四**：定时对账。比对订单状态和库存状态是否一致，发现差异后告警人工处理。

这种方式保证了"如果本地事务成功了，消息一定会被发出去"——因为消息记录和业务操作在同一个本地事务中。

## 4. 常见误区

### 4.1 "加个锁就行"

遇到并发问题就加锁，是最常见的反应。但锁是有代价的：

- 锁降低了并发度。100 个请求串行等锁，吞吐量急剧下降。
- 分布式锁有可靠性风险。Redis 宕机了，锁就失效了。
- 锁的粒度不对会导致死锁。A 锁了资源 1 等资源 2，B 锁了资源 2 等资源 1。

在很多场景中，乐观锁或数据库原子操作比分布式锁更合适。只有在确实需要互斥访问共享资源时才用分布式锁。

### 4.2 "所有操作都要强一致"

不是所有操作都需要强一致。支付扣款需要强一致——不能多扣也不能少扣。但发送积分可以最终一致——积分晚几秒到账没有问题。发送通知可以更弱——通知丢失了也不会造成资金损失。

把所有操作都要求强一致，会导致系统的性能和可用性急剧下降。**按业务影响分级，核心操作用强一致，非核心操作用最终一致。**

### 4.3 忽视补偿机制

最终一致性方案中，消息消费可能失败。如果只依赖消息重试，有些场景重试也无法解决（比如下游服务 bug 导致消费一直失败）。

需要有**对账机制**作为最后的保底。定期比对上下游的数据，发现不一致时触发补偿操作或人工介入。对账是最终一致性方案的必备组成部分。

### 4.4 忽视超时问题

调用外部服务没有设超时，或者超时设得太长。结果是：外部服务卡了，本地的线程一直在等待，线程池被耗尽，整个服务不可用。

每个外部调用都必须设超时。超时后的处理策略也要明确：是重试？是降级？是返回错误？

### 4.5 "先扣款后发货" vs "先发货后扣款"

跨服务操作时，操作顺序的选择影响失败后的处理复杂度。

**先扣款后发货**：如果发货失败，需要退款。退款是一个明确的操作，可以执行。
**先发货后扣款**：如果扣款失败，需要"追回已发出的货物"。这个操作可能无法执行。

原则是：**先执行容易补偿的操作，后执行不可逆的操作。** 退款比追回货物容易得多。

## 5. 设计方法

### 5.1 并发问题的分析框架

遇到并发问题时，按以下步骤分析：

1. **识别共享资源**：哪些数据会被多个请求同时访问和修改？（库存、账户余额、优惠券状态）
2. **识别并发场景**：什么情况下会出现并发？（高峰期抢购、消息重复消费、定时任务和在线请求冲突）
3. **确定一致性要求**：这个操作需要强一致还是最终一致？
4. **选择控制手段**：数据库原子操作？乐观锁？分布式锁？消息队列？
5. **设计补偿方案**：如果一致性被破坏了，怎么发现？怎么修复？

### 5.2 常见场景的推荐方案

| 场景 | 推荐方案 |
|------|---------|
| 库存扣减（普通） | 数据库原子 UPDATE |
| 库存扣减（秒杀） | Redis 预扣 + 异步落库 |
| 支付回调幂等 | 状态机 + 唯一索引 |
| 订单创建防重 | 请求唯一键 + 唯一索引 |
| 跨服务数据一致 | 本地事务 + 消息 + 对账 |
| 余额扣减 | 数据库行锁 + 流水记录 |

## 6. 业务案例

### 6.1 案例：支付回调的幂等处理

支付渠道在用户支付成功后回调订单服务。但回调可能重复（网络抖动、渠道重试）。如果不做幂等，同一笔支付可能触发两次订单状态更新、两次库存扣减、两次积分发放。

处理方案：

```java
public void handlePaymentCallback(PaymentCallback callback) {
    // 1. 查询支付单
    Payment payment = paymentRepository.findByPaymentNo(callback.getPaymentNo());
    
    // 2. 幂等检查：如果已经处理过，直接返回
    if (payment.getStatus() == PayStatus.SUCCESS) {
        return;  // 幂等返回
    }
    
    // 3. 状态机校验：只有 PAYING 状态才能变为 SUCCESS
    if (payment.getStatus() != PayStatus.PAYING) {
        log.warn("支付单状态不正确: {}", payment.getStatus());
        return;
    }
    
    // 4. 更新支付单状态（乐观锁防止并发）
    int affected = paymentRepository.updateStatus(
        payment.getId(), PayStatus.PAYING, PayStatus.SUCCESS, 
        callback.getChannelTransactionId());
    
    if (affected == 0) {
        // 并发更新，另一个线程已经处理了
        return;
    }
    
    // 5. 发事件通知下游
    eventPublisher.publish(new PaymentSuccessEvent(payment.getOrderId()));
}
```

三层防护：状态检查 → 状态机校验 → 乐观锁更新。任何一层都能挡住重复请求。

### 6.2 案例：下单流程的一致性设计

下单涉及多个操作：创建订单、锁库存、冻结优惠券。这三个操作需要保持一致——要么都成功，要么都不成功。

**如果在同一个数据库中**：用数据库事务包裹，简单有效。

**如果跨服务**：

```
1. 创建订单（状态=待确认） -- 本地事务
2. 调库存服务锁库存
   - 成功 → 继续
   - 失败 → 取消订单，结束
3. 调优惠券服务冻结优惠券
   - 成功 → 确认订单（状态=待支付）
   - 失败 → 释放库存，取消订单，结束
```

如果步骤 3 失败了，需要补偿步骤 2（释放库存）。如果补偿也失败了（库存服务不可用），需要记录到补偿日志，定时任务重试补偿。

最后还有对账：每天比对"订单已取消但库存未释放"和"订单已取消但优惠券未退回"的数据，作为最终保底。

## 7. 架构判断清单

- 如果扣减类操作（库存、余额）用了"先查后改"的两步逻辑而没有原子保护，说明在并发下一定会出问题。
- 如果支付回调的处理没有幂等机制，说明重复回调会导致数据错误。
- 如果跨服务操作只考虑了成功路径没有考虑失败补偿，说明一致性方案不完整。
- 如果系统中没有对账机制，说明数据不一致时无法被发现。
- 如果所有操作都用了分布式锁，需要评估是否过度使用——很多场景用乐观锁或数据库原子操作就够了。
- 如果外部调用没有设超时，说明存在线程池耗尽的风险。
- 如果消息消费逻辑不是幂等的，说明消息重复消费会导致数据错误。

## 8. 本章小结

并发与一致性是分布式系统中最难处理也最容易出错的部分。核心挑战在于：多个操作同时访问和修改共享数据时，如何保证结果的正确性。

处理并发问题的关键不是记住一堆锁和事务的用法，而是**分清楚每个场景需要什么级别的一致性，然后选择成本最低的方案**。能用数据库原子操作解决的就不用分布式锁，能用最终一致的就不用强一致。

对工程师来说，写代码时要养成一个习惯：每当写"先查后改"的逻辑时，问自己"如果有两个请求同时执行这段代码会怎样？"这一个问题就能避免大部分的并发 bug。
