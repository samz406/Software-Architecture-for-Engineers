# 第 27 章：技术债：什么时候该还，什么时候先忍

## 1. 从一个真实问题开始

一个电商系统的优惠券模块，最初只支持"满减券"——满 100 减 10。代码很简单：

```java
if (orderAmount >= coupon.getThreshold()) {
    discount = coupon.getDiscountAmount();
}
```

后来产品加了"折扣券"——打八折。工程师在原来的代码旁边加了一个 if：

```java
if ("FULL_REDUCE".equals(coupon.getType())) {
    discount = coupon.getDiscountAmount();
} else if ("DISCOUNT".equals(coupon.getType())) {
    discount = orderAmount * (1 - coupon.getDiscountRate());
}
```

再后来加了"随机金额券""阶梯满减券""品类限定券""新人专享券"。每种券的计算逻辑都用 if-else 加在后面。两年后这段代码变成了三百多行的 if-else 链，有十二个分支，嵌套最深的地方有四层。

团队里每次改优惠券逻辑都提心吊胆。上个月一个工程师改了"阶梯满减"的计算，不小心影响了"品类限定"的判断条件，导致线上一批订单少收了钱。

大家都知道这段代码需要重构——用策略模式把不同券类型的计算逻辑拆开。但每次提出来，得到的回复都是："下季度吧，现在大促优先。""先把需求做完，有空再说。"

三年了，从来没有"有空"的时候。

**这就是技术债——大家都知道有问题，但一直没有修。** 问题在于：怎么判断什么时候必须还、什么时候可以先忍？怎么在业务压力下找到还债的时机？

## 2. 问题为什么会变复杂

### 2.1 技术债是怎么产生的

技术债不一定是"写了烂代码"。很多技术债在产生的时候是合理的选择。

创业初期只有满减券，写一个简单的 if 判断完全正确。这不是烂代码，而是"在当时的业务复杂度下最合理的实现"。

技术债产生的根本原因是**业务变化超出了原有设计的承载范围**。原来的设计是为一种优惠类型准备的，现在要支持十二种。这种"设计和现实的差距"就是技术债。

其他常见的技术债来源：

- **赶工期的妥协**：为了按时上线，跳过了抽象设计，直接硬编码。
- **知识不足**：当时不知道更好的做法，写了能用但不优雅的实现。
- **演进中的遗留**：系统经过多次修改，原来合理的结构被改得不合理了。
- **依赖过时**：使用的框架或中间件版本太老，但升级需要大量改造。

### 2.2 技术债的"利息"

技术债像真正的债务一样会产生利息——随着时间推移，不还债的成本在增加。

那段三百行的 if-else 链：
- 每次改优惠券逻辑需要多花两到三天理解和测试（对比重构后可能只需要半天）。
- 每次改动都有引入 bug 的风险。
- 新人接手这段代码需要额外一周的学习成本。

这些"利息"每个月都在支付。如果这段代码一个月被修改两次，每次多花两天，一年下来就是四十八天的额外成本——远超重构需要的两周。

### 2.3 技术债不会自己消失

有些团队希望技术债能"等到有时间再处理"。但在大多数团队中，永远不会有"专门还债的时间"——业务需求总是源源不断地来。

技术债如果不主动处理，只会越积越多。直到某天变成了系统中"没人敢动的代码"——一动就出问题，一出问题就要花更多时间修。

## 3. 核心概念解释

### 3.1 技术债的分类

不是所有的技术债都值得立即处理。技术债可以按影响程度和修复成本分类：

**高影响、低成本**：优先处理。比如一个缺失的索引导致某个查询每天超时十几次——加个索引就解决了。

**高影响、高成本**：规划处理。比如订单表需要分库分表——影响大但改造成本也大，需要专门排期。

**低影响、低成本**：顺便处理。比如一个方法命名不规范——不需要专门排期，下次改到这个文件时顺便改了。

**低影响、高成本**：先不处理。比如一个内部工具的界面不好用——难用但不影响业务，改造成本又大，可以先忍。

### 3.2 如何识别技术债

技术债不总是显而易见的。以下几个信号说明可能存在技术债：

**频繁出现相似的 bug**："又是优惠券计算错了"——同一个模块反复出问题，通常是因为代码结构有问题。

**改动成本异常高**："加一种优惠券类型为什么要两周？"——改动成本和功能复杂度不成比例。

**只有特定的人能改**："这段代码只有老王能动"——如果一段代码只有一个人敢改，说明代码的可维护性有问题。

**修改一处影响多处**："我只改了优惠计算，为什么退款也出问题了？"——说明模块之间的耦合太紧。

**重复代码到处散落**：同一段逻辑在三个地方各写了一遍，改了一处忘了改另外两处。

### 3.3 如何量化技术债

说服管理层投入资源还债，需要把技术债翻译成管理层能理解的语言。

**时间成本**："因为优惠券模块的代码结构问题，每次修改平均多花两天。过去半年改了六次，多花了十二天。重构需要十天，三个月就能回本。"

**故障成本**："过去三个月优惠券模块导致了四次线上故障，总影响用户约八千人次，客服处理了三百多个投诉。重构后预计故障减少 70%。"

**机会成本**："因为优惠券模块改起来太慢，产品的'限时秒杀券'需求排了两个月还没做。竞品已经上线了这个功能。"

### 3.4 什么时候该还

**必须立即还的**：
- 技术债已经导致了线上故障且有重复发生的风险。
- 技术债阻塞了关键业务需求的交付。
- 技术债导致的安全漏洞（比如明文存储密码）。

**应该计划还的**：
- 技术债的"利息"已经超过了还债的成本（上面的量化方法可以判断）。
- 相关模块即将有大的业务需求改动，趁这个机会一起重构。
- 系统即将面临流量或数据量的大幅增长（比如大促前发现某些慢查询没有索引）。

**可以先忍的**：
- 技术债所在的模块很少被修改（半年没有任何需求涉及它）。
- 修复成本远高于当前的影响。
- 系统即将下线或被替换。

### 3.5 怎么还

还技术债不是"停下业务做重构"。在大多数团队中，这不现实。更可行的方式：

**边交付边治理**："童子军规则"——每次修改一个文件时，让它比修改前更好一点。这次改优惠券模块的需求，顺便把涉及到的那两个 if-else 分支抽成策略类。不需要一次全改，但每次改一点。

**搭便车**：如果产品需求恰好涉及技术债所在的模块，把重构和需求一起做。"这个需求反正要改优惠券计算逻辑，我们顺便把策略模式的框架搭起来。新增的券类型直接用新框架，老的逐步迁移。"

**专项还债**：对于高影响的技术债，确实需要专门排期。但要控制范围——不要一次把所有技术债都还了。每个迭代安排一到两个高优先级的还债任务。

## 4. 常见误区

### 4.1 "所有技术债都要还"

不是所有技术债都值得还。如果一段代码很少被修改、不影响系统稳定性、没有安全风险，即使代码不够优雅也没必要专门重构。

"代码不好看"不等于"技术债需要还"。技术债是否需要处理，取决于它造成的实际影响，不取决于代码是否符合你的审美。

### 4.2 "重构就是还债"

有些工程师把任何代码修改都叫"还技术债"。想用新框架替换旧框架、想把代码风格改成自己喜欢的方式、想把单体拆成微服务——这些不一定是在"还债"，可能只是技术偏好。

真正的还债应该有明确的收益：减少 bug、降低改动成本、提升性能、消除安全风险。

### 4.3 "提了三年没人批"

如果重构提案三年都没通过，通常不是管理层不重视技术，而是提案的方式有问题。

常见的问题：
- 只说"代码需要重构"，没有量化影响和收益。
- 要求一次性的大量资源投入，让管理层觉得风险太大。
- 没有分阶段的方案，全有或全无。

更好的方式：量化技术债的成本 → 给出分阶段的改造方案 → 第一阶段成本小、风险低、收益明确 → 用第一阶段的成功争取后续阶段的资源。

### 4.4 "技术债是管理的错"

把技术债全归咎于"工期太紧""管理层不重视技术"是不公平的。工期紧的时候做了技术妥协，这是在已知约束下的合理决策。关键不是不产生技术债，而是有意识地记录、跟踪、计划处理。

有些技术债是工程师自己造成的——不了解更好的方案、懒得设计、复制粘贴。这类技术债的源头在于工程师的能力和态度，不是管理的问题。

### 4.5 一次还清

试图一次性解决所有技术债——"花两个月做一次大重构"。这种做法风险很大：
- 改动范围太大，测试覆盖不全，容易引入新 bug。
- 两个月内不交付业务需求，业务团队不接受。
- 如果重构中途遇到问题需要回滚，两个月的工作全部白费。

分步处理、持续改进比一次性大改更安全也更可行。

## 5. 设计方法

### 5.1 技术债登记和管理

建立一个技术债清单，记录已知的技术债：

| 编号 | 描述 | 影响 | 修复成本 | 优先级 | 状态 |
|-----|------|-----|---------|-------|------|
| TD-001 | 优惠券计算 if-else 链 | 每次改动多2天，半年4次故障 | 10天 | 高 | 计划中 |
| TD-002 | 订单表缺少联合索引 | P95 超过2秒 | 1小时 | 高 | 已解决 |
| TD-003 | 旧版支付SDK未升级 | 不支持新渠道 | 5天 | 中 | 待排期 |
| TD-004 | 内部管理后台UI陈旧 | 操作不便但不影响功能 | 15天 | 低 | 暂不处理 |

定期（每月或每季度）review 这个清单，更新优先级和状态。

### 5.2 优先级评估

用一个简单的公式评估技术债的处理优先级：

**优先级 = 影响频率 × 单次影响成本 / 修复成本**

- 影响频率：每月影响几次
- 单次影响成本：每次影响多少天人力、多少次故障、多少用户受影响
- 修复成本：需要多少天的工作量

优先级最高的先处理。

### 5.3 向管理层汇报技术债

定期（每季度）向管理层汇报技术债的状况：

```markdown
## 技术债季度报告

### 当前状况
- 高优先级技术债 3 项，中优先级 5 项
- 上季度因技术债导致的故障：4 次
- 上季度因技术债产生的额外开发成本：约 20 人天

### 上季度处理情况
- 已解决 2 项高优先级债务
- 效果：相关模块故障减少 60%，修改效率提升 40%

### 本季度计划
- 计划处理 TD-001（优惠券模块重构），预计 10 天
- 计划处理 TD-003（支付SDK升级），预计 5 天
```

用数据说话，让管理层看到技术债的成本和还债的收益。

## 6. 业务案例

### 6.1 案例：优惠券模块的技术债处理

**现状分析**：优惠券计算逻辑是一段三百行的 if-else 链，支持十二种券类型。

量化影响：
- 过去六个月因这段代码导致的线上故障：4 次
- 每次修改平均额外花费时间：2 天（对比正常的 0.5 天）
- 过去六个月共修改 6 次，额外花费：12 天
- 新人理解这段代码的平均时间：3 天

修复方案：用策略模式重构。每种券类型一个策略类，统一的 CouponCalculator 接口。

```java
public interface CouponCalculator {
    CouponType supportType();
    Money calculate(Order order, Coupon coupon);
}

// 满减券
public class FullReduceCalculator implements CouponCalculator {
    public CouponType supportType() { return CouponType.FULL_REDUCE; }
    public Money calculate(Order order, Coupon coupon) {
        if (order.getAmount().gte(coupon.getThreshold())) {
            return coupon.getDiscountAmount();
        }
        return Money.ZERO;
    }
}
```

预计修复成本：10 天。

**分步执行**：

不是一次性把十二种券类型全部迁移。

第一步（搭便车）：产品需求要新增"限时折扣券"。在做这个需求时，搭建策略模式的框架，把新增的"限时折扣券"和最简单的"满减券"用新框架实现。老的十一种券类型暂时保持 if-else，通过一个 Adapter 兼容。耗时：3 天（其中 2 天是需求本身的工作量）。

第二步（专项还债）：在下一个迭代中安排 3 天，把使用频率最高的四种券类型迁移到新框架。

第三步（持续治理）：之后每次有涉及优惠券的需求，顺便迁移一两种老的券类型。两个月后全部迁移完成。

**效果**：重构完成后，新增一种券类型从平均 4 天降到 1 天。六个月内优惠券相关的线上故障从 4 次降到 0 次。

## 7. 架构判断清单

- 如果一个模块每次修改都提心吊胆，说明这里有需要处理的技术债。
- 如果同一类 bug 反复出现，根因很可能是技术债而不是工程师粗心。
- 如果添加一个简单功能需要改动五个以上的文件，说明模块之间的耦合可能是技术债。
- 如果重构提案连续两个季度没有排期，尝试换一种方式——更小的范围、更明确的量化、搭业务需求的便车。
- 如果团队花在修 bug 和排查问题上的时间超过了写新功能的时间，说明技术债已经到了必须处理的程度。
- 如果技术债清单超过二十项且半年没有更新，说明清单只是"写了"但没有"用"。

## 8. 本章小结

技术债不是坏事——它是在约束下做出合理妥协的自然结果。问题不在于产生技术债，而在于不记录、不跟踪、不计划处理。

管理技术债的关键是：能识别哪些是技术债、能量化技术债的影响、能判断什么时候该还什么时候先忍、能找到务实的还债方式。

最实用的还债策略不是"停下来大重构"，而是"边做业务边治理"——搭便车、童子军规则、每个迭代处理一两个高优先级的债务。这样既不影响业务交付节奏，又能让系统逐步改善。
