# 第 16 章：安全架构：不要等出事才重视

## 1. 从一个真实问题开始

某天下午，运营同事在群里发了一张截图：有人在论坛上卖他们平台的用户数据，包括手机号、收货地址和最近的订单记录。价格不贵，打包一万条卖几百块。

安全团队介入后发现，数据泄露的源头是一个内部管理后台的导出接口。这个接口本来是给客服用的，可以按条件导出用户信息用于售后处理。但这个接口有几个问题：

- 没有导出数量限制，一次可以导出全量数据；
- 没有操作审计日志，不知道谁在什么时候导出了多少数据；
- 接口鉴权只做了登录校验，没有做权限校验——只要登录了管理后台，任何角色都可以调用这个导出接口；
- 导出的数据没有脱敏，手机号、身份证号、收货地址全部明文。

最后查出来是一个外包客服的账号被盗了。攻击者用这个账号调用导出接口，批量拉取了用户数据。

这个事故的根源不是某段代码写错了，而是安全在这个系统的设计中根本不是一个被认真对待的关注点。认证做了，但授权没做细；接口有了，但访问控制没有分级；数据能查，但敏感字段没有脱敏。每个环节都是"能用就行"的水平，合在一起就是一个漏洞百出的系统。

## 2. 问题为什么会变复杂

### 2.1 安全不是一个功能模块

很多工程师把安全当作一个功能来实现：做一个登录页面、做一个权限表、做一个加密方法。但安全不是一个功能模块，而是一组贯穿系统各层的约束。

认证是约束——确保请求来自合法用户。
授权是约束——确保用户只能访问自己有权限的资源。
加密是约束——确保数据在传输和存储过程中不被泄露。
审计是约束——确保所有敏感操作都有记录可查。
防刷是约束——确保接口不被恶意高频调用。

这些约束不像业务功能那样有明确的需求方。没有产品经理会提需求说"请给导出接口加上权限校验"。安全约束的需求方是系统本身——如果不做，系统在正常使用时看不出问题，但在被攻击或被滥用时就会暴露。

### 2.2 安全问题的特点：低频高危

安全问题和性能问题不同。性能问题是持续的——系统每天都在响应请求，慢了用户马上能感知。安全问题是间歇的——系统可能运行了一年都没被攻击，但一旦被攻击，损失可能无法挽回。

这种"低频高危"的特点导致安全在日常开发中很容易被忽略。赶工期时优先砍的就是安全相关的需求："先上线再说，权限下个版本补""脱敏以后再做""审计日志先不记了"。每次砍一点，风险就积累一点。

### 2.3 安全边界随系统复杂度扩大

一个简单的单体应用，安全边界比较容易管理。但随着系统演进为微服务、开放了第三方接口、接入了外部系统，安全边界就变得复杂了：

- 服务之间的调用需要认证吗？还是默认内网可信？
- 第三方调用的接口和内部接口用同一套鉴权吗？
- 用户数据在服务之间流转时，谁负责脱敏？
- 日志里会不会把用户的手机号、密码打出来？

每增加一个对外接口、每接入一个外部系统，安全攻击面就扩大一点。

## 3. 核心概念解释

### 3.1 认证（Authentication）

认证回答的问题是：**你是谁？**

常见的认证方式：
- **Session + Cookie**：服务端生成 Session，通过 Cookie 传递 Session ID。适合传统 Web 应用。
- **Token（JWT）**：服务端签发 Token，客户端每次请求携带。适合前后端分离和移动端。
- **OAuth 2.0**：第三方授权协议，适合需要接入微信登录、支付宝登录等场景。
- **API Key + Secret**：适合服务间调用和第三方开放接口。

选择认证方式时需要考虑的因素：
- 客户端类型（浏览器、App、服务端）；
- 是否需要跨域；
- Token 的过期和刷新策略；
- 是否需要支持单点登录（SSO）。

### 3.2 授权（Authorization）

授权回答的问题是：**你能做什么？**

常见的权限模型：

**RBAC（基于角色的访问控制）**：用户 → 角色 → 权限。最常用的模型。比如：客服角色可以查看订单、修改收货地址，但不能导出用户数据。

**ABAC（基于属性的访问控制）**：根据用户属性、资源属性、环境属性来判断权限。比如：只有订单所属的销售人员才能查看该订单详情（资源属性过滤）。

**数据权限**：不只控制"能不能做某个操作"，还控制"能看到哪些数据"。比如：区域经理只能看到自己区域的订单。

很多系统只做了功能权限（菜单级别、按钮级别），没有做数据权限。导致一个客服能看到所有用户的数据，而不只是自己负责的。

### 3.3 数据安全

数据安全涉及三个阶段：

**传输安全**：数据在网络上传输时不被窃听。HTTPS 是基础，敏感接口还需要加签验签。

**存储安全**：数据在数据库中不以明文存储。密码必须用 bcrypt 等不可逆算法加密。身份证号、银行卡号等敏感信息用可逆加密存储（需要查询或展示时解密）。

**使用安全**：数据在使用过程中不被泄露。日志中不打印敏感字段。前端展示手机号时中间四位打星号。导出的数据需要脱敏。

### 3.4 接口安全

对外接口面临的安全威胁包括：

**重放攻击**：攻击者截获一个合法请求，原样重复发送。防御方法：请求中带时间戳 + 随机数 + 签名，服务端校验时间戳在有效期内且随机数未被使用过。

**参数篡改**：攻击者修改请求参数。防御方法：对请求参数做签名，服务端验签。

**越权访问**：用户 A 通过修改请求中的用户 ID，访问用户 B 的数据。防御方法：接口中不信任客户端传的用户 ID，从 Token 中提取当前用户身份。

**频率滥用**：恶意高频调用接口。防御方法：限流（按用户、按 IP、按接口分别限流）。

## 4. 常见误区

### 4.1 "内网是安全的"

很多团队对内部服务之间的调用不做认证，理由是"都在内网，没有外部攻击"。

但内网不安全的原因不只是外部攻击：
- 内部员工可能恶意操作；
- 某个服务被入侵后可以横向移动；
- 测试环境和生产环境的网络隔离可能不严格；
- 一个有 bug 的服务可能误调用其他服务的敏感接口。

至少应该做到：服务间调用有身份标识（比如通过 API Key 或 mTLS），敏感操作有审计日志。

### 4.2 "加密了就安全了"

加密是安全的一个手段，但不是全部。密码用了 bcrypt 加密，但如果登录接口没有限流，攻击者可以暴力尝试。数据库加密了，但如果管理后台的导出接口没有权限控制，数据还是可以被批量拉取。

安全是一个系统工程，认证、授权、加密、脱敏、限流、审计每个环节都要做，缺一个都会成为短板。

### 4.3 "安全是安全团队的事"

安全团队可以做渗透测试、安全扫描、制定安全规范。但大部分安全问题是在日常开发中引入的：

- 工程师在日志里打印了用户手机号；
- 接口没有校验当前用户是否有权访问请求的资源；
- SQL 拼接没有用参数化查询，存在注入风险；
- 前端把 API Key 写在了 JavaScript 代码里。

这些问题只有写代码的人才能在第一时间避免。安全意识需要内化到日常开发习惯中。

### 4.4 权限设计过于粗放

很多系统的权限只做到菜单级别：管理员能看到所有菜单，普通用户只能看到部分菜单。但接口层面没有做权限校验——只要知道接口地址，绕过前端直接调用也能成功。

权限校验必须在后端做。前端隐藏菜单和按钮只是用户体验的优化，不是安全保障。

### 4.5 忽视操作审计

很多系统不记录敏感操作的日志：谁在什么时候查看了哪个用户的信息、谁导出了多少数据、谁修改了哪个订单的状态。

等出了安全事故才发现无从追溯。审计日志不是为了抓人，而是为了事后能还原发生了什么、影响了多少数据、应该怎么修复。

## 5. 设计方法

### 5.1 安全设计的分层思路

安全设计可以分四层来考虑：

**第一层：接入层安全**
- HTTPS 全站加密；
- 请求限流（按 IP、按用户）；
- 防 SQL 注入、XSS 攻击（参数校验和转义）；
- 防重放攻击（时间戳 + 签名）。

**第二层：认证授权层**
- 统一的认证网关（所有请求经过认证后才进入业务服务）；
- 基于角色的功能权限控制；
- 基于属性的数据权限控制；
- Token 管理（签发、刷新、吊销）。

**第三层：数据安全层**
- 密码不可逆加密（bcrypt/scrypt）；
- 敏感字段可逆加密（AES + 密钥管理）；
- 日志脱敏（手机号、身份证号、银行卡号在日志中打星）；
- 数据导出脱敏和数量限制。

**第四层：审计层**
- 登录日志（谁在什么时间从哪个 IP 登录）；
- 操作日志（谁执行了什么敏感操作）；
- 数据访问日志（谁查看了哪些敏感数据）；
- 异常行为告警（短时间内大量登录失败、批量导出数据）。

### 5.2 权限模型设计实践

对于大多数业务系统，RBAC（角色-权限模型）是最实用的选择。

**数据模型：**

```sql
-- 用户表
CREATE TABLE sys_user (id, username, password_hash, status, ...);

-- 角色表
CREATE TABLE sys_role (id, role_name, role_code, description, ...);

-- 权限表
CREATE TABLE sys_permission (id, permission_code, permission_name, resource_type, ...);

-- 用户-角色关联
CREATE TABLE sys_user_role (user_id, role_id);

-- 角色-权限关联
CREATE TABLE sys_role_permission (role_id, permission_id);
```

**接口权限校验：**

```java
@PreAuthorize("hasPermission('order:export')")
@GetMapping("/orders/export")
public void exportOrders(...) {
    // 只有拥有 order:export 权限的用户才能调用
}
```

**数据权限校验：**

接口级权限控制"能不能做这个操作"，数据权限控制"能看到哪些数据"。

```java
public List<Order> queryOrders(QueryParam param) {
    // 从当前用户身份获取数据范围
    DataScope scope = dataPermissionService.getScope(currentUser);
    // 自动在查询条件中加上数据范围限制
    param.setRegionIds(scope.getRegionIds());
    return orderRepository.query(param);
}
```

### 5.3 接口签名设计

对外接口（特别是支付回调、第三方调用）需要做签名验证：

**签名流程：**

1. 将所有请求参数按字母序排列，拼接成字符串；
2. 在字符串末尾拼接密钥（Secret）；
3. 对拼接结果做 HMAC-SHA256 哈希，得到签名；
4. 将签名放在请求头或参数中发送；
5. 服务端用同样的方法计算签名，对比是否一致。

**防重放：**

在签名参数中加入 timestamp 和 nonce（随机数）。服务端校验：
- timestamp 和当前时间差不超过 5 分钟（防止旧请求重放）；
- nonce 在 Redis 中不存在（防止同一请求重放），校验通过后将 nonce 写入 Redis 并设置 5 分钟过期。

### 5.4 敏感数据脱敏策略

**日志脱敏**：在日志框架中配置脱敏规则，对手机号、身份证号、银行卡号等字段自动打星。

```
原始数据：13812345678
脱敏后：138****5678
```

**存储脱敏**：密码用 bcrypt 加密，敏感字段（如身份证号）用 AES 加密。加密密钥通过密钥管理服务（KMS）管理，不硬编码在代码里。

**展示脱敏**：前端展示敏感数据时根据用户权限决定是否脱敏。高权限用户（如风控人员）可以查看完整信息，普通客服只能看到脱敏后的数据。

**导出脱敏**：数据导出接口必须做脱敏处理，并且限制单次导出数量。导出操作必须记录审计日志。

## 6. 业务案例

### 6.1 案例：支付系统的安全设计

支付系统是安全要求最高的业务系统之一。以一个典型的支付流程为例，看看安全措施是如何贯穿其中的。

**下单阶段：**
- 用户身份认证：通过 Token 验证用户身份；
- 订单创建接口做幂等校验：防止重复下单；
- 订单金额由服务端计算：不信任客户端传的金额。

**支付阶段：**
- 调用支付渠道前做签名：防止请求被篡改；
- 支付渠道的密钥通过 KMS 管理：不存在代码库中；
- 支付请求有超时控制：防止长时间挂起。

**回调阶段：**
- 验证回调来源的签名：确认确实是支付渠道发来的；
- 校验回调中的订单号和金额：防止伪造回调；
- 幂等处理：同一笔支付的回调只处理一次；
- 回调处理结果有审计日志。

**退款阶段：**
- 退款操作需要特定权限；
- 退款金额不能超过原始支付金额；
- 大额退款需要二次审批；
- 所有退款操作记录审计日志。

**对账阶段：**
- 每天对比平台的支付记录和渠道的交易记录；
- 差异数据自动标记并通知财务人员；
- 对账数据只有特定角色可以查看。

### 6.2 案例：用户数据保护

一个会员系统存储了大量用户个人信息：姓名、手机号、身份证号、收货地址。如何在满足业务需要的同时保护这些数据？

**存储层：**
- 手机号、身份证号在数据库中加密存储；
- 密码使用 bcrypt 加密，不可逆；
- 加密密钥通过 KMS 管理，定期轮换。

**查询层：**
- 用户查询自己的信息：返回完整数据（用户看自己的数据是合理的）；
- 客服查询用户信息：手机号中间四位打星，身份证号只显示前三后四；
- 批量导出：所有敏感字段脱敏，单次最多导出 1000 条，需要 order:export 权限；
- 所有查看用户敏感信息的操作记录访问日志。

**接口层：**
- 查询用户信息的接口做数据权限校验：普通客服只能查询自己负责的用户；
- 接口限流：单个账号每分钟最多查询 30 次用户信息；
- 异常行为检测：如果一个账号短时间内查询了大量不同用户的信息，自动触发告警。

## 7. 架构判断清单

- 如果系统的接口权限校验只在前端做（隐藏按钮/菜单），没有在后端做，说明权限体系形同虚设。
- 如果密码使用 MD5 加密或明文存储，说明存储安全不达标。
- 如果日志中可以找到用户的手机号、身份证号等明文信息，说明日志脱敏没有做。
- 如果一个导出接口没有数量限制、没有权限校验、没有审计日志，说明这是一个高风险的数据泄露入口。
- 如果服务间调用没有任何身份认证机制，说明内网信任模型过于宽松。
- 如果对外接口没有签名验证和防重放机制，说明接口容易被伪造和重放攻击。
- 如果敏感操作（删除数据、修改金额、导出信息）没有审计日志，说明事后无法追溯和定责。
- 如果加密密钥硬编码在代码或配置文件中，说明密钥管理不安全。

## 8. 本章小结

安全不是开发完成后再补的一层壳，而是从系统设计之初就应该融入的一组约束。认证确保"你是谁"，授权确保"你能做什么"，加密确保"数据不泄露"，审计确保"出了事能追溯"。

在实际项目中，安全往往是最容易被压缩工期的部分。因为它不直接产生业务价值——在系统没有被攻击之前，有没有做好安全看不出区别。但一旦出了安全事故，损失往往远超开发安全功能的成本。用户数据泄露可能导致法律风险，支付漏洞可能导致直接经济损失，接口被刷可能导致服务不可用。

对工程师来说，安全意识不需要变成安全专家的水平。但几个基本的习惯应该内化到日常开发中：不在日志里打印敏感数据、不信任客户端传入的用户身份、接口权限在后端校验、密码用不可逆加密、对外接口做签名和防重放。这些习惯的成本很低，但能避免大部分常见的安全问题。
