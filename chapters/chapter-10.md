# 第 10 章：接口设计：系统之间如何说话

## 1. 从一个真实问题开始

两个团队在联调一个支付回调接口。支付服务在用户支付成功后，会回调订单服务的接口来更新订单状态。

联调一周，进展缓慢。双方的对话大致是这样的：

订单团队："你的回调传的是什么格式？文档上写的和实际传的不一样。"
支付团队："文档可能没更新。你看下我们测试环境传的请求就知道了。"
订单团队："你传的 orderId 是你们内部的支付单号还是我们的订单号？"
支付团队："是支付单号，你们用支付单号反查一下就行。"
订单团队："但你们创建支付单的时候没返回支付单号给我们，我们怎么反查？"
支付团队："那你传 orderId 给我，我在回调的时候带上。"
订单团队："回调失败的时候你们会重试吗？"
支付团队："会重试三次。"
订单团队："那我们收到重复回调怎么办？"
支付团队："你们自己做幂等吧。"
订单团队："你们回调超时的判定是多少？如果我们处理慢会不会被你们当作失败重试？"
支付团队："……这个没定义。"

一个接口的联调暴露了一连串问题：字段语义不明确、文档和实际不一致、缺少幂等约定、缺少超时约定、错误处理没有规范。

这些问题不是"开发能力不行"导致的。两个团队的工程师都很资深。问题出在**接口没有被当作一个需要仔细设计的契约来对待**。大家觉得"接口嘛，对着调就行"，结果就是联调时一堆意外。

## 2. 问题为什么会变复杂

### 2.1 接口是系统之间的契约

两个人面对面沟通，有问题可以当场问。两个系统之间的"沟通"靠的是接口。接口定义了：我接受什么输入、返回什么输出、在什么条件下返回什么错误、调用方需要遵守什么约定。

接口设计不好的后果不是"难看"，而是"出错"。字段含义模糊导致调用方理解错误，缺少幂等约定导致数据重复，缺少版本管理导致升级时调用方全部报错。

### 2.2 接口一旦发布就很难改

内部代码改了就改了，编译不通过的地方全部修一遍就行。但接口一旦发布给外部调用方，改动就变得困难：

- 删除一个字段——调用方的代码可能在用这个字段；
- 修改一个字段的类型——调用方的反序列化会失败；
- 修改接口的语义——调用方按旧语义处理会出 bug。

接口的修改需要考虑向后兼容。这意味着接口设计时的每一个决定——字段名、类型、是否必填、错误码的含义——都需要比内部代码更谨慎。

### 2.3 坏接口的传播效应

一个设计不好的接口，会把它的问题传播给所有调用方。如果支付回调接口的错误码定义混乱（有时返回字符串"success"，有时返回数字 0，有时返回 null 表示成功），所有调用方都要写一堆兼容逻辑来处理这些不一致。

更糟糕的是，调用方可能会把这个接口的坏设计"转发"给它的下游——因为它要把支付状态传给其他模块，而支付接口的混乱格式会渗透到整个调用链中。

## 3. 核心概念解释

### 3.1 接口设计的基本要素

一个完整的接口定义应该包含：

**请求参数**：每个参数的名称、类型、是否必填、取值范围、业务含义。
**返回结果**：成功时返回什么、不同失败场景返回什么错误码。
**幂等性**：同一个请求重复调用是否会产生副作用。
**超时约定**：调用方应该设置多少超时时间。
**频率限制**：每秒/每分钟最多调用多少次。
**版本信息**：当前接口版本，与上一版本的差异。

很多团队只定义了前两项，后面的全靠口头约定或者联调时再讨论。

### 3.2 接口命名和语义

接口应该用**业务动作**命名，而不是用技术操作命名。

好的命名：
- `POST /orders` — 创建订单
- `POST /orders/{id}/cancel` — 取消订单
- `POST /payments/{id}/refund` — 发起退款

不好的命名：
- `POST /api/execute?action=createOrder` — 把所有操作塞在一个接口里
- `POST /orders/update` — "update"太模糊，是更新什么？收货地址？状态？备注？
- `GET /data?type=order&id=123` — 把接口当成数据库查询

接口的语义应该对调用方是自明的。看到接口名就知道它做什么，看到参数名就知道每个参数是什么意思。不需要翻文档才能理解。

### 3.3 错误码设计

错误码是接口设计中最容易被忽视但影响最大的部分。

**反模式：只有两种状态。** 成功返回 `code=0`，失败返回 `code=-1`，错误信息全靠 message 字段的文本内容。调用方不知道是参数错了、权限不够、数据不存在还是服务内部错误。

**反模式：错误码太细。** 给每种业务场景定义一个错误码，一个模块有上百个错误码。调用方根本记不住。

**好的做法**是分层设计错误码：

```
// 第一层：HTTP 状态码表示大类
200 — 成功
400 — 请求参数错误（调用方的问题）
401 — 未认证
403 — 无权限
404 — 资源不存在
409 — 业务冲突（比如重复支付）
500 — 服务内部错误（服务方的问题）

// 第二层：业务错误码表示具体原因
{
    "code": "ORDER_NOT_FOUND",
    "message": "订单不存在",
    "detail": "orderId=12345 未找到对应订单记录"
}
```

错误码的关键设计原则：
- 调用方应该能根据错误码决定"接下来怎么办"——是重试、是提示用户、还是报告开发者。
- 不要用数字做错误码（1001、2003），用有意义的字符串（ORDER_NOT_FOUND、INSUFFICIENT_BALANCE），可读性好得多。
- 区分"调用方可以重试"的错误和"不可重试"的错误。参数错误不可重试，服务超时可以重试。

### 3.4 幂等设计

幂等的定义：同一个请求调用一次和调用多次，效果相同。

为什么需要幂等？因为在分布式环境中，调用方不一定知道请求是否成功。网络超时了，服务端可能已经处理成功了，也可能没有。调用方最安全的做法是重试——但如果接口不是幂等的，重试可能导致重复创建、重复扣款。

实现幂等的常用方法：

**请求带唯一标识**：调用方在每个请求中带一个唯一的 requestId。服务端收到请求后先查 requestId 是否已处理过，处理过就直接返回上一次的结果。

```java
public PayResult pay(PayRequest request) {
    // 先查是否已处理
    PayResult existing = payResultCache.get(request.getRequestId());
    if (existing != null) {
        return existing;  // 幂等返回
    }
    // 未处理，执行支付逻辑
    PayResult result = doPayment(request);
    payResultCache.put(request.getRequestId(), result);
    return result;
}
```

**状态机约束**：利用业务状态来天然保证幂等。比如"支付成功回调"的处理逻辑是把订单状态从"待支付"改为"已支付"。如果重复回调，第二次发现订单已经是"已支付"状态，就直接返回成功。

### 3.5 接口版本管理

接口会演进。新版接口可能加了字段、改了语义、调整了返回格式。但旧版调用方可能还没升级。

版本管理的常见方式：
- URL 路径：`/v1/orders`、`/v2/orders`
- 请求头：`API-Version: 2`
- 参数：`?version=2`

版本管理的原则：
- 新版本上线后，旧版本至少保留一到两个迭代周期，给调用方升级时间。
- 只在不兼容变更时升大版本。兼容性变更（新增可选字段）不需要升版本。
- 记录每个版本的变更日志，调用方能知道升级后会有什么变化。

## 4. 常见误区

### 4.1 接口变成"远程数据库查询"

有些接口的设计像是把数据库的查询能力直接暴露了出来：

```
GET /orders?fields=id,status,amount&filter=status=paid&sort=created_at:desc&page=1&size=20
```

看起来很灵活，但问题是：调用方需要了解你的数据结构和字段名才能使用。接口变成了一个远程 SQL 客户端。

更好的做法是：按业务场景定义接口。"查询待发货订单列表"和"查询退款中的订单列表"是两个不同的业务场景，可以定义为两个接口，各自返回该场景需要的字段。

### 4.2 "一个接口搞定所有事"

有些系统用一个通用接口处理所有操作：

```
POST /api/execute
{
    "action": "createOrder",
    "params": { ... }
}
```

这种"万能接口"的问题是：所有操作的参数混在一起，没有类型安全，没有独立的文档，错误码混杂，权限无法按操作粒度控制。

### 4.3 返回过多内部细节

接口返回中包含了调用方不需要的内部信息：内部 ID、数据库表名、堆栈信息、内部状态码。

这不仅是信息泄露的安全风险，还增加了调用方的理解成本。返回值应该只包含调用方需要的业务信息。

### 4.4 忽视字段类型和格式约定

金额用 int 还是 string？时间用时间戳还是格式化字符串？布尔值用 true/false 还是 1/0？

这些看似小的问题如果不统一，联调时全是坑。一个系统的所有接口应该遵循统一的类型约定：

- 金额：用字符串表示，避免浮点精度问题。如 "100.00"。
- 时间：用 ISO 8601 格式。如 "2024-01-15T10:30:00+08:00"。
- ID：用字符串。虽然现在是数字 ID，将来可能换成 UUID。
- 枚举：用有意义的字符串。如 "PAID" 而不是 2。

### 4.5 文档和实际不一致

接口文档是写给调用方看的契约。如果文档说返回的 amount 是"分"，但实际返回的是"元"，调用方会按文档开发，上线后金额差一百倍。

接口文档必须和实际实现保持同步。用 Swagger/OpenAPI 等工具从代码自动生成文档是一个有效的做法——代码改了文档自动更新。

## 5. 设计方法

### 5.1 接口设计流程

**第一步：明确接口的使用场景。** 谁会调用这个接口？在什么场景下调用？需要什么数据？会怎么处理错误？

**第二步：定义核心字段。** 只包含该场景必需的字段。不要把数据库表的所有字段都暴露出去。

**第三步：定义错误码和异常情况。** 列出所有可能的失败场景：参数错误、数据不存在、权限不足、业务规则不满足、服务不可用。每种场景对应一个明确的错误码。

**第四步：确定幂等方案。** 写操作接口必须考虑幂等。用什么机制保证幂等？requestId？状态机？业务唯一键？

**第五步：确定超时和重试约定。** 这个接口的正常响应时间是多少？调用方应该设多少超时？超时后应该重试还是放弃？

**第六步：写文档。** 在开发之前写，而不是开发完之后补。文档即契约。

### 5.2 接口契约模板

一个完整的接口定义可以按以下模板：

```
接口名称：创建支付单
路径：POST /v1/payments
场景：订单确认后创建支付单，获取支付渠道的支付链接

请求参数：
- order_id (string, 必填): 订单号
- amount (string, 必填): 支付金额，单位元，保留两位小数
- channel (string, 必填): 支付渠道，枚举值: ALIPAY / WECHAT / UNIONPAY
- return_url (string, 可选): 支付完成后的跳转地址
- request_id (string, 必填): 请求唯一标识，用于幂等

成功响应：
- payment_id (string): 支付单号
- pay_url (string): 支付链接
- expire_time (string): 支付链接过期时间

错误码：
- INVALID_AMOUNT: 金额格式错误或金额为零
- ORDER_NOT_FOUND: 订单不存在
- ORDER_ALREADY_PAID: 订单已支付
- CHANNEL_UNAVAILABLE: 支付渠道不可用
- DUPLICATE_REQUEST: 重复请求（幂等，返回上一次的结果）

超时约定：建议调用方设置 5 秒超时
幂等：基于 request_id，同一 request_id 的重复调用返回相同结果
频率限制：单用户每分钟最多 10 次
```

## 6. 业务案例

### 6.1 案例：支付回调接口的设计演进

**第一版（问题版）**：

```json
// 回调请求
POST /order/payCallback
{
    "id": "PAY20240115001",
    "status": 1,
    "money": 9900,
    "time": "20240115103000"
}
```

问题：
- id 是什么 ID？支付单号？订单号？没有说明。
- status=1 是什么意思？成功？待处理？没有定义。
- money 的单位是什么？分？元？
- time 的格式不标准，不带时区。
- 没有签名验证，任何人都可以伪造回调。
- 没有幂等约定，重复回调可能重复更新状态。

**第二版（改进版）**：

```json
// 回调请求
POST /v1/orders/payment-callback
Headers:
  X-Signature: HmacSHA256签名
  X-Timestamp: 1705289400
  X-Nonce: abc123

{
    "payment_id": "PAY20240115001",
    "order_id": "ORD20240115001",
    "status": "SUCCESS",
    "amount": "99.00",
    "paid_at": "2024-01-15T10:30:00+08:00",
    "channel": "ALIPAY",
    "channel_transaction_id": "2024011522001001"
}

// 成功响应
{ "code": "OK" }

// 幂等响应（已处理过）
{ "code": "OK" }

// 失败响应
{ "code": "ORDER_NOT_FOUND", "message": "订单不存在" }
```

改进点：
- 字段名称语义明确，status 用枚举字符串。
- 金额用字符串、单位元、保留两位小数。
- 时间用 ISO 8601 带时区。
- 有签名验证防伪造。
- 有幂等处理——重复回调返回 OK 而不是报错。
- 有明确的错误码定义。

### 6.2 案例：订单查询接口的设计

**反面设计**：一个"万能查询"接口

```
GET /orders?userId=xxx&status=paid&startTime=xxx&endTime=xxx
    &minAmount=100&maxAmount=500&productName=手机
    &sort=amount:desc&fields=id,status,amount,items
```

看起来灵活，但调用方需要了解所有可查询的字段和排序规则。而且这个接口在数据库层面很难优化——不同的查询条件组合对应不同的索引。

**正面设计**：按场景拆分

```
// 用户查自己的订单列表（按时间倒序，分页）
GET /v1/users/me/orders?page=1&size=20

// 商家查待发货订单（按下单时间正序）
GET /v1/merchant/orders/pending-shipment?page=1&size=20

// 客服搜索订单（按订单号或手机号）
GET /v1/admin/orders/search?keyword=ORD20240115001
```

每个接口场景明确、返回字段可控、数据库查询可优化。

## 7. 架构判断清单

- 如果联调一个接口花的时间超过开发这个接口的时间，说明接口设计或文档质量有问题。
- 如果调用方需要看源码才能正确调用你的接口，说明接口的自描述性不够。
- 如果一个接口的错误码只有"成功"和"失败"两种，说明错误处理设计缺失。
- 如果写操作接口没有幂等方案，说明在分布式环境下会出问题。
- 如果接口文档和实际行为不一致，说明缺少文档和代码的同步机制。
- 如果一个接口返回了超过 20 个字段，说明可能返回了调用方不需要的信息，考虑按场景拆分。
- 如果金额字段用了浮点数类型，说明会有精度问题。

## 8. 本章小结

接口是系统之间的契约，而不是"能调通就行"的连接管道。好的接口设计能减少联调时间、降低集成错误、让系统更容易演进。坏的接口设计会把问题传播给所有调用方，并且随着时间推移越来越难修正。

接口设计的核心原则是：字段语义明确、错误码分层清晰、写操作保证幂等、有超时和重试约定、文档和实现同步。这些看起来都是"基本功"，但在实际项目中做到位的不多。

对工程师来说，设计一个接口时多花半小时想清楚字段含义、错误场景、幂等方案，能节省联调时几天的扯皮时间。接口设计是一个需要克制的工作——不是功能越多越好，不是字段越多越灵活，而是刚好满足调用方的需要、不多也不少。
